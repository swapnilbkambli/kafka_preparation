Terraform Best Practices for Kafka

  • Use remote state (S3 + DynamoDB lock) — Kafka topic changes from two engineers simultaneously can cause drift
  • Use workspaces or separate state files per environment (dev/staging/prod)
  • terraform plan in CI, terraform apply only via protected main branch with approval gate
  • Never store Kafka credentials in .tfvars files. Use Vault, AWS Secrets Manager, or environment variables
  • Import existing topics before creating: terraform import kafka_topic.payments payments.transactions.v1
  • Set prevent_destroy = true on critical topics to require two-step deletion
  • Use lifecycle { ignore_changes = [config["retention.ms"]] } for configs managed operationally

Rolling Restart Playbook

# rolling_restart.yml
---
- name: Kafka Rolling Restart
  hosts: kafka_brokers
  serial: 1                    # One broker at a time
  max_fail_percentage: 0       # Stop on any failure
  gather_facts: true

  pre_tasks:
    - name: Check UnderReplicatedPartitions before restart
      command: >
        kafka-topics.sh --bootstrap-server {{ groups['kafka_brokers'][0] }}:9092
        --describe --under-replicated-partitions
      register: urp_check
      changed_when: false
      delegate_to: "{{ groups['kafka_brokers'][0] }}"

    - name: Fail if URPs exist before restart
      fail:
        msg: "UnderReplicatedPartitions detected! Fix before restarting. Output: {{ urp_check.stdout }}"
      when: urp_check.stdout | length > 0

    - name: Set controlled shutdown flag
      lineinfile:
        path: /opt/kafka/config/server.properties
        regexp: "^controlled.shutdown.enable"
        line: "controlled.shutdown.enable=true"

  tasks:
    - name: Restart Kafka broker
      systemd:
        name: kafka
        state: restarted

    - name: Wait for broker to rejoin cluster (port 9092 listening)
      wait_for:
        port: 9092
        timeout: 120

    - name: Wait for URPs to return to 0 after restart
      shell: >
        kafka-topics.sh --bootstrap-server {{ groups['kafka_brokers'][0] }}:9092
        --describe --under-replicated-partitions | wc -l
      register: urp_count
      until: urp_count.stdout | int == 0
      retries: 30
      delay: 10
      delegate_to: "{{ groups['kafka_brokers'][0] }}"
      changed_when: false

  post_tasks:
    - name: Verify broker is in ISR
      debug:
        msg: "Broker {{ inventory_hostname }} restarted successfully. URP count: {{ urp_count.stdout }}"

Strimzi Operator Deployment via Helm

# Add Strimzi Helm repo

# Install Strimzi operator (watches all namespaces)
helm install strimzi-kafka-operator strimzi/strimzi-kafka-operator \
  --namespace strimzi-system \
  --create-namespace \
  --set watchNamespaces="{kafka-prod,kafka-staging}" \
  --set logLevel=INFO \
  --set resources.requests.cpu=200m \
  --set resources.requests.memory=256Mi \
  --set resources.limits.cpu=1000m \
  --set resources.limits.memory=512Mi

Helm Release Management Best Practices

  • Pin chart version in Chart.yaml dependencies — never use * or latest
  • Store values.yaml per environment in Git; use helm diff upgrade (helm-diff plugin) before applying
  • Use --atomic flag in CI: rolls back automatically on timeout
  • Separate values.yaml for secrets overlay; pull from Vault via helm secrets plugin or ESO (External Secrets Operator)
  • Test upgrades in staging with production-equivalent config before promoting
  • Use helm test hook with a small producer/consumer pod to verify cluster health post-upgrade

GitOps Workflow for Topic Changes

PR-based topic lifecycle 
Developer/SRE:
  1. git checkout -b feature/add-audit-topic
  2. Create clusters/production/topics/audit-events.yaml:
       apiVersion: kafka.strimzi.io/v1beta2
       kind: KafkaTopic
       metadata:
         name: audit.events.v1
         labels:
           strimzi.io/cluster: hsbc-kafka
       spec:
         partitions: 12
         replicas: 3
         config:
           retention.ms: "2592000000"   # 30 days
           min.insync.replicas: "2"
           cleanup.policy: delete
  3. git push origin feature/add-audit-topic
  4. Open PR → automated checks run:
       - yamllint (syntax)
       - kubectl dry-run --server-side (admission validation)
       - conftest/OPA policy checks (naming conventions,
                    min replicas = 3, min.ISR = 2 required)
  5. Peer review → approval → merge to main
  6. ArgoCD detects change → syncs → Strimzi Topic Operator
     creates/updates the topic on the cluster
  7. Strimzi updates KafkaTopic.status.observedGeneration

Audit trail: every topic change is a Git commit with author,
timestamp, and PR review record.

OPA/Conftest Policies for Kafka CRDs

# policy/kafka_topic.rego
package kafka.topic

import future.keywords.if
import future.keywords.in

deny[msg] if {
  input.kind == "KafkaTopic"
  input.spec.replicas < 3
  msg := sprintf("Topic '%s' must have replicas >= 3, got %d",
                 [input.metadata.name, input.spec.replicas])
}

deny[msg] if {
  input.kind == "KafkaTopic"
  min_isr := to_number(input.spec.config["min.insync.replicas"])
  min_isr < 2
  msg := sprintf("Topic '%s' min.insync.replicas must be >= 2",
                 [input.metadata.name])
}

deny[msg] if {
  input.kind == "KafkaTopic"
  not regex.match(`^[a-z][a-z0-9.-]{2,249}$`, input.metadata.name)
  msg := sprintf("Topic name '%s' must match ^[a-z][a-z0-9.-]{2,249}$",
                 [input.metadata.name])
}

warn[msg] if {
  input.kind == "KafkaTopic"
  not input.spec.config["retention.ms"]
  msg := sprintf("Topic '%s' should have explicit retention.ms set",
                 [input.metadata.name])
}

CI/CD for Kafka Connectors

Connector Deployment Pipeline

GitHub Actions workflow for connector deployment 
Source repo structure:
connectors/
├── debezium-postgres/
│   ├── connector-config.json.j2   # Jinja2 template
│   ├── tests/
│   │   └── integration_test.py
│   └── Makefile
└── s3-sink/
    ├── connector-config.json.j2
    └── tests/

Pipeline stages:
  1. Lint: jsonlint + custom schema validator on connector config
  2. Unit test: mock Connect API, verify config merging logic
  3. Deploy to staging:
       - Render template: j2 connector-config.json.j2 > /tmp/config.json
       - kubectl apply -f connector.yaml (Strimzi KafkaConnector CR)
       - Wait for RUNNING state (poll /connectors/{name}/status)
  4. Integration test (staging):
       - Produce N records to source topic
       - Verify N records appear in sink (S3, DB, etc.)
       - Verify DLQ is empty
  5. Approval gate (production): manual approval in GitHub Actions
  6. Deploy to production (same steps as staging)
  7. Smoke test: verify connector RUNNING, lag < 100 after 5 min
          
# .github/workflows/connector-deploy.yml
name: Deploy Kafka Connector
on:
  push:
    paths: ['connectors/**']
    branches: [main]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4

      - name: Render connector config
        run: |
          pip install j2cli
          j2 connectors/${{ env.CONNECTOR }}/connector-config.json.j2 \
            -e STAGING_DB_HOST=${{ secrets.STAGING_DB_HOST }} \
            > /tmp/connector-config.json

      - name: Deploy to staging via Strimzi
        run: |
          kubectl apply -f - <<EOF
          apiVersion: kafka.strimzi.io/v1beta2
          kind: KafkaConnector
          metadata:
            name: debezium-payments
            labels:
              strimzi.io/cluster: kafka-connect-cluster
          spec:
            class: io.debezium.connector.postgresql.PostgresConnector
            tasksMax: 4
            config: $(cat /tmp/connector-config.json)
          EOF

      - name: Wait for connector RUNNING state
        run: |
          for i in $(seq 1 30); do
            STATUS=$(kubectl get kafkaconnector debezium-payments \
              -o jsonpath='{.status.connectorStatus.connector.state}')
            [ "$STATUS" = "RUNNING" ] && exit 0
            sleep 10
          done
          echo "Connector failed to reach RUNNING state"
          exit 1

  deploy-production:
    needs: deploy-staging
    environment: production    # requires manual approval in GitHub
    runs-on: ubuntu-latest
    steps:
      # Same steps as staging with production secrets

Automated Partition Scaling

Partitions can only be increased, never decreased. Over-partitioning is worse than under-partitioning. Start with a formula and review regularly. 
Partition Sizing Formula:
  target_rate = max(producer_rate, consumer_rate)  # msg/sec
  target_partitions = ceil(target_rate / throughput_per_partition)
  throughput_per_partition ≈ 10 MB/s (NVMe) or 5 MB/s (spinning disk)

Example:
  producer_rate = 500 MB/s
  throughput_per_partition = 10 MB/s (NVMe SSDs)
  partitions = ceil(500/10) = 50 → round to 48 (multiple of consumers)

Automation (KEDA-based or custom):
  - Monitor bytes-in-rate per partition via Prometheus
  - If avg > 8 MB/s per partition for 10 consecutive minutes:
      → Open automated PR to increase partitions by 50%
      → Require human approval (increasing partitions is irreversible)
  - Never auto-apply partition changes — require PR approval

Q & A

Senior How would you manage Kafka topic configuration drift between staging and production?

The root cause of drift is usually out-of-band changes: someone runs kafka-configs.sh --alter directly on prod for an emergency and forgets to update the code. Preventing this requires making the "right path" (Git PR) easier than the "wrong path" (direct kubectl/kafka-configs).

My approach:

  1. Single source of truth in Git: All topic configs in YAML (Strimzi KafkaTopic CRs) in a GitOps repo. ArgoCD's selfHeal=true automatically reverts any out-of-band change within minutes.
  2. Drift detection job: Scheduled CronJob in CI (daily) that runs kafka-topics.sh --describe for all topics and compares against expected values from Git. Posts diff to Slack if drift found.
  3. Config validation in PR: OPA/conftest policies check that staging and prod configs have the same replication factor, min.ISR, and cleanup policy. Retention may differ (staging = 1 day, prod = 7 days) — this is intentional and encoded in variables.
  4. Access controls: RBAC on Kubernetes means only the Strimzi operator ServiceAccount has permission to modify topics. Engineers don't get direct kafka-admin access in production — they must go through Git.

This makes drift structurally impossible for most config changes, and detectable within hours for emergency overrides.

Senior Describe how you'd implement zero-touch Kafka cluster upgrades using GitOps.

Kafka upgrades have two phases: the binary version bump, then the inter.broker.protocol.version (IBP) bump. These must be separate steps because rolling back the binary is possible; rolling back IBP is not.

GitOps workflow:

  1. Engineer opens PR updating the Strimzi Kafka CR: spec.kafka.version: 3.6.0 → 3.7.0. The IBP field is intentionally left at 3.6.
  2. CI runs: yamllint, schema validation, kubectl diff --server-side shows the diff. Required reviewer approves.
  3. Merge to staging branch → ArgoCD syncs staging → Strimzi operator performs rolling restart one broker at a time, gating on UnderReplicatedPartitions=0 before each broker. CI runs integration tests post-upgrade.
  4. After 48-hour soak in staging: promote to main → ArgoCD syncs production with the same binary version change.
  5. Second PR: bump inter.broker.protocol.version: "3.7" and log.message.format.version: "3.7". Same PR → review → staging → production flow. This is the point of no return.

Strimzi does the hard work: It validates cluster health before each broker restart, waits for ISR recovery, and handles the IBP bump sequentially. The SRE's job is the PR review — the automation handles execution.

Rollback gate: If integration tests fail in staging after the binary bump (before IBP), revert the PR. Strimzi will roll back the binary version on next sync. After IBP bump, rollback is impossible — this is why the soak period and two-PR approach matter.
