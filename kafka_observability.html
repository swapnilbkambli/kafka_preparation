📉Consumer Lag Monitoring

kafka-consumer-groups.sh (CLI)

Built-in tool. Good for ad-hoc checks. Not suitable for continuous monitoring — polling overhead and no historical tracking. Use for incident investigation, not alerting.

Burrow (LinkedIn)

Dedicated consumer lag monitor. Reads __consumer_offsets directly. Provides lag status as OK/STALLED/STOPPED/ERR — not just a number. Has built-in alerting. REST API for integration. More operationally complex.

kminion

Modern Prometheus exporter for Kafka. Exports consumer group lag as Prometheus metrics. Simpler than Burrow. Good for Grafana-based monitoring stacks. Actively maintained.

Confluent Control Center

Commercial product. Full monitoring UI including consumer lag, schema registry, Connect, ksqlDB. Built-in alerting. High operational value for large Kafka deployments. License cost.

💡
Interview insight: Consumer lag alone is not enough for alerting. A group with lag=1M that's DECREASING is healthy (catching up). A group with lag=1K that's INCREASING is concerning. Alert on lag growth rate (derivative), not absolute lag. Burrow's "STALLED" state detects this — consumer is committing offsets but lag is not decreasing.
PROMETHEUS QUERY FOR LAG ALERTING (growth rate): ────────────────────────────────────────────────────────────────── # Growing lag (increase over last 10 min, filter noise) increase(kminion_consumer_group_topic_partition_lag[10m]) > 50000 # Rate of lag change per second (positive = growing, negative = catching up) rate(kminion_consumer_group_topic_partition_lag[5m]) # Time to catch up (current lag / consumption rate - production rate) # Useful for SLA estimation in dashboards

🎯Observability Q&A
Q What's the first dashboard you'd look at during a Kafka incident at 3am? Mid
In order of priority:
  1. UnderReplicatedPartitions — Any value > 0 means data durability is at risk. This is the single most important broker health metric. If URP > 0, everything else is secondary until this is resolved.
  2. OfflinePartitionsCount — If any partition has no leader, that data is unavailable to producers and consumers. Immediate P1 impact.
  3. Consumer lag trend — Is the incident causing consumers to fall behind? Which groups? This tells you business impact.
  4. Request queue size + IO thread idle % — Is the broker keeping up with the load? If queue is growing, the broker is the bottleneck.
  5. Disk usage — Is the disk full? Quick sanity check that rules out a simple cause.
Within 2 minutes of waking up: URP? Offline? Lag growing? Disk full? If all are OK, dig deeper into latency metrics.
Q Prometheus shows consumer lag = 0 but records still seem delayed. How do you investigate? Senior
Lag=0 means the consumer has committed offsets up to the latest offset. The delay is happening AFTER consumption:
  • Processing time: Consumer fetches records and commits, but actual downstream processing (DB write, API call) happens asynchronously and is slow. The lag metric doesn't capture processing latency — only fetch position.
  • End-to-end latency metric needed: Add a custom metric: timestamp when record was produced (in message headers or payload) vs timestamp when processing completed. This gives true end-to-end latency independent of Kafka lag.
  • Auto-commit before processing: If enable.auto.commit=true, offsets are committed on a timer regardless of processing status. Lag=0 but records aren't processed yet.
  • Check downstream: Is the DB slow? Is the downstream API backed up? Latency is in the sink, not Kafka.
Lesson: Consumer lag is a Kafka-level metric. Business SLA requires end-to-end latency tracking: produce-timestamp to processed-timestamp. Build that instrumentation into your application.
