🌐What is Kafka REST Proxy & Why Use It?
KAFKA REST PROXY ARCHITECTURE: ────────────────────────────────────────────────────────────────── HTTP Clients REST Proxy Kafka ───────────── ───────────── ───── Python script ──POST──► ┌─────────────┐ produce ► Topic A IoT device ──POST──► │ Confluent │ Topic B Bash / curl ──GET──► │ REST Proxy │ consume ◄ Topic C Legacy system ──HTTP──► │ (:8082) │ metadata Cluster Mobile app ──HTTP──► └─────────────┘ │ Translates │ │ HTTP ↔ │ │ Kafka Proto│ └─────────────┘ Stateful: holds consumer connections per consumer instance ID

Primary Use Case

Allows clients that cannot use the Kafka binary protocol (non-JVM, IoT, legacy systems, scripting) to produce and consume Kafka messages via standard HTTP. No Kafka client library required — just HTTP.

What It Is NOT

Not a replacement for native Kafka clients in performance-sensitive paths. Not a streaming interface — HTTP is request/response, not streaming. Not suitable for high-throughput producers (latency overhead per HTTP request).

Stateful Consumer Sessions

Unlike the native client, REST Proxy maintains consumer state server-side. A consumer instance is created (POST /consumers), then repeatedly polls (GET /records). The proxy holds the Kafka consumer object — requests must route to the same proxy instance.

Schema Registry Integration

REST Proxy integrates with Schema Registry. Set Content-Type: application/vnd.kafka.avro.v2+json and REST Proxy serializes/deserializes Avro automatically, looking up schemas from SR.

📤Producing Messages via REST
# Produce JSON messages to a topic POST /topics/payments Content-Type: application/vnd.kafka.json.v2+json { "records": [ {"key": "acc-001", "value": {"amount": 150.00, "currency": "USD"}}, {"key": "acc-002", "value": {"amount": 75.50, "currency": "GBP"}} ] } Response 200: { "key_schema_id": null, "value_schema_id": null, "offsets": [ {"partition": 3, "offset": 10004, "error": null}, {"partition": 1, "offset": 20301, "error": null} ] } # Produce Avro with Schema Registry POST /topics/payments Content-Type: application/vnd.kafka.avro.v2+json { "value_schema": "{\"type\":\"record\",\"name\":\"Payment\",\"fields\":[...]}", "records": [ {"value": {"id": "PAY-001", "amount": 100.0}} ] } # Produce to a specific partition POST /topics/payments/partitions/2 { "records": [{"value": {"amount": 200.0}}] }
⚠️
Performance caveat: Each POST /topics call goes through HTTP parsing, JSON serialization, and a synchronous Kafka produce call. Throughput is orders of magnitude lower than native Kafka producers. For bulk ingestion, batch multiple records in one POST (up to the configured max batch size), but still — don't use REST Proxy for high-throughput paths. Use it for low-frequency, convenience-driven producers.
📥Consuming Messages via REST
CONSUMER LIFECYCLE (3 steps, must use same proxy instance): ────────────────────────────────────────────────────────────────── Step 1: Create consumer instance (initializes Kafka consumer) POST /consumers/my-consumer-group Content-Type: application/vnd.kafka.v2+json {"name": "my-instance", "format": "json", "auto.offset.reset": "earliest"} Response: {"instance_id": "my-instance", "base_uri": "http://proxy-1:8082/consumers/my-consumer-group/instances/my-instance"} Step 2: Subscribe to topic(s) POST /consumers/my-consumer-group/instances/my-instance/subscription {"topics": ["payments"]} Step 3: Poll for records (call repeatedly) GET /consumers/my-consumer-group/instances/my-instance/records Accept: application/vnd.kafka.json.v2+json Response: [ {"topic": "payments", "partition": 3, "offset": 10004, "key": "acc-001", "value": {"amount": 150.00}}, ... ] Step 4: Commit offsets (manual commit) POST /consumers/my-consumer-group/instances/my-instance/offsets {"offsets": [{"topic": "payments", "partition": 3, "offset": 10005}]} Step 5: Delete consumer instance when done DELETE /consumers/my-consumer-group/instances/my-instance
⚠️
Sticky routing requirement: After Step 1, the consumer instance lives on a specific proxy server. All subsequent calls (subscribe, poll, commit, delete) MUST go to the same proxy instance. Use sticky sessions on your load balancer (cookie-based or consistent hashing on instance ID). Calling a different proxy server returns 404 — the instance doesn't exist there.
🚨
Consumer leak: If a client creates a consumer instance (Step 1) but never deletes it (Step 5) — even after the HTTP connection drops — the consumer instance remains active on the proxy server, holding a Kafka consumer connection. This causes partition assignment issues and eventually exhausts proxy resources. Always DELETE the consumer instance when done, even on error. Set consumer.instance.timeout.ms on the proxy to auto-expire idle instances.

# List topics GET /topics Response: ["payments", "orders", "fraud-alerts", ...] # Get topic metadata (partitions, replication, configs) GET /topics/payments Response: {"name":"payments","configs":{"cleanup.policy":"delete",...}, "partitions":[{"partition":0,"leader":1,"replicas":[...],"isr":[...]},...]} # Get partition metadata GET /topics/payments/partitions GET /topics/payments/partitions/0 # Get broker list GET /brokers Response: {"brokers": [1, 2, 3]} NOTE: REST Proxy admin APIs are READ-ONLY (metadata only). For write operations (create topic, alter configs, reassign partitions) use kafka-topics.sh, kafka-configs.sh, or Kafka Admin Client API directly.
🔐Security

Proxy → Kafka Auth

REST Proxy authenticates to Kafka as a client. Configure client.security.protocol, client.ssl.*, or client.sasl.* properties in the proxy config. The proxy acts as a privileged Kafka client on behalf of HTTP callers.

Client → Proxy Auth

HTTP clients auth to the proxy using HTTP Basic Auth or API keys (depending on deployment). Add a reverse proxy (nginx, Kong) for auth enforcement. REST Proxy itself has limited built-in auth — secure it at the network/gateway level.

Per-Client ACLs

In the community version, the proxy uses a single Kafka identity for all HTTP clients — ACLs apply to the proxy's service account, not per HTTP client. Confluent Platform's REST Proxy supports per-client auth propagation for granular ACLs.

TLS

Enable TLS on the REST Proxy HTTP listener (listeners.https.*). Use valid certificates. REST Proxy → Kafka communication should also be TLS (client.ssl.truststore.*).

⚠️
Security gap: Without per-client auth propagation, any HTTP client reaching the REST Proxy endpoint can produce/consume any topic the proxy's service account has access to. This is a significant security risk in multi-tenant environments. Either use Confluent Platform's per-client auth, or run separate REST Proxy instances per team/application with different Kafka service accounts.

🎯Concepts & Scenario Q&A
Q An HTTP client is getting 404 errors when trying to poll records, even though it created a consumer instance successfully. What's wrong? Mid
Classic sticky session routing failure. The consumer instance was created on REST Proxy instance A. The subsequent poll request was routed to REST Proxy instance B (by the load balancer) — which has no record of this consumer instance.

Fix:
  • Configure the load balancer for sticky sessions based on the consumer instance base_uri or a cookie. Route all requests for the same consumer instance to the same proxy backend.
  • Alternatively: assign each consumer application to a specific proxy instance (bypass LB for consumer sessions).
  • Also check: did the consumer instance expire? If the client didn't poll within consumer.instance.timeout.ms (default 300s), the proxy auto-deletes the instance.
Prevention: Design client code to handle 404 on poll by re-creating the consumer instance and re-subscribing. This makes the client resilient to proxy restarts and expiry.
Q Why would you choose REST Proxy over a native Kafka client? What are the trade-offs? Basic
Choose REST Proxy when:
  • The producer/consumer is written in a language without a mature Kafka client (some niche languages, old legacy systems).
  • Firewall/security policy only allows HTTP/HTTPS — Kafka port 9092 is blocked.
  • You need a quick proof-of-concept without implementing Kafka client setup.
  • IoT devices or embedded systems that can only do HTTP.
Trade-offs:
  • Lower throughput (HTTP overhead, synchronous per-batch)
  • Higher latency (extra network hop through proxy)
  • Stateful consumer sessions complicate load balancing
  • Limited delivery semantics (no transactions, no idempotent producer equivalent)
  • Additional infrastructure to operate and scale
In a bank context: REST Proxy is valuable for integrating with legacy batch systems that can only speak HTTP/SOAP. For all new services, native Kafka clients (Java, Python kafka-python, Go sarama) are strongly preferred.
