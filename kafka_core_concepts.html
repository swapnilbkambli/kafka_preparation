🧱 Kafka Building Blocks

Broker

A single Kafka server process. Stores data, serves reads/writes. A cluster is 3+ brokers. One broker is the Controller (manages metadata, leader elections).

Topic

A named log/stream. Producers write to it; consumers read from it. Topics are split into partitions for parallelism and scalability.

Partition

An ordered, immutable sequence of records. The unit of parallelism. Each partition lives on one broker (leader) and is replicated to others (followers). Records within a partition are ordered; across partitions they are not.

Offset

A sequential ID for each record within a partition. Consumers track offsets to know where they left off. Stored in the internal topic __consumer_offsets.

Producer

Publishes records to topics. Routes records to partitions via a partitioner (key hash by default, or round-robin if no key). Controls durability via acks.

Consumer Group

One or more consumers sharing a group.id. Each partition is consumed by exactly one member. Enables horizontal scaling — add consumers to scale out reads.

Record / Message

The unit of data: key (optional, used for partitioning), value (payload), headers (metadata), and timestamp.

Log Segment

Partitions are stored as log segments on disk — rolling files of fixed size or time window (e.g., 1 GB / 7 days). Active segment is appended to; older ones are eligible for retention/compaction.

TOPIC: orders (6 partitions, RF=3) Broker 1 ────────────────────────────── P0 [Leader] P1 [Follower] P4 [Follower] Broker 2 ────────────────────────────── P1 [Leader] P2 [Follower] P5 [Follower] Broker 3 ────────────────────────────── P2 [Leader] P0 [Follower] P3 [Leader] Consumer Group A: Consumer-1 → P0, P1 Consumer-2 → P2, P3 Consumer-3 → P4, P5
💡
Key rule: You cannot have more consumers in a group than there are partitions. Extra consumers sit idle. This is why partition count planning at topic creation is critical — you can increase partitions but never decrease them.
🔁 Replication & ISR

Replication Factor (RF)

How many copies of each partition exist. RF=3 means one leader + two followers. Industry standard for production. RF=1 is no fault tolerance.

Leader

The one broker that handles all reads and writes for a partition. Elected per partition. Followers only replicate from the leader.

ISR (In-Sync Replicas)

Replicas that are fully caught up with the leader (within replica.lag.time.max.ms). Only ISR members are eligible to be elected as leader on failure.

Under-Replicated Partitions

Partitions where ISR count < RF. A critical alert signal — means some replicas are lagging or a broker is down. Check metric UnderReplicatedPartitions.

📥 Consumers, Consumer Groups & Offsets

Offset Commit

Consumers commit offsets to __consumer_offsets to record progress. Auto-commit (enable.auto.commit=true) can cause duplicates or data loss; manual commit gives control.

Rebalance

When a consumer joins/leaves/crashes, Kafka re-assigns partitions across the group. During rebalance, all consumption stops. Minimize with session.timeout.ms and max.poll.interval.ms.

Consumer Lag

Difference between the latest offset (producer end) and the consumer's committed offset. High lag = consumer falling behind. Key SRE alert metric.

Cooperative Rebalancing

Introduced in Kafka 2.4 (CooperativeStickyAssignor). Only re-assigns partitions that need to move — consumers keep others during rebalance, eliminating the "stop the world" pause.

auto.offset.reset

earliest — start from the oldest record when no committed offset exists.
latest — start from new records only. Choose based on your data loss vs replay tolerance.

Partition Assignment

Strategies: RangeAssignor (default), RoundRobinAssignor, StickyAssignor, CooperativeStickyAssignor. Sticky minimizes partition movement during rebalances.

🎯 Interview Q&A — Basic
Q What is a Kafka partition and why does it matter? Basic
A partition is an ordered, append-only log — the fundamental unit of storage and parallelism in Kafka. It matters because:
  • Scalability: More partitions = more consumers in parallel = higher throughput.
  • Ordering: Order is guaranteed within a partition (via key-based routing), not across them.
  • Replication: Each partition is replicated independently, giving fault tolerance at the partition level.
The tradeoff: more partitions = more file handles, memory, and leader election overhead. Rule of thumb: start with 3–6x the number of brokers; don't over-partition early.
Q Explain ISR and why it matters for durability. Basic
ISR (In-Sync Replicas) is the set of replicas fully caught up with the leader. When a producer uses acks=all, Kafka waits for all ISR members to acknowledge the write before confirming success.
  • A follower is removed from ISR if it hasn't fetched from the leader within replica.lag.time.max.ms.
  • Only ISR members can become the new leader on failover — this prevents data loss.
  • Setting unclean.leader.election.enable=false enforces this — if no ISR member is available, the partition goes offline rather than electing a stale replica.
In a bank's context (like HSBC), you always want acks=all + min.insync.replicas=2 + unclean.leader.election.enable=false for financial data.
Q What happens during a consumer group rebalance? Basic
A rebalance is triggered when a consumer joins, leaves, or crashes, or when partitions are added to a topic.
  • The Group Coordinator (a broker) orchestrates it.
  • In the classic (eager) protocol: all consumers stop consumption, revoke their partitions, rejoin, and get new assignments — causing a full stop-the-world pause.
  • In the cooperative/incremental protocol (Kafka 2.4+): only the affected partitions are revoked and reassigned; others continue normally.
SRE concern: Frequent rebalances cause consumer lag spikes. Root causes: long GC pauses, slow poll() loops, network issues. Tune max.poll.interval.ms and session.timeout.ms accordingly.
🎯 Interview Q&A — Mid-Level
Q How does Kafka achieve high throughput on a single broker? Mid
Kafka is designed around OS primitives that are extremely fast:
  • Sequential disk I/O: Appending to a log is orders of magnitude faster than random writes. Kafka never modifies records in place.
  • Page cache: Kafka relies on the OS page cache instead of JVM heap. Recent data is served from RAM; older data is read from disk.
  • Zero-copy: Uses sendfile() syscall — data is transferred from page cache to network socket without copying to userspace. Hugely reduces CPU overhead for consumers.
  • Batching: Producers batch records (batch.size, linger.ms); brokers write batches; consumers fetch batches. Fewer round-trips, better compression ratios.
  • Compression: Batches are compressed as a unit (LZ4, Snappy, ZSTD), further reducing I/O and network.
Q What is log compaction and when would you use it? Mid
Log compaction is a retention policy where Kafka keeps the latest value for each key in a partition, rather than deleting data after a time/size window.
  • Enables changelog / event-sourcing patterns: consumers can rebuild state by replaying a compacted topic (e.g., a product catalog, user preferences).
  • Records with a null value (tombstones) signal deletion — the key is eventually removed during compaction.
  • Compaction runs in the background via dedicated cleaner threads (log.cleaner.threads).
  • Compacted topics always retain the active segment (head) untouched; only older closed segments are compacted.
Use cases: Kafka Streams changelog topics, database CDC (Change Data Capture) snapshots, config stores.
Q You see UnderReplicatedPartitions > 0 on a broker. Walk me through your investigation. Mid
This is a critical alert — some partitions have fewer in-sync replicas than expected.
  • Step 1 — Identify scope: kafka-topics.sh --describe --under-replicated-partitions — list affected partitions and which broker is lagging.
  • Step 2 — Check broker health: Is the lagging broker's process running? Network reachable? GC logs showing long pauses?
  • Step 3 — Check disk: df -h on the broker. Kafka stops replication if a broker's disk is full.
  • Step 4 — Check replica lag: JMX metric ReplicaFetcherManager.MaxLag. If lag is rising, the follower is not keeping up — check CPU, I/O, network.
  • Step 5 — Broker just restarted? Lag is expected during catch-up — monitor for convergence. If it doesn't converge, investigate throttling (replica.fetch.max.bytes).
  • Resolution: Fix the root cause (disk, network, GC). In extreme cases, run preferred replica election to rebalance leaders after recovery.
🎯 Interview Q&A — Senior / SRE Level
Q How would you handle a broker that keeps getting removed from the cluster due to frequent GC pauses? Senior
This is a JVM tuning + Kafka configuration problem.
  • Short-term: Increase replica.lag.time.max.ms to tolerate longer pauses without ISR shrinkage — buys time without dropping the follower from ISR repeatedly.
  • Root cause — JVM: Switch from CMS to G1GC (default since JDK 9) or ZGC for sub-10ms pauses. Set -XX:MaxGCPauseMillis=20. Tune heap: Kafka brokers typically need 4–8 GB heap, not more — the rest of RAM should be free for OS page cache.
  • Heap sizing mistake: Giving the JVM too much heap causes longer GC cycles. Kafka's performance depends on the OS page cache, not JVM heap. A 32 GB box should have 6 GB JVM heap and 26 GB for page cache.
  • Monitoring: Alert on jvm.gc.collection_time and kafka.server:type=ReplicaFetcherManager,name=MaxLag together.
  • Long-term: Profile heap allocations — large message sizes or high partition counts can amplify GC pressure.
Q A financial topic requires strict ordering and no data loss. How do you configure it end-to-end? Senior
Producer side:
  • acks=all — wait for all ISR to acknowledge.
  • enable.idempotence=true — deduplicates retries on the broker side.
  • max.in.flight.requests.per.connection=1 — with idempotence enabled you can use up to 5, but set to 1 if ordering is critical and idempotence is off (legacy).
  • retries=Integer.MAX_VALUE + delivery.timeout.ms appropriate for business SLA.
  • Use a single partition key per entity (e.g., account ID) to guarantee per-entity ordering.
Broker / topic side:
  • replication.factor=3, min.insync.replicas=2.
  • unclean.leader.election.enable=false.
  • Retention policy aligned with business audit requirements (e.g., 7 years for financial records — use tiered storage or compaction).
Consumer side:
  • Commit offsets after successful processing (manual commit).
  • Idempotent consumer logic (handle replays gracefully).
  • Use transactions if writing results to another Kafka topic — ensures read-process-write atomicity.
