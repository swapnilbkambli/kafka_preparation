🛡️Kafka Security Layers
KAFKA SECURITY MODEL (3 independent layers): ────────────────────────────────────────────────────────────────── Layer 1: ENCRYPTION IN TRANSIT (TLS/SSL) Client → Broker: TLS 1.2/1.3 (encrypt all data on the wire) Broker → Broker: TLS (inter-broker) Client → ZooKeeper: TLS (if ZK mode) Layer 2: AUTHENTICATION (Who is this client?) Options: SASL/PLAIN, SASL/SCRAM, SASL/GSSAPI (Kerberos), SASL/OAUTHBEARER, mTLS Layer 3: AUTHORIZATION (What can this client do?) ACLs: per resource (Topic, Group, TransactionalId, Cluster) per operation (Read, Write, Create, Delete, Alter, Describe...) per principal (user, group) Authorizer class: SimpleAclAuthorizer (built-in) or custom RBAC INDEPENDENT CONFIGURATION: listeners=PLAINTEXT://:9092,SSL://:9093,SASL_SSL://:9094 Each listener can have different security protocol. Inter-broker can differ from client-facing listener.

Security Protocol Options

PLAINTEXT — No encryption, no auth (dev only)
SSL — TLS encryption, optional client cert auth
SASL_PLAINTEXT — SASL auth, no encryption (internal network only)
SASL_SSL — SASL auth + TLS encryption (production standard)

Principal

The authenticated identity of a client. With SASL: User:alice. With mTLS: User:CN=alice,OU=payments. ACLs bind permissions to principals. A principal can have multiple ACLs across multiple resources.

Super Users

Principals listed in super.users=User:admin;User:kafka-broker bypass all ACL checks. Use sparingly — only for admin service accounts and inter-broker communication in environments where you don't want per-topic ACLs for brokers.

HSBC Context

In a regulated financial institution: all traffic must be encrypted (SASL_SSL everywhere). All clients must be authenticated. All topic access must be ACL-controlled per team. Audit logging of all access events to SIEM.

🔒TLS / SSL Configuration
BROKER SSL CONFIG (server.properties): ────────────────────────────────────────────────────────────────── listeners=SASL_SSL://:9093 advertised.listeners=SASL_SSL://broker1.hsbc.internal:9093 ssl.keystore.location=/etc/kafka/certs/kafka-broker.keystore.jks ssl.keystore.password=changeit ssl.key.password=changeit ssl.truststore.location=/etc/kafka/certs/kafka.truststore.jks ssl.truststore.password=changeit # Require client certificate (mTLS) — optional ssl.client.auth=required # required | requested | none ssl.protocol=TLSv1.3 # Enforce TLS 1.3 (disable 1.2 in regulated env) ssl.enabled.protocols=TLSv1.3,TLSv1.2 ssl.cipher.suites=TLS_AES_256_GCM_SHA384,TLS_AES_128_GCM_SHA256 # Certificate verification ssl.endpoint.identification.algorithm=https # Verify broker hostname in cert PRODUCER/CONSUMER SSL CONFIG: ────────────────────────────────────────────────────────────────── security.protocol=SASL_SSL ssl.truststore.location=/etc/kafka/client.truststore.jks ssl.truststore.password=changeit # If mTLS (client cert required): ssl.keystore.location=/etc/kafka/client.keystore.jks ssl.keystore.password=changeit
⚠️
Performance impact of TLS: TLS encryption adds CPU overhead and disables zero-copy (sendfile()). Expect ~20–30% throughput reduction vs plaintext. Mitigation: use AES-NI hardware acceleration (most modern CPUs support this — verify with openssl speed aes-256-gcm). Use TLS session resumption to amortize handshake cost for long-lived connections.
💡
Certificate management: Use short-lived certificates (90 days) from an internal PKI / Vault PKI engine. Automate rotation — manual certificate management at scale leads to outages when certs expire. Set up monitoring on cert expiry dates (alert at 30 days, critical at 7 days).

📋ACLs & Authorization
ACL STRUCTURE: Principal + ResourceType + ResourceName + Operation + Allow/Deny ────────────────────────────────────────────────────────────────── Resource Types: Topic, Group, Cluster, TransactionalId, DelegationToken Operations per resource: Topic: Read, Write, Create, Delete, Alter, Describe, AlterConfigs, DescribeConfigs Group: Read, Describe, Delete Cluster: Create, Alter, Describe, AlterConfigs, DescribeConfigs, IdempotentWrite ACL EXAMPLES: # Allow producer to write to a topic kafka-acls.sh --bootstrap-server broker:9092 \ --add --allow-principal User:payments-producer \ --operation Write --topic payments # Allow consumer to read from topic + commit offsets to group kafka-acls.sh --bootstrap-server broker:9092 \ --add --allow-principal User:payments-consumer \ --operation Read --topic payments kafka-acls.sh --bootstrap-server broker:9092 \ --add --allow-principal User:payments-consumer \ --operation Read --group payments-processor-group # Allow wildcard (all topics with prefix "payments-") kafka-acls.sh --bootstrap-server broker:9092 \ --add --allow-principal User:payments-team \ --operation Read,Write --topic 'payments-' --resource-pattern-type prefixed # List ACLs kafka-acls.sh --bootstrap-server broker:9092 --list --topic payments # Deny specific operation (DENY takes precedence over ALLOW) kafka-acls.sh --bootstrap-server broker:9092 \ --add --deny-principal User:analytics-reader \ --operation Delete --topic payments
⚠️
Common ACL mistakes:
1. Forgetting the consumer group ACL — consumers need Read on the topic AND Read on their consumer group. Without group ACL, they get "Authorization failed" on offset commits.
2. Idempotent producers need IDEMPOTENT_WRITE on Cluster resource: --operation IdempotentWrite --cluster.
3. Transactional producers need Write on TransactionalId resource.
4. ACL changes take effect immediately — no broker restart needed.

🔄Inter-Broker Security
INTER-BROKER AUTH CONFIG: ────────────────────────────────────────────────────────────────── # Separate listener for inter-broker traffic (best practice) listeners=SASL_SSL://0.0.0.0:9092,SASL_SSL://0.0.0.0:9093 listener.security.protocol.map=CLIENT:SASL_SSL,REPLICATION:SASL_SSL inter.broker.listener.name=REPLICATION # Inter-broker SASL mechanism sasl.mechanism.inter.broker.protocol=SCRAM-SHA-512 # Broker-to-broker credential (in JAAS config) listener.name.replication.scram-sha-512.sasl.jaas.config=\ org.apache.kafka.common.security.scram.ScramLoginModule required \ username="kafka-broker" password="broker-secret"; # Grant broker principal super.users access (or specific ACLs) super.users=User:kafka-broker # KRaft: controller-broker communication also authenticated # controller.listener.names=CONTROLLER # listener.security.protocol.map=CONTROLLER:SASL_SSL
💡
Why separate listener for replication? Different security policies can be applied to client vs replication traffic. Replication listener can be on a separate network interface (management network) isolated from client traffic. This reduces attack surface: compromise of client network doesn't expose replication traffic.
💾Encryption at Rest

Kafka Native (None)

Kafka does NOT encrypt data at rest natively. Log segments on disk are plaintext. Encryption at rest must be implemented at the OS/storage layer.

OS/Volume Encryption

Use Linux LUKS encryption (dm-crypt) or cloud provider disk encryption (AWS EBS encryption, Azure Disk Encryption). Transparent to Kafka — no code change. Key managed by KMS. Standard approach for compliance.

Cloud Provider Managed

AWS MSK: EBS encryption with KMS CMK, automatically enabled. Confluent Cloud: AES-256 at rest, managed. Azure HDInsight: Azure Disk Encryption. Cloud-managed is simplest for new deployments.

Application-Level Encryption

Producers encrypt sensitive fields before producing (field-level encryption). Authorized consumers decrypt. Schema Registry can store encryption metadata. Most complex but provides finest-grained control — specific fields encrypted, not entire messages.

✅
HSBC compliance approach: Layer 1 (OS disk encryption) covers regulatory requirements for data at rest. Layer 2 (TLS in transit) covers data in motion. For PII and sensitive financial data: add field-level encryption at the application layer. This way, even if a broker's disk is physically stolen, data is unreadable at two levels of encryption.
📝Audit Logging

Authorizer Audit Log

Kafka's SimpleAclAuthorizer logs all ALLOW/DENY decisions to the broker log. Format: Principal=User:alice,Operation=WRITE,Topic=payments,Result=ALLOW. Parse and ship to SIEM (Splunk, ELK). Noisy but complete.

Confluent Audit Log

Confluent Platform (commercial) provides structured audit events as Avro records in a Kafka topic. More efficient than log parsing. Includes authentication events, authorization, and admin operations. Schema-versioned via SR.

What to Audit

Authentication: every login attempt (success and failure). Authorization: every ACL decision. Admin: topic create/delete/alter, ACL changes, quota changes, config changes. Data access: produce/consume on sensitive topics (selective — don't log all reads).

SIEM Integration

Ship broker authorizer logs to SIEM via Kafka → Logstash/Filebeat → Elasticsearch or Splunk. Create alerts for: repeated auth failures (brute force), unexpected topic access, ACL changes outside change-management windows.

🎯Security Q&A
Q A consumer is getting "Authorization failed" even though you've granted READ on the topic. What did you miss? Mid
Almost always: missing consumer group ACL. Consumers need two ACLs:
  1. READ on the Topic resource (to fetch records)
  2. READ on the Group resource (to commit offsets to the group coordinator)
The error "Authorization failed" on group operations is easy to miss because the consumer may successfully read for a while (reads work) but then fails when it tries to commit offsets. Check the full exception — it usually specifies which resource type failed authorization.

Fix: kafka-acls.sh --add --allow-principal User:consumer-app --operation Read --group payments-processor-group
Q How do you handle Kafka certificate rotation without downtime? Senior
Certificate rotation without downtime requires that both old and new certificates are trusted during the transition:
  1. Add new CA cert to truststore (before rotating leaf certs): Update the broker truststores to include BOTH old CA cert and new CA cert. Brokers accept clients with certs from either CA. No restart if using ssl.client.auth=requested (soft check) — brokers pick up new truststore dynamically in some versions.
  2. Rotate leaf certificates on clients: Update client keystores with certs signed by the new CA, one at a time. Brokers trust both CAs → no interruption.
  3. Rotate leaf certificates on brokers: Rolling restart of brokers with new keystore. One at a time, wait for ISR recovery between each.
  4. Remove old CA from truststore: After all clients and brokers are using new certs: remove old CA cert from truststores. This is the final cleanup step.
Automate with Vault PKI or cert-manager: Use short-lived certs (90 days) from HashiCorp Vault PKI engine. Automated rotation triggered 30 days before expiry. The dual-trust window during rotation is handled by the automation framework.
