Cluster Sizing & Hardware

The Sizing Decision Tree

Step 1 — Quantify your workload 
Key inputs to gather from application teams:
  P = peak message rate (msg/s)     → get this with 2× headroom
  S = average message size (bytes)
  RF = replication factor (default: 3)
  R = retention period (days)
  C = number of consumer groups reading each topic
  BW_NIC = network interface capacity per broker

Producer throughput per broker (inbound):
  T_in = P × S   (bytes/sec total ingested)

Replication traffic adds (RF−1) copies:
  T_replication = T_in × (RF − 1)

Consumer fetch traffic (assume C consumer groups):
  T_out = T_in × C

Total per-broker network:
  T_total = T_in + T_replication + T_out
  → Target: T_total ≤ 60% of NIC capacity
  → If 10GbE (1,200 MB/s), safe limit ≈ 720 MB/s per broker
Step 2 — Minimum broker count for network 
brokers_net = ceil(T_total / (0.6 × BW_NIC))

Example:
  P=500K msg/s, S=2KB, RF=3, C=5 consumer groups
  T_in         = 500K × 2KB    = 1,000 MB/s
  T_replication= 1,000 × 2     = 2,000 MB/s
  T_out        = 1,000 × 5     = 5,000 MB/s
  T_total      = 8,000 MB/s
  brokers_net  = ceil(8,000 / 720) = 12 brokers
Step 3 — Disk capacity per broker 
disk_total = T_in × retention_seconds × RF
disk_per_broker = disk_total / num_brokers + 20% headroom

Example (7-day retention, 12 brokers):
  disk_total    = 1,000 MB/s × 604,800s × 3 = 1.8 PB
  disk_per_broker = 1.8PB / 12 + 20%        = 180 TB

→ Use JBOD: 12 × 16TB NVMe per broker = 192TB ✓

Replication, Rack Awareness, min.ISR, RPO & RTO

Replication Factor Decision

RFBroker Failures ToleratedUse CaseCost
10Never in production. Only scratch/test topics.1× storage
20 (1 can be lost if it's not the only ISR)Acceptable only for non-critical, high-volume analytics topics where some loss is OK.2× storage
31Production standard. Survives 1 broker failure with min.ISR=2 and no data loss.3× storage
52Mission-critical topics (e.g. payments audit log). Rarely needed if cluster is healthy.5× storage
Rule: RF=3 + min.ISR=2 is the universal production standard. The only valid reason to go to RF=5 is regulatory data durability requirements (e.g., cannot lose any payment record under any 2-fault scenario).

Rack-Aware Replica Placement

Without rack awareness (3 brokers, same rack/AZ) 
Topic: payments, RF=3
  Partition 0: Leader=Broker1(AZ-A), Replica=Broker2(AZ-A), Replica=Broker3(AZ-A)

AZ-A failure → ALL replicas lost → partition OFFLINE → data loss possible
With rack awareness (broker.rack = AZ label) 
# broker.rack in server.properties:
broker.rack=us-east-1a   (Broker 1, 2)
broker.rack=us-east-1b   (Broker 3, 4)
broker.rack=us-east-1c   (Broker 5, 6)

Topic: payments, RF=3
  Partition 0: Leader=Broker1(AZ-A), Replica=Broker3(AZ-B), Replica=Broker5(AZ-C)
  Partition 1: Leader=Broker3(AZ-B), Replica=Broker5(AZ-C), Replica=Broker2(AZ-A)

AZ-A failure → Broker3 or Broker5 elected leader → partition STAYS ONLINE ✓
AZ-B failure → still 2 replicas alive (AZ-A + AZ-C) → ISR=2 ≥ min.ISR ✓
How rack-aware assignment works 
Kafka's rack-aware algorithm (kafka-topics.sh --create):
  1. Sort brokers, interleave by rack: A1, B1, C1, A2, B2, C2, ...
  2. For each partition, assign leader to next broker in the list
  3. Assign RF-1 replicas to the NEXT RF-1 brokers in the interleaved list
  → Guarantees each AZ gets exactly 1 replica per partition (when RF = AZ count)

Verify placement:
  kafka-topics.sh --describe --topic payments --bootstrap-server :9092
  # Check Replicas and Isr columns — IDs should span different rack groups

min.insync.replicas and the Availability vs Durability Tradeoff

Configuration pairing:
  acks=all + min.insync.replicas=2 + replication.factor=3

What this means:
  Producer write is only ACK'd after 2 of 3 replicas have persisted it.
  If only 1 replica is in-sync, the broker returns NOT_ENOUGH_REPLICAS error.
  Producer retries → write waits until a second broker recovers.

Scenarios:
  All 3 up:  ISR={1,2,3} → writes proceed, can lose 1 broker with 0 data loss
  1 down:    ISR={1,2}   → writes proceed (2 ≥ min.ISR=2), 1 more failure = stall
  2 down:    ISR={1}     → writes BLOCKED (1 < min.ISR=2) → producer gets error
             ↑ This is AVAILABILITY sacrificed to guarantee DURABILITY

Tuning for different topics:
  Topic type               min.ISR  acks  Tradeoff
  ─────────────────────────────────────────────────
  Payments, audit (PCI)      2       all  Durability over availability
  Notifications, alerts      1       1    Availability over durability
  Analytics (can replay)     1       1    Maximum throughput
  Exactly-once transactions  2       all  Required for EOS

Partition Count Strategy

Why Partitions Matter

  • Parallelism unit: A consumer group can have at most as many active consumers as there are partitions. Too few partitions = consumers idle.
  • Throughput unit: Each partition can sustain ~10–100 MB/s depending on disk type. More partitions = more parallelism = more total throughput.
  • Ordering unit: Kafka guarantees order within a partition only. If strict global ordering is required, you must use 1 partition (at the cost of all parallelism).
  • Irreversible increase: Partitions can only be increased, never decreased. Adding partitions redistributes which keys go to which partition, breaking any key-based ordering guarantee.

Partition Sizing Formula

Formula:
  partitions = max(
    ceil(target_producer_throughput / throughput_per_partition),
    ceil(target_consumer_throughput / throughput_per_partition),
    desired_consumer_parallelism
  )

Throughput per partition benchmark (rule of thumb):
  NVMe SSD:           50–100 MB/s per partition
  SAS/SATA SSD:       20–50 MB/s per partition
  Spinning disk HDD:  5–10 MB/s per partition

Example:
  Payments topic:
    Producer rate:     200 MB/s
    Consumer groups:   3 (fraud, accounting, audit)
    Max consumer rate: 100 MB/s per group
    Desired consumers per group: 12

  Throughput: ceil(200/50) = 4 partitions — insufficient
  Consumer parallelism: 12 per group
  Final: max(4, 12) = 12 partitions

  Round to a multiple that divides evenly by consumer count
  and is a round number: 12 or 24 ✓
Over-partitioning costs: Each partition = a file descriptor on disk + 1 thread in the leader epoch tracker + memory for replication state. A broker with 10,000 partitions as leader may need 2–3 GB heap just for partition metadata. Confluent recommends max 4,000 partitions per broker as a practical ceiling (ZK). KRaft raises this to millions.

Partition Count Rules of Thumb

GuidelineWhy
Start conservatively (12–24 per topic). You can always increase.Partition count is one-way. It's easier to add than to deal with key distribution changing after an increase.
Partition count should be a multiple of the number of consumers AND the number of brokers.Ensures even distribution. E.g., 6 brokers + 12 consumers → 24 partitions (24/6=4 per broker, 24/12=2 per consumer).
If order matters within a logical group, ensure messages with the same key go to the same partition. Don't change partition count after keys are in use.Kafka's default partitioner uses hash(key) % numPartitions. Changing numPartitions changes hash targets → ordering broken for existing keys.
For compacted topics, keep partition count moderate (12–48). Don't over-partition compacted topics.Compaction runs per partition. Too many small partitions → more compaction threads → more resource overhead per broker.
Global aggregate: aim for <4,000 partitions per broker (ZK mode), <50,000 (KRaft mode).Controller overhead scales with total partition count. ZK chokes above ~200K cluster-wide.

Retention Strategy

Retention Configuration Decision

Three axes of retention:
  1. Time:  log.retention.ms      (default: 7 days = 604800000)
  2. Size:  log.retention.bytes   (default: -1 = unlimited per partition)
  3. Both:  Kafka deletes segments that violate EITHER condition

Compaction:
  cleanup.policy=compact → keep only last value per key (tombstone = null)
  cleanup.policy=delete,compact → both (compact + time/size delete)

Per-topic override (always prefer per-topic over cluster defaults):
  kafka-configs.sh --bootstrap-server :9092 \
    --entity-type topics --entity-name payments.v1 --alter \
    --add-config 'retention.ms=604800000,retention.bytes=107374182400'
Topic CategoryRetention PolicyRationale
Payments / financial transactions7 days time + unlimited size. Archive to S3 via S3 Sink for 7 years.Regulatory: PCI-DSS / data retention laws require long-term storage. Kafka is the hot layer; S3 is cold archive.
CDC (Debezium database changes)cleanup.policy=compact — keep latest state per key.Consumers that restart late can reconstruct current state from compacted topic without needing full history.
Application events (notifications, emails)3–7 days deleteLong enough for consumers to recover from outages. No long-term value.
Analytics / data pipeline1–3 days. Sink to data warehouse via Kafka Connect.High volume. Data lands in warehouse; Kafka is transit, not storage.
Audit logs30 days time + compact. Long-term: S3 or WORM storage.Audit trails need immutability and long retention. Kafka + S3 Sink + S3 Object Lock.
Tiered Storage (Kafka 3.6+): Set log.local.retention.ms=86400000 (1 day local) and log.retention.ms=2592000000 (30 days total). Kafka keeps 1 day hot on disk, automatically offloads older segments to S3/GCS. Consumers can still fetch from remote storage transparently. Drastically reduces local disk requirements for long-retention topics.

Producer & Consumer Best Practices

Producer Best Practices

PracticeConfiguration / CodeWhy
Always use acks=all for critical data acks=all Ensures the write is on all min.ISR replicas before ACK. Prevents data loss on broker crash.
Enable idempotent producer enable.idempotence=true (implied by acks=all + retries>0 in modern clients) Assigns PID + sequence number per partition. Broker deduplicates retried messages. Exactly-once on the produce side.
Tune batch size and linger batch.size=65536 (64KB), linger.ms=20 linger.ms=0 (default) sends immediately — tiny batches, terrible throughput. 20ms allows batching without visible latency impact for most use cases.
Enable compression compression.type=lz4 (balanced) or zstd (high ratio) Reduces network and disk I/O. CPU cost is usually worth it. lz4 is fastest; zstd best compression ratio.
Size buffer.memory appropriately buffer.memory=134217728 (128MB) for high-throughput If buffer is full, send() blocks for max.block.ms (60s default). Set buffer large enough to absorb broker slowdowns.
Set retry budget with backoff retries=Integer.MAX_VALUE, retry.backoff.ms=100, delivery.timeout.ms=120000 Retries must stay bounded. delivery.timeout.ms is the total time budget. Infinite retries within a delivery window = effectively "keep trying for 2 min".
Never catch and swallow send callbacks Always check RecordMetadata and Exception in callback Silent discard of send errors causes data loss with no visibility. Always log or alert on send failures.
Use Schema Registry for all topics Avro or Protobuf serializers with SR URL Decouples schema from message. Enables schema evolution without breaking consumers. Prevents serialization bugs going to production.
Choose meaningful partition keys E.g., customerId or orderId, not random UUID Key determines partition → ordering and co-location of related events. Hot partitions occur if key cardinality is low (e.g., country code). Use key+secondary if cardinality is low.
Avoid synchronous sends in latency-sensitive paths Use async send with callback, not future.get() future.get() blocks until ACK — one message at a time. Async allows pipelining of up to max.in.flight.requests.per.connection (5) batches.

Consumer Best Practices

PracticeConfiguration / CodeWhy
Disable auto-commit; commit manually after processing enable.auto.commit=false; call consumer.commitSync() after processing batch Auto-commit can commit offsets before processing is complete, causing data loss on crash. Manual commit ensures at-least-once semantics.
Design for idempotent processing Use database upsert, deduplication by message ID, or external idempotency key store At-least-once delivery means duplicates will occur (rebalance, crash). Idempotent consumers make duplicates harmless.
Use cooperative-sticky rebalance assignor partition.assignment.strategy=CooperativeStickyAssignor Eager (default) rebalance revokes ALL partitions from ALL consumers, causing lag spike. Cooperative rebalance only moves partitions that need to change. Critical for high-partition-count topics.
Tune max.poll.interval.ms to match processing time max.poll.interval.ms=600000 (10 min) for slow processors If poll() is not called within this interval, the consumer is kicked from the group. Set to 2× worst-case processing time for a full batch.
Control batch size with max.poll.records max.poll.records=500 (default 500; reduce for slow processors) If processing 500 records takes longer than max.poll.interval.ms, reduce records per poll so the consumer can heartbeat between batches.
Separate heartbeat thread from processing thread This is built into modern Kafka clients (Java 2.x+): heartbeat runs on a background thread Old clients (0.x) had single-threaded poll → long processing = missed heartbeat. Modern clients handle this, but you still must call poll() regularly to trigger rebalance notifications.
Handle deserialization failures with a DLQ Try/catch DeserializationException, produce to topicname.dlq with original bytes + headers A single bad message can halt the entire partition if not handled. DLQ allows the pipeline to continue and investigate errors separately.
Monitor lag as a health metric, not throughput Alert on growing lag (rate > 0), not just lag magnitude A consumer with lag=10,000 is fine if it's stable. A consumer with lag=100 that's growing is the alert. Use Burrow or kminion for STALLED state detection.
Set isolation.level=read_committed for transactional topics isolation.level=read_committed Without this, consumers read uncommitted transactional messages (including aborted ones). Essential if consuming from a topic written by a transactional producer.

DR Solution & Primary/DR Flip

DR Architecture with MirrorMaker 2

Active-Passive Topology (recommended for most cases) 
                    ┌─────────────────────────────────┐
  Applications ────▶│   PRIMARY (DC-A / AZ: 3 zones)  │
  (Read/Write)      │   6 brokers, RF=3, rack-aware    │
                    └─────────────┬───────────────────┘
                                  │ MirrorMaker 2
                                  │ Replicates: topics + consumer offsets + ACLs
                                  ▼
                    ┌─────────────────────────────────┐
                    │   DR (DC-B / separate region)    │
                    │   6 brokers, RF=3, rack-aware    │
                    │   Topics prefixed: dc-a.*        │
                    └─────────────────────────────────┘

MirrorMaker 2 runs in DC-B (consumes from DC-A, writes locally).
Runs as Kafka Connect cluster (distributed mode, 3+ workers) for HA.
Active-Active Topology (bidirectional) 
  DC-A ←──── MM2 ─────▶ DC-B

  DC-A topics: dc-b.events (mirrored FROM DC-B)
  DC-B topics: dc-a.events (mirrored FROM DC-A)

  Use case: geo-distributed producers in both DCs.
  Challenge: consumers must read from both sets of topics and deduplicate.
  Complexity: high. Avoid unless truly required.
Offset sync: MM2 syncs consumer group offsets every 60s by default (sync.group.offsets.interval.seconds=60). Reduce to 10s for lower RPO. The last ~10s of messages before failover may be replayed — all consumers must be idempotent.

How to Flip Primary and DR Sides

  1. Pre-conditions before the flip

    MM2 replication lag should be near-zero. Verify with kafka-consumer-groups.sh --describe --group mm2-... on the DR cluster — lag should be <1000 records. DR cluster health: URP=0, all brokers running.

  2. Planned failover (maintenance window) — stop producers on DC-A 
    # Signal applications to stop writing to DC-A
# OR: change load balancer routing weight to 0 for DC-A endpoint
# Wait until MM2 lag reaches 0 (all data replicated):
kafka-consumer-groups.sh --bootstrap-server dc-b:9092 \
  --describe --group mm2-dc-a-to-dc-b
# Lag column should be 0 for all partitions
  3. Translate consumer offsets 
    # MM2 has synced offsets automatically. Verify with:
kafka-consumer-groups.sh --bootstrap-server dc-b:9092 \
  --describe --group your-consumer-group
# Should show correct offsets under dc-a.* topic names

# If using IdentityReplicationPolicy (no topic prefix):
# Consumers will resume at same offsets after DNS update
  4. Switch DNS / Load Balancer to DC-B

    Update bootstrap.servers DNS record from DC-A brokers to DC-B brokers. Use short TTL (30s) for fast propagation. Alternatively use a virtual IP or service mesh endpoint.

  5. Validate on DC-B

    Monitor consumer lag on DC-B. Confirm producers are writing to DC-B. Confirm consumers are reading from DC-B with correct offsets. Check URPs = 0.

  6. Reverse MM2 (DC-B → DC-A)

    Once DC-B is confirmed primary, set up a new MM2 replication job: DC-B → DC-A. This allows failback when DC-A recovers. Prevents DC-A from going stale if it's needed again.

  7. Failback (when DC-A is healthy) 
    Failback is the same procedure in reverse:
  1. Ensure DC-B → DC-A MM2 lag ≈ 0
  2. Stop producers on DC-B
  3. Wait for full sync
  4. Translate offsets (or use synced offsets)
  5. Switch DNS back to DC-A
  6. Re-establish DC-A → DC-B MM2 replication
Split-brain risk during flip: Between steps 2 and 4, no producers should write to EITHER cluster. If applications auto-reconnect before DNS propagates, you may get writes to both clusters that are never merged. Use an application-level "maintenance mode" flag or circuit breaker to hold producers during the DNS TTL window.

OS Patching & Kafka Version Upgrades

Zero-Downtime OS Patching (Rolling Procedure)

  1. Pre-patch health gate 
    # Must be ZERO before touching any broker:
kafka-topics.sh --bootstrap-server :9092 \
  --describe --under-replicated-partitions
# If non-zero: DO NOT PROCEED. Fix URP first.

# Check active controller:
kafka-metadata-quorum.sh --bootstrap-server :9092 describe
# or: check JMX ActiveControllerCount = 1
  2. Pick a non-controller broker first

    Patching the controller causes a controller election (brief ~1s in KRaft). Do controller last to minimize disruption.

  3. Perform controlled shutdown 
    # Kafka will move its leaders to other brokers before stopping
# (requires controlled.shutdown.enable=true — default: true)
systemctl stop kafka

# Verify the broker is gone from cluster:
kafka-broker-api-versions.sh --bootstrap-server :9092 2>&1 | \
  grep -v "BROKER_ID_TO_PATCH"
  4. Apply OS patch / kernel update 
    # RHEL/CentOS:
yum update -y --security && reboot

# Ubuntu:
apt-get upgrade -y && reboot
  5. Post-reboot: start Kafka, wait for ISR recovery 
    systemctl start kafka

# Poll URP until 0 (max wait: 10 minutes):
while true; do
  URP=$(kafka-topics.sh --bootstrap-server :9092 \
        --describe --under-replicated-partitions | wc -l)
  echo "URP: $URP"
  [ "$URP" -eq 0 ] && break
  sleep 15
done
  6. URP = 0 → proceed to next broker

    Only move to the next broker once the previous one has fully rejoined ISR. Never patch two brokers simultaneously.

  7. Controller broker last

    When patching the controller broker, KRaft elects a new controller in <1s. ZK-based clusters: allow 10–30s for election. Validate ActiveControllerCount=1 after restart.

Automation: Encode this as an Ansible playbook with serial: 1, pre/post URP check tasks, and max_fail_percentage: 0. See the IaC page for the full playbook. Never run ad-hoc patching — always use the playbook.

Kafka Version Upgrade (Two-Phase Rolling Upgrade)

Why Two Phases? 
Phase 1: Upgrade binary (JAR files) — REVERSIBLE
  All brokers run new binary but communicate using the OLD
  inter.broker.protocol.version (IBP). Mixed-version cluster is safe.

Phase 2: Bump IBP and log.message.format.version — IRREVERSIBLE
  Once IBP is set to the new version on all brokers, you cannot
  downgrade without wiping broker state. Do this only after Phase 1
  has been stable in production for ≥ 48 hours.
Phase 1 — Binary upgrade (per broker, rolling) 
# In server.properties: KEEP OLD VALUES during phase 1:
inter.broker.protocol.version=3.6   ← don't change yet
log.message.format.version=3.6      ← don't change yet

For each broker:
  1. URP check (must be 0)
  2. Stop Kafka: systemctl stop kafka
  3. Swap JAR: update symlink to new kafka binary
  4. Start Kafka: systemctl start kafka
  5. Verify broker joins, URP returns to 0
  6. Move to next broker

After all brokers on new binary:
  - Test for 48h, monitor metrics
  - If rollback needed: repeat with old binary (just swap JAR)
Phase 2 — IBP bump (cluster-wide, IRREVERSIBLE) 
# Update ALL brokers' server.properties simultaneously
# (via Ansible/Strimzi), then rolling restart:
inter.broker.protocol.version=3.7   ← new version
log.message.format.version=3.7      ← new version

# Or use dynamic config for kafka cluster-level (Kafka 3.3+):
kafka-configs.sh --bootstrap-server :9092 \
  --entity-type brokers --entity-default --alter \
  --add-config inter.broker.protocol.version=3.7

⚠️  POINT OF NO RETURN: After all brokers use IBP=3.7,
    downgrading requires wiping and restoring from replica.

Password & Certificate Rotation

SASL/SCRAM Password Rotation (Zero Downtime)

How SCRAM credential rotation works 
SCRAM credentials are stored in ZooKeeper (ZK mode) or
the Kafka cluster itself (KRaft: __cluster_metadata).
No broker restart is required to update SCRAM passwords.

Step 1: Add NEW credential alongside existing one
  kafka-configs.sh --bootstrap-server :9092 \
    --entity-type users --entity-name payments-producer \
    --alter \
    --add-config 'SCRAM-SHA-512=[password=new-secret-password]'

  ↑ This REPLACES the existing credential immediately.
  ↑ Client using old password will FAIL after this step.

  → Therefore: update clients BEFORE rotating the broker config.

Correct zero-downtime sequence:
  1. Generate new password → store in Vault
  2. Deploy new client config (new password) to a SUBSET of pods
     (canary: 1 of 10 pods gets new password)
  3. Verify canary pods authenticate successfully
  4. Run kafka-configs.sh to update broker with new password
  5. Deploy new password to ALL remaining pods
  6. Verify all clients connected
  7. (Optional) Remove old password from Vault/secrets manager
Strimzi KafkaUser automation 
# Update KafkaUser CR with new password secret reference:
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaUser
metadata:
  name: payments-producer
spec:
  authentication:
    type: scram-sha-512
    password:
      valueFrom:
        secretKeyRef:
          name: payments-producer-secret  # K8s Secret
          key: password

# Strimzi User Operator watches this CR.
# When the K8s Secret is rotated by ESO (from Vault),
# Strimzi automatically updates the SCRAM credential on the broker.
# Clients using the new password from the Secret can authenticate immediately.

TLS Certificate Rotation (Zero Downtime)

Problem: broker has a single keystore → swap causes downtime 
Solution — Double-cert trick (dual-cert keystores):
  Phase 1: Add new cert to keystore ALONGSIDE old cert
    keytool -importcert -alias new-cert -keystore kafka.keystore.jks
    → Brokers now have both old and new cert
    → Clients trust EITHER cert (truststore has both CA certs)

  Phase 2: Rolling restart (one broker at a time)
    → Each broker restarted presents new cert as primary
    → Clients connected to other brokers still have valid connections

  Phase 3: Remove old cert from keystores
    keytool -delete -alias old-cert -keystore kafka.keystore.jks
    → Clean up old cert after all brokers have rotated

  Phase 4: Remove old CA from truststores (if CA changed)
    → Only after all clients have updated to trust new CA
cert-manager + Strimzi automation 
# cert-manager issues certificates with 90-day lifetime.
# Strimzi detects cert expiry 30 days in advance and performs
# an automatic rolling restart with the double-cert trick.
# Zero operator action required.

# Monitor cert expiry:
kubectl get certificates -n kafka-prod
# READY=True, EXPIRY should be >30 days away

# Force immediate rotation (for compromised cert):
kubectl annotate certificate kafka-broker-cert \
  cert-manager.io/issuer-kind=ClusterIssuer \
  force-renew="$(date)" -n kafka-prod

Vault-Based Secrets Lifecycle

All Kafka credentials flow:
  Vault PKI     → TLS certificates (auto-renewed via cert-manager)
  Vault KV      → SCRAM passwords (rotated via ESO + KafkaUser CR)
  Vault KV      → Schema Registry API keys
  Vault KV      → Kafka Connect connector credentials (DB passwords, S3 keys)

Rotation schedule:
  TLS certs:            90-day lifetime, auto-rotated at day 60
  SCRAM passwords:      Rotate every 90 days OR on personnel change
  API keys (if used):   Rotate every 180 days
  Service account keys: Rotate every 90 days

Audit:
  Vault audit log captures every secret access.
  Feed to SIEM (Splunk/Elastic) to detect anomalous credential reads.
  Alert on: secret read from unexpected IP, high-frequency reads,
            reads outside business hours.

HA for Platform Components

Schema Registry HA

Architecture:
  3 Schema Registry instances behind a Load Balancer (or K8s Service)
  One instance is the LEADER (handles writes)
  Others are FOLLOWERs (serve reads from their cache)

Leader election:
  Uses Kafka's own consumer group protocol on the _schemas topic.
  If leader dies, a follower becomes leader in seconds.

Configuration for multiple instances:
  kafkastore.bootstrap.servers=kafka:9092
  listeners=http://0.0.0.0:8081
  host.name=sr-instance-1.internal  # unique per instance

Client resilience:
  url=http://sr-1:8081,http://sr-2:8081,http://sr-3:8081
  ↑ Client caches schema locally — continues working even if ALL SR instances die,
    as long as no new schemas are needed.

Availability:
  Load balancer health check: GET /subjects (200=healthy)
  PDB: maxUnavailable=1 (Kubernetes)
  Anti-affinity: spread across 3 AZs

Kafka Connect HA

Distributed mode: 3+ worker nodes form a Connect cluster
  - Connector/task assignments distributed across workers
  - If a worker dies: tasks reassigned to surviving workers in <60s
  - Internal topics (connect-configs, connect-offsets, connect-status):
      replication.factor=3, min.insync.replicas=2

Worker configuration for HA:
  group.id=hsbc-connect-cluster           # All workers share same group
  offset.flush.interval.ms=10000          # 10s offset flush
  config.storage.topic=connect-configs    # RF=3
  offset.storage.topic=connect-offsets    # RF=3, 25 partitions
  status.storage.topic=connect-status     # RF=3

K8s: Run as Deployment (not StatefulSet — workers are stateless)
  replicas: 3
  PDB: maxUnavailable=1
  Anti-affinity: spread across AZs

DLQ for resilience:
  errors.tolerance=all
  errors.deadletterqueue.topic.name=my-connector.dlq
  → Failed records go to DLQ; connector continues processing

ksqlDB HA

Clustered ksqlDB (production):
  3+ ksqlDB server nodes with shared Kafka bootstrap.servers

State store HA:
  num.standby.replicas=1 (or 2 for critical queries)
  → Each partition's RocksDB state store has a warm standby
    on another node. Failover is fast (<10s).

Internal topics:
  _confluent-ksql-default__command_topic     RF=3  (query DDL log)
  _confluent-ksql-*-ksql_processing_log      RF=3  (processing errors)

Load balancer:
  ksqlDB servers are stateful — use sticky sessions for pull queries.
  Push queries (EMIT CHANGES) can go to any server.

K8s: StatefulSet (for stable hostname + PVC per node)
  replicas: 3
  volumeClaimTemplates: RocksDB on fast SSD (8Gi+)
  PDB: maxUnavailable=1
  Anti-affinity: spread across AZs

REST Proxy HA

Challenge: REST Proxy is stateful for consumer sessions.
  Consumer instance is created on a specific proxy server.
  Subsequent poll/commit/delete MUST go to the same server.

Solutions:
  Option A: Sticky sessions at load balancer (simplest)
    nginx: ip_hash or cookie-based sticky
    AWS ALB: sticky sessions (duration-based cookie)
    → If the proxy instance dies, consumer session is lost.
      Client must re-create consumer instance. Acceptable for most use cases.

  Option B: Only use REST Proxy for producing (stateless path)
    POST /topics/{topic} is fully stateless — any server handles it.
    Use native Kafka clients for consuming (avoids sticky routing problem entirely).

  Option C: Client-side session affinity
    REST Proxy returns X-Instance-ID header after consumer create.
    Client tracks which proxy URL hosts its consumer instance.
    If that proxy is down: catch 404, re-create consumer, seek to last committed offset.

K8s: Deployment (stateless for produce path)
  replicas: 3
  PDB: maxUnavailable=1
  Anti-affinity: spread across AZs
  consumer.instance.timeout.ms: 300000 (5 min leak prevention)

KRaft Controller HA

KRaft quorum: 3 or 5 dedicated controller nodes

Quorum fault tolerance:
  3 controllers → tolerate 1 failure (need 2/3 for quorum)
  5 controllers → tolerate 2 failures (need 3/5 for quorum)

Controller failover:
  Active controller fails → follower with most up-to-date log becomes leader
  Failover time: typically <500ms (Raft is fast)
  During failover: no new metadata operations, but existing produce/consume continues

Controller nodes (KRaft-only clusters):
  process.roles=controller          ← dedicated controllers
  node.id=1 (2, 3)
  controller.quorum.voters=1@ctrl-1:9093,2@ctrl-2:9093,3@ctrl-3:9093

  K8s: StatefulSet, 3 replicas, 1 per AZ
  No Kafka data stored on controllers (they only store metadata)
  Modest hardware: 4 vCPU, 16 GB RAM, 100 GB SSD is sufficient

Monitoring:
  kafka.controller:type=KafkaController,name=ActiveControllerCount
  kafka-metadata-quorum.sh --describe → shows leader, lag, offset

ZooKeeper HA (legacy clusters)

ZK ensemble sizing:
  3 nodes → tolerate 1 failure
  5 nodes → tolerate 2 failures (recommended for critical production)

ZK requires majority for writes:
  3-node with 1 down → still writable (2/3 = majority)
  3-node with 2 down → read-only at best, usually unavailable

ZK node hardware: modest — 4 vCPU, 8 GB RAM, fast SSD for txn log
ZK txn log: dedicated disk, separate from snapshot disk
  dataLogDir=/data/zk/txlog     ← fast dedicated SSD
  dataDir=/data/zk/data         ← can share with OS disk

ZK config (zoo.cfg):
  tickTime=2000
  initLimit=10
  syncLimit=5
  server.1=zk-1:2888:3888
  server.2=zk-2:2888:3888
  server.3=zk-3:2888:3888

Monitoring:
  ZK 4-letter commands: echo ruok | nc zk-1 2181
  Prometheus zkmetrics via zookeeper_exporter
  Alert on: OutstandingRequests > 10, AvgLatency > 10ms

⚠️  Plan ZK → KRaft migration. ZK support removed in Kafka 4.0.
