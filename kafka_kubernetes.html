💡
HSBC context: Many financial institutions run Kafka on VMs for predictable performance and regulatory isolation. K8s adoption for Kafka is growing but requires careful storage tuning. Key concern: noisy-neighbor effects from shared K8s nodes affecting Kafka I/O latency. Use dedicated node pools (node selectors/taints) for Kafka workloads.
🎛️Strimzi Operator
STRIMZI OPERATOR ARCHITECTURE: ────────────────────────────────────────────────────────────────── kubectl apply -f kafka.yaml (Kafka CRD) │ ▼ Strimzi Cluster Operator (Deployment in kafka namespace) Watches: Kafka, KafkaTopic, KafkaUser, KafkaConnect, KafkaMirrorMaker2... │ ├── Reconciles Kafka CR → Creates/Updates: │ StatefulSet (brokers) │ StatefulSet (ZK or KRaft controllers) │ Services (headless, external) │ ConfigMaps (server.properties generated from CR) │ Secrets (TLS certs, JAAS configs) │ PodDisruptionBudgets │ ├── Topic Operator: │ Watches KafkaTopic CRs → calls Kafka AdminClient │ Creates/updates topics in Kafka │ └── User Operator: Watches KafkaUser CRs → creates SCRAM users, ACLs Stores credentials in K8s Secrets

Cluster Operator

Main operator component. Watches Kafka and related CRDs. Manages the full lifecycle: creation, configuration updates, rolling upgrades, TLS cert rotation, cluster scaling. Runs the reconciliation loop.

Topic Operator

Watches KafkaTopic CRDs and reconciles with actual Kafka topics via AdminClient. Enables "topic as code" — define topics in Git, apply to cluster. Supports bidirectional sync (KafkaTopic CR → Kafka and Kafka → KafkaTopic CR).

User Operator

Watches KafkaUser CRDs. Creates/manages SCRAM users and ACLs. Stores generated passwords in K8s Secrets — applications reference the Secret, not hardcoded credentials.

Automated TLS

Strimzi generates and manages all TLS certificates: cluster CA, client CA, broker certs, ZK certs. Handles automatic rotation (configurable). All inter-component communication is TLS by default — zero manual cert management.

💾Storage & PVCs

StorageClass Selection

Use SSDs, not HDDs or network-attached storage (NFS, EFS). For AWS: gp3 (3000 IOPS base, scalable) or io2 (high IOPS). For GCP: pd-ssd. For Azure: Premium_LRS. Avoid shared storage — Kafka segments are sequential writes; shared storage adds latency.

JBOD Storage

Strimzi supports JBOD (multiple volumes per broker). Define multiple volumes in the CR — each gets its own PVC. Kafka distributes partition logs across volumes for higher I/O parallelism. Preferred over RAID from Kafka's perspective (Kafka handles replication, RAID adds write amplification).

PVC Sizing

Size = peak_throughput × retention_period × replication_factor ÷ num_brokers + 30% headroom. Always provision more than you think you need — PVC expansion is disruptive (may require pod restart). Alert at 70% usage.

deleteClaim: false

Critical setting — always set deleteClaim: false in production. If set to true, deleting the Kafka CR (even accidentally) deletes all PVCs and data. With false: data survives Kafka CR deletion. PVCs must be manually cleaned up.

🚨
Local PersistentVolumes (LocalPV): For best performance, use K8s Local PersistentVolumes — PVs backed by actual local NVMe disks on the node. This gives bare-metal disk performance. Trade-off: the pod MUST run on the same node as the disk (no live migration). Use with strict node assignment via StatefulSet volume claim templates + node selectors. Strimzi supports this out of the box.

🛡️Pod Disruption Budgets
STRIMZI AUTO-CREATES PDB: ────────────────────────────────────────────────────────────────── # For 3 Kafka brokers, Strimzi creates: apiVersion: policy/v1 kind: PodDisruptionBudget metadata: name: kafka-production-kafka spec: maxUnavailable: 1 # At most 1 broker can be disrupted at once selector: matchLabels: strimzi.io/name: kafka-production-kafka EFFECT: - kubectl drain node-1 → Kubernetes checks PDB before evicting pods - If Kafka broker on node-1 is the only disruption → allowed (maxUnavailable=1) - If another broker is already down → kubectl drain BLOCKED until first recovers - Prevents accidental multi-broker failure during K8s node maintenance PDB TUNING: - maxUnavailable=1 is standard for 3-broker clusters (RF=3, tolerates 1 failure) - For 5-broker clusters: maxUnavailable=2 is safe - ZooKeeper PDB: maxUnavailable=1 (ZK 3-node needs 2 for quorum)
✅
Interview tip: PDBs protect against voluntary disruptions (node drain, cluster upgrades). They do NOT protect against involuntary disruptions (node crash, OOM kill). For involuntary failures: replication factor + min.insync.replicas are the protection. PDB and RF/ISR are complementary, not alternatives.
🔄Rolling Upgrades via Strimzi
1
Update Kafka version in CR
Change spec.kafka.version: 3.7.0 to 3.8.0. Also update inter.broker.protocol.version and log.message.format.version to the old version first (two-phase upgrade).
2
Strimzi Cluster Operator detects the change
Operator reconciliation loop detects CR diff. Plans a rolling update of the StatefulSet. Uses the same rolling restart protocol as manual operations (one pod at a time, health check before next).
3
Pod restart with health checks
Strimzi waits for each broker pod to be Ready (readiness probe passes). Readiness probe checks broker is accepting connections. Additionally checks UnderReplicatedPartitions = 0 before proceeding to the next broker.
4
Update inter-broker protocol version
After all brokers are on new Kafka version: update inter.broker.protocol.version to the new version in the CR. This triggers another rolling restart to enable new protocol features.
5
Verify and monitor
Watch operator logs: kubectl logs deployment/strimzi-cluster-operator -n kafka -f. Watch for URP alerts during upgrade. Whole process is automated — no manual intervention needed.

🎯Kubernetes Q&A
Q A Kafka broker pod is evicted due to memory pressure on the node. What happens to data and how do you prevent this? Mid
What happens to data: The broker pod dies. Other brokers detect it via heartbeat/ZK timeout. Leader elections occur for partitions led by the evicted broker. Followers in ISR become new leaders. Data written to the evicted broker's ISR partitions is safe (replicated). Data NOT yet replicated to any ISR follower (possible with acks=1) could be lost if the evicted broker's disk is also gone.

Prevention:
  1. Set requests=limits (Guaranteed QoS): K8s evicts BestEffort pods first, then Burstable. Guaranteed QoS pods are only evicted as last resort (system critical processes at risk). Kafka brokers should be Guaranteed QoS.
  2. Dedicated node pool: Use dedicated K8s nodes for Kafka (node selector/taints). No other workloads compete for memory on broker nodes.
  3. Set correct JVM heap: Oversized heap → pod memory limit approached → OOM kill by K8s. Kubernetes memory limit = JVM heap + off-heap (page cache, NIO buffers, metaspace). Set limit ~= 2× JVM heap.
  4. PDB: PDB prevents voluntary eviction of too many pods at once but doesn't prevent OOM eviction (involuntary).
Q How does Strimzi solve the "advertised.listeners" problem that makes Kafka tricky on K8s? Mid
The core problem: Kafka brokers return their advertised.listeners in metadata responses. After a client bootstraps, it's told "connect to broker-1 at broker1.kafka.svc.cluster.local:9092." If the client is external (outside the cluster), this internal DNS name is unreachable.

Strimzi's solution: It configures EACH broker with the correct listener for each access route:
  • Internal listener: kafka-production-kafka-0.kafka-production-kafka-brokers.kafka.svc.cluster.local:9092
  • External listener: auto-detected external IP/DNS from the LoadBalancer/NodePort service for that specific pod
Each pod gets its own Service (e.g., kafka-production-kafka-0) with its own external address. Strimzi watches for the external IP assignment from the cloud provider LB and patches each broker's advertised.listeners with the correct per-broker external address. Clients bootstrapping externally get metadata pointing to per-broker external addresses — and can connect to any broker directly.
