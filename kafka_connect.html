Worker

A JVM process running the Connect framework. Manages connector lifecycle, assigns tasks, provides REST API. Workers form a cluster via Kafka consumer group protocol — no ZooKeeper needed.

Connector

A logical job configuration. Defines what to connect (DB, S3, etc.), how many tasks, and all connector-specific settings. A connector is just metadata — tasks do the actual work.

Task

The unit of work. A connector splits its work into N tasks (parallelism). Each task runs in a worker thread. Source tasks read from external system → produce to Kafka. Sink tasks consume from Kafka → write to external system.

Converter

Serializes/deserializes data between Connect's internal SourceRecord/SinkRecord format and Kafka bytes. Common: AvroConverter (uses SR), JsonConverter, StringConverter.

Transforms (SMTs)

Lightweight per-record transformations applied in the Connect pipeline — before writing to Kafka (source) or after reading from Kafka (sink). Examples: field masking, routing, timestamp conversion.

Config, Offsets, Status Topics

Connect stores all cluster state in Kafka topics. Config and status have replication factor ≥ 3 for HA. Offsets topic tracks source connector positions — highly written, use high replication and compaction.

Important: Task failure does NOT automatically cause the connector to fail or restart (by default). The connector remains in RUNNING state with a FAILED task — you get partial data loss until someone notices. Set up alerting on task failures via the REST API or connect-status topic monitoring.
📍Offset Management

Source Connector Offsets

Source connectors track their position in the source system (e.g., DB sequence number, file position, API cursor) in the connect-offsets Kafka topic. On restart, the connector resumes from the last committed offset — no data duplication (usually).

Sink Connector Offsets

Sink connectors are Kafka consumers — their offsets are stored in __consumer_offsets. The group ID is derived from the connector name. Sink connectors only commit Kafka offsets after successfully writing to the target system.

Offset Reset

To replay a source connector: delete its entries in connect-offsets (using a producer with null values for the offset keys) and restart the connector. For sink connectors: reset the consumer group offset using kafka-consumer-groups.sh.

Offset Commit Interval

Source tasks commit offsets to connect-offsets on a timer (offset.flush.interval.ms, default 60s). If a task fails before committing, it replays from the last committed position — at-least-once delivery for source connectors.

🚨Error Handling & Dead Letter Queue
SINK CONNECTOR ERROR HANDLING CONFIG: ───────────────────────────────────────────────────── { "errors.tolerance": "all", # "none" (fail fast) | "all" (skip+log) "errors.log.enable": "true", # log error details "errors.log.include.messages": "true", # include message payload in log "errors.deadletterqueue.topic.name": "my-sink-dlq", # DLQ topic "errors.deadletterqueue.topic.replication.factor": "3", "errors.deadletterqueue.context.headers.enable": "true" # add error context headers } DLQ MESSAGE HEADERS (with context.headers.enable=true): __connect.errors.exception.class.name → class of the exception __connect.errors.exception.message → error message __connect.errors.exception.stacktrace → full stack trace __connect.errors.offset → Kafka offset of failed record __connect.errors.connector.name → which connector failed __connect.errors.task.id → which task failed

errors.tolerance=none (default)

First error causes the task to fail and stop. Requires manual intervention to restart. Safe for critical pipelines where data loss or skipping is unacceptable.

errors.tolerance=all

Skip malformed or unwritable records, log them, optionally send to DLQ. Processing continues. Risk: silent data loss if DLQ is not monitored. Set up alerts on DLQ topic lag.

DLQ Topic

A Kafka topic where failed records are written along with error headers. Operations team can inspect, fix, and replay from the DLQ. Must be separate from source topics and monitored for consumer lag.

Retry Behavior

Connect does not have built-in exponential backoff for retries. Some connectors implement their own. For transient failures (DB unavailable), the task typically fails and requires restart — unlike producers which have built-in retry with backoff.

🔧Single Message Transforms (SMTs)

SMTs are stateless per-record transformations applied in the Connect pipeline. Not designed for heavy processing — use Kafka Streams or Flink for complex logic. SMTs are chained and applied in order.

SMTPurposeExample Use
MaskFieldReplace sensitive field values with null or literalMask PII before writing to Kafka: mask SSN, credit card
ReplaceFieldInclude/exclude/rename fieldsDrop internal DB columns before publishing
ExtractFieldExtract a nested field as the top-level valueExtract payload from CDC envelope
InsertFieldAdd a static or dynamic fieldAdd source system name or ingestion timestamp
TimestampConverterConvert timestamp formatsUnix epoch → ISO-8601 string
RegexRouterRoute records to different topics based on regexRoute by table name: orders_* → orders-topic
ValueToKeySet message key from a field in the valueUse customer_id field as partition key
FlattenFlatten nested structs into top-level fieldsNormalize deeply nested JSON from REST source
FilterDrop records matching a conditionDrop DELETE events from CDC stream
CastType cast individual fieldsCast string "123" → integer 123
⚠️
SMT limitations: SMTs are stateless and process one record at a time. They cannot: join streams, aggregate, look up external data, or do windowed operations. For those, use Kafka Streams, ksqlDB, or Flink downstream of the connector.
🎯Exactly-Once in Kafka Connect

Source Connectors

Source connectors are at-least-once by default (offsets committed periodically, not atomically with produce). Exactly-once requires the source system to be replayable AND the connector to use Kafka transactions (available in Connect 3.3+ with exactly.once.source.support=enabled).

Sink Connectors

Exactly-once for sinks requires the target system to be idempotent (e.g., upsert by primary key) or transactional. The sink connector commits Kafka offsets only after successful write. If write fails, re-reads same records — idempotency prevents duplicates.

Kafka 3.3+ EOS Source

Broker-side EOS for source connectors: Connect writes source offsets and Kafka records atomically in a transaction. Even if the worker crashes between write and commit, replay produces the same records — no duplicates.

Practical Advice

For most production cases: use at-least-once + idempotent sinks. True EOS adds complexity and latency. Design your sink to handle duplicates (upsert, dedup key) rather than relying on EOS guarantee.

🎯Concepts & Scenario Q&A
Q A Debezium Postgres connector task is FAILED and you see "replication slot does not exist" in logs. What happened and how do you fix it? Senior
Debezium uses a Postgres replication slot to receive WAL changes. Replication slots persist on the DB side and must be dropped manually. Common causes of "slot does not exist":
  • Someone manually dropped the slot on the DB — perhaps a DBA cleaning up what they thought was an old slot.
  • The slot was auto-dropped (unusual, but possible after DB restore from backup).
Fix:
  1. Stop the Debezium connector.
  2. Delete the connector's offset in connect-offsets topic (send a null-value message for the connector's partition key) — this resets its tracked position.
  3. Restart the connector — Debezium will create a new replication slot and take a new snapshot of the table before streaming changes.
  4. Expect a full table re-read (snapshot phase) — which may cause duplicate records downstream. Ensure sink is idempotent.
Prevention: Never manually drop a Debezium replication slot while the connector is active. Use the Debezium connector DELETE API instead — it cleans up the slot automatically.
Q How does Kafka Connect handle worker failure? What happens to running tasks? Mid
Connect workers form a consumer group (group ID = connect-cluster by default). When a worker fails:
  • The group coordinator detects the failure via missed heartbeats (session.timeout.ms).
  • A rebalance is triggered among surviving workers.
  • The failed worker's connectors and tasks are redistributed to surviving workers.
  • Tasks resume from their last committed offset in connect-offsets (source) or __consumer_offsets (sink).
  • There may be duplicate processing for records after the last commit — sink must handle duplicates.
Time to recover: Typically 30–90 seconds depending on session.timeout.ms (45s default) + rebalance time. Keep worker count ≥ 3 for HA.
Q When should you use Kafka Connect vs writing a custom Kafka producer/consumer? Mid
Use Kafka Connect when:
  • A proven connector exists for your source/sink system (JDBC, S3, Elasticsearch, Debezium CDC).
  • You need distributed, fault-tolerant ingestion/egress with minimal code.
  • The transformation logic is simple enough for SMTs.
  • You want centralized connector management via REST API.
Use custom producer/consumer when:
  • No connector exists for your system.
  • You need complex business logic, stateful processing, or joins during ingestion.
  • You need fine-grained control over error handling, batching, or protocol specifics.
  • The connector overhead (REST API, distributed state) is unnecessary for a simple point-to-point integration.
Hybrid approach: Use Debezium for CDC into Kafka, then Kafka Streams for transformation, then a sink connector to write to target. Best of both worlds.
