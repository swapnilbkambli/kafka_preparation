Defining SLOs for Kafka Platform

SLO Definition Template 
Service: Kafka Cluster (Production)
Window: 30-day rolling

SLO 1 — Producer Availability
  SLI: % produce requests with acks received / total produce requests
  Target: 99.95% (26 min downtime/month budget)
  Exclude: client-side errors (TOPIC_AUTH_FAILED, INVALID_MSG), planned maintenance

SLO 2 — Produce Latency
  SLI: % produce requests completing P99 < 200ms (end-to-end)
  Target: 99.9%
  Measurement: broker-side RequestTotalTimeMs.p99 for Produce

SLO 3 — Consumer Lag
  SLI: % of time all critical consumer groups lag < 10,000 records
  Target: 99.5%
  Scope: consumer groups tagged slo=critical

SLO 4 — Data Durability
  SLI: % of 1-min windows where UnderReplicatedPartitions = 0
  Target: 99.9%

SLO 5 — Platform Availability (Broker Uptime)
  SLI: % of time at least min.insync.replicas brokers available per partition
  Target: 99.99% (4.4 min/month)

Error Budget Policy

Error Budget = 100% - SLO Target 
Monthly error budget for 99.95% SLO:
  30 days × 24h × 60min = 43,200 min
  0.05% × 43,200 = 21.6 minutes of allowed downtime

Budget consumption triggers:
  >50% consumed (10.8 min)  → freeze non-critical changes
  >75% consumed (16.2 min)  → freeze ALL changes, postmortem required
  100% consumed (21.6 min)  → incident review, root cause to engineering
  Rolling 4-week <10% used   → team earns reliability investment credit

Burn Rate Alert (Prometheus):
  Fast burn (1h window):   rate > 14.4x budget burn rate → PagerDuty
  Slow burn (6h window):   rate > 6x budget burn rate   → Slack warning

PromQL:
  1 - (sum(rate(kafka_produce_errors_total[1h])) /
       sum(rate(kafka_produce_requests_total[1h])))
Error Budget Burn Rates: A 14.4x burn rate on a 1h window means you'll exhaust the monthly budget in 50 hours. Alert fast burns with tight windows AND slow burns with wide windows — you need both to avoid missing gradual degradation.

Failure Mode Analysis

Broker Failure

ScenarioImpactDetectionRecovery
Single broker crash Partitions where it was leader become unavailable until election (~30s ZK / <1s KRaft). Followers of remaining brokers continue serving. ActiveControllerCount drops event; UnderReplicatedPartitions spike; broker disappears from /brokers/ids Auto: Controller triggers leader election for offline partitions. Manual: restart broker, watch ISR recovery.
Broker disk full Broker stops accepting writes. Partitions it leads go offline. ISR shrinks across all its partitions. LogFlushOffsetCheckpointMs errors; DiskFreeSpace alert; kafka.log.Log errors in broker log Emergency: free space (delete old segments with kafka-delete-records.sh, adjust retention). Permanent: expand disk or add JBOD volume.
Broker GC pause (>30s) Session timeout causes controller to consider broker dead. ZK session expires. Leader elections for all broker's partitions. GCTimeMs > 10s alert; log shows "Broker heartbeat expired" Tune JVM heap. Switch to G1GC. Reduce -Xmx to leave more page cache. In extreme: add RAM.
All brokers crash Complete cluster unavailability. Producers get timeout/connection refused. Consumers stall. All broker metrics disappear. Monitoring goes silent. Restart in order: ZK (if applicable) → Controller/KRaft controllers → Brokers. Verify metadata integrity before accepting traffic.

Controller / ZooKeeper / KRaft Failure

ScenarioImpactRecovery TimeAction
Active Controller crash (ZK mode) No new leader elections or topic changes until new controller elected. Existing produce/consume continues for in-sync partitions. 6–30s (ZK session timeout) Any broker can become controller. Monitor ActiveControllerCount — must return to 1. If stuck: restart all brokers.
ZK quorum loss (minority) ZK still serves reads. No new controller elections possible. Existing state frozen but cluster operates. Auto when quorum restores Restore ZK quorum. ZK ensemble needs majority (3 nodes: need 2, 5 nodes: need 3).
ZK quorum loss (majority) ZK unavailable. If controller crashes during this, cluster is stuck. No metadata changes. Manual intervention Restore ZK nodes. Critical: pre-plan ZK ensemble sizing. Prefer 5-node ensemble in prod.
KRaft controller quorum loss (minority) Leader election proceeds with remaining quorum. Single controller failure is transparent (<1s failover). <1 second Restore failed controller nodes. KRaft is far more resilient than ZK for this scenario.
KRaft quorum loss (majority) No metadata operations possible. Cluster freezes for topology changes. Ongoing produce/consume may continue. Manual quorum recovery Use kafka-metadata-quorum.sh --replication-factor-reset as last resort. Restore from backup of __cluster_metadata log.

Network Partition

Split-Brain Scenario (3 brokers, RF=3, min.ISR=2) 
DC-A: Broker 1 (leader), Broker 2
DC-B: Broker 3
Network partition isolates DC-B from DC-A

Timeline:
T=0   Partition begins
T=30s Broker 3 loses ZK session → evicted from ISR
      Broker 1 continues serving with ISR={1,2} ✓ (quorum met)
T=30s Broker 3 in DC-B tries to elect itself (if ZK in DC-B)
      → Split-brain: broker 3 becomes leader for its view

Impact of unclean.leader.election.enable=true:
  - DC-B producers can write to broker 3
  - After partition heals, log diverges → data loss possible

Impact of unclean.leader.election.enable=false (recommended):
  - DC-B partitions go OFFLINE when broker 3 is only ISR member
  - No data loss possible, but DC-B is unavailable

Recovery after partition heals:
  - ISR reconciliation by controller
  - Follower with stale data truncates to leader's high-water mark
  - Rejected messages from isolated period are LOST
Data Loss Risk: With unclean.leader.election.enable=true + a network partition, data loss is virtually guaranteed when the partition heals. Always set to false in production.

Disk / Storage Failure

Failure TypeKafka BehaviorMitigation
Single disk failure (no JBOD) Broker shuts down entirely. All partitions on that broker become unavailable. JBOD (multiple disks). With JBOD, only partitions on the failed disk are affected; broker stays running.
Single disk failure (JBOD) Partitions on failed disk go offline. Broker continues serving partitions on healthy disks. ISR shrinks for affected partitions. JBOD is the SRE best practice. Replace disk, reassign partitions (kafka-reassign-partitions.sh).
Disk I/O saturation Log flush latency spikes → broker appears slow → replica lag → ISR shrinkage. Producer latency increases. Monitor LogFlushRateAndTimeMs. Use io scheduler=none for NVMe. Separate Kafka data from OS on different disks.
Filesystem corruption Broker may fail to load partition logs. Kafka log recovery runs on startup (replays from last checkpoint). Kafka log recovery is automatic. For severe corruption: restore from replica on another broker. num.recovery.threads.per.data.dir=4 speeds startup.

Consumer Failure Modes

FailureSymptomRoot CauseFix
Consumer group rebalance loop Lag oscillates, consumers restart repeatedly, rebalance-rate-per-hour > 4 Processing time > max.poll.interval.ms. Consumer is kicked from group during processing. Increase max.poll.interval.ms. Reduce max.poll.records. Move heavy processing to async worker thread.
Zombie consumer Partition assigned to consumer, but no progress. Lag grows. Heartbeat still valid. Consumer thread deadlocked or stuck, but heartbeat thread still running (separate threads in modern clients). Add application-level watchdog. Monitor poll-idle-ratio-avg — should not be 0.
Offset commit failure Consumer reprocesses records after restart. Exactly-once guarantees violated. Auto-commit during rebalance, or manual commit failures not caught. Use enable.auto.commit=false. Implement idempotent consumers. Use transactions for EOS.
Deserialization failure Consumer crashes on specific record. DLQ not configured. Entire partition blocked. Schema mismatch, corrupt record, or incompatible schema evolution. Configure DLQ (errors.tolerance=all in Connect, or custom try/catch). Enforce Schema Registry compatibility.

Incident Runbooks

Runbook: UnderReplicatedPartitions > 0

  1. Assess scope 
    # How many URPs and which brokers?
kafka-topics.sh --bootstrap-server :9092 \
  --describe --under-replicated-partitions

# Which broker is lagging (the replica)?
kafka-topics.sh --bootstrap-server :9092 \
  --describe --under-replicated-partitions | awk '{print $8}' | sort | uniq -c
  2. Check replica lag metric on the lagging broker 
    # JMX: kafka.server:type=ReplicaFetcherManager,name=MaxLag,clientId=Replica
# Also check: kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions
  3. Check if broker is healthy

    CPU, disk I/O, network bandwidth. Look for GC pauses in broker log. Check kafka.server:type=BrokerTopicMetrics for disk latency.

  4. Is the broker in ISR?

    If broker is in ISR but URP > 0, data is being written faster than it can replicate (replica.lag.time.max.ms exceeded). Throttle producers or add brokers.

  5. If broker is OOS (out of sync)

    Check replica.lag.time.max.ms (default 30s). If disk I/O is causing lag, check IOPS. If network, check replica.fetch.max.bytes and num.replica.fetchers.

  6. Resolution path 
    URP stays high after broker recovery?
→ Manual reassignment: kafka-reassign-partitions.sh
→ Throttle replication to avoid impacting production:
  kafka-configs.sh --bootstrap-server :9092 --entity-type brokers \
    --entity-name BROKER_ID --alter \
    --add-config 'leader.replication.throttled.rate=50000000,
                  follower.replication.throttled.rate=50000000'

Runbook: Consumer Lag Spike

  1. Identify which consumer groups are lagging 
    kafka-consumer-groups.sh --bootstrap-server :9092 --list
kafka-consumer-groups.sh --bootstrap-server :9092 \
  --describe --group GROUPNAME
# Look for: LAG column, CONSUMER-ID, HOST

# Burrow: GET /v3/kafka/local/consumer/GROUPNAME/lag
  2. Determine: is lag growing or stable?

    Growing lag = consumer can't keep up with producer rate. Stable high lag = consumer is at steady state, just behind. Growing is the emergency.

  3. Check consumer health

    Is consumer running? Check rebalance-rate-per-hour — if high, fix rebalance first. Check max.poll.interval.ms violations in consumer logs.

  4. Check if it's a throughput problem 
    # Producer rate vs consumer rate comparison
kafka-consumer-groups.sh --bootstrap-server :9092 \
  --describe --group GROUPNAME --verbose

# Tune consumer to increase throughput:
fetch.min.bytes=1048576    # 1MB minimum fetch
max.poll.records=2000
fetch.max.wait.ms=500
  5. Scale out if needed

    Add consumer instances (up to partition count). Verify partition count ≥ desired consumer parallelism. If not, increase partition count and do rolling restart of consumers.

Runbook: OfflinePartitionsCount > 0

P1 Incident. Offline partitions mean writes and reads are completely failing for those topics. This impacts application SLAs directly.
  1. Find offline partitions 
    kafka-topics.sh --bootstrap-server :9092 \
  --describe --unavailable-partitions
# Shows partitions with no leader elected
  2. Identify why — are the partition's ISR members running?

    If all ISR members are down → partition goes offline with unclean.leader.election.enable=false. Restore at least one ISR broker.

  3. Unclean leader election (last resort) 
    # Only if data loss is acceptable and availability > durability:
kafka-leader-election.sh --bootstrap-server :9092 \
  --election-type UNCLEAN \
  --topic TOPICNAME --partition 0
# Or for all partitions:
kafka-leader-election.sh --bootstrap-server :9092 \
  --election-type UNCLEAN --all-topic-partitions
  4. Post-incident: root cause

    Document which broker failed. Review replication.factor (should be ≥ 3). Review min.insync.replicas (should be RF-1). Consider adding brokers to reduce single-broker partition density.

Runbook: Broker Disk Full

  1. Immediate: pause writes to that broker — Route producers away if possible via partition reassignment with throttle.
  2. Emergency space recovery 
    # Option 1: Reduce retention immediately
kafka-configs.sh --bootstrap-server :9092 \
  --entity-type topics --entity-name TOPICNAME --alter \
  --add-config retention.ms=3600000  # 1 hour

# Option 2: Delete records up to offset
kafka-delete-records.sh --bootstrap-server :9092 \
  --offset-json-file delete-offsets.json

# Option 3: Find and delete temp/log files
ls -lhS /kafka/data/  # find largest files
# Never delete .log files manually — use Kafka's tools
  3. Long-term: expand storage

    Add JBOD volume (online via kafka-configs.sh --alter --add-config log.dirs=...). Or migrate partitions to other brokers with throttled reassignment.

  4. Alert tuning

    Set disk alert at 70% to give lead time. Track growth rate. Model projected full time in Grafana.

Disaster Recovery

RTO and RPO for Kafka

ScenarioTarget RTOTarget RPOStrategy
Single broker failure < 1 min (KRaft) / < 60s (ZK) 0 (no data loss with RF≥2, min.ISR) Auto leader election. No operator action needed.
AZ failure < 5 min 0 with stretch cluster across 3 AZs Spread replicas across AZs. Use rack-aware replication. RF=3 with 1 replica per AZ.
Full DC failure 15–30 min < 30s (async MM2) to 0 (sync replication) MirrorMaker 2 active-passive. Failover: redirect producers to DR cluster. Consumers reconnect.
Data corruption / accidental deletion 1–4 hours Last backup point Topic-level backup to S3 (Kafka S3 Sink connector). Point-in-time replay from offsets.
Full cluster loss 2–8 hours Depends on replication lag to DR DR cluster warm standby + MM2 sync. Automate failover via DNS/load balancer switch.

MirrorMaker 2 Active-Passive Failover

Normal Operation 
Production Cluster (DC-A)           DR Cluster (DC-B)
┌─────────────────────────┐         ┌─────────────────────────┐
│  Topics: payments,       │──MM2──▶│  Topics: dc-a.payments, │
│          orders, events  │        │  dc-a.orders, dc-a.events│
│  Producers writing here  │        │  (prefixed by MM2)       │
└─────────────────────────┘         └─────────────────────────┘
                                     MM2 also syncs:
                                     - Consumer offsets
                                     - ACLs (if configured)
                                     - Topic configs
Failover Procedure 
1. Detect DC-A failure (monitoring alert)
2. Stop MM2 to prevent stale writes to DC-B
3. Translate consumer offsets:
   kafka-mirror-maker.sh --translate-offsets \
     --consumer-config dc-b-consumer.properties \
     --topics-to-migrate payments,orders
4. Update producer bootstrap.servers to DC-B
5. Update consumer bootstrap.servers to DC-B
   Consumers will resume from translated offsets (no replay needed)
6. Verify lag = 0 on DC-B consumers
7. (After DC-A recovery) set up reverse MM2 DC-B → DC-A
8. Failback once DC-A is fully healthy
Offset Translation Gap: MM2 syncs offsets periodically (default 60s). The last 60s of messages before failover may require replay. Design consumers to be idempotent to handle this.

Backup Strategies

MethodMechanismRPOTrade-off
MirrorMaker 2 Consumer → Producer replication between clusters Async: seconds to minutes Best for DR. Topic data + offset sync. Adds consumer load to source cluster.
S3 Sink Connector Kafka → S3 via Connect. Stores as Parquet/Avro/JSON Flush interval (default 1000 records or 1min) Good for cold backup and analytics. Replay requires writing back to Kafka.
Tiered Storage (Kafka 3.6+) Remote log manager uploads segments to S3/GCS/Azure Segment roll interval (default 1GB or 1 week) Native Kafka solution. Reduces local disk. Not a DR solution — same cluster metadata required.
Filesystem snapshot Pause partition, snapshot EBS/PD volume, resume Point in time Fast restore. Requires coordination. Risk of partial writes if not done carefully.

Chaos Engineering

Chaos Hypothesis Framework for Kafka

For each chaos experiment, define: Steady State → Hypothesis → Blast Radius → Rollback Plan → Observation

Example: Broker Kill Experiment 
Steady State:
  - consumer lag < 1000 for all critical groups
  - UnderReplicatedPartitions = 0
  - P99 produce latency < 100ms

Hypothesis:
  Killing broker-2 (non-controller) will cause transient URP spike
  (<30s) and then auto-recovery. Consumer lag will not exceed 10,000.

Blast Radius:
  Partitions where broker-2 is leader (~33% of partitions in 3-broker cluster)
  Estimated: 10-15 seconds of elevated latency for those partitions

Rollback:
  Restart broker-2: systemctl start kafka

Experiment:
  kill -9 $(jps | grep Kafka | awk '{print $1}') on broker-2

Observe:
  - URP alert fires? Time to recovery?
  - Consumer lag spike? Duration?
  - Producer errors? (ErrorRate metric)
  - Auto leader election completed in expected time?

Conclusion:
  PASS if: recovery < 60s, no consumer lag breach, no data loss
  FAIL if: manual intervention required, lag exceeded threshold

Capacity Planning

Kafka Sizing Formula

Storage Required (per broker) 
Storage = (msg_rate × avg_msg_size × retention_seconds × replication_factor)
          ÷ num_brokers

Example:
  msg_rate = 500,000 msg/s
  avg_msg_size = 1 KB
  retention = 7 days = 604,800s
  RF = 3, brokers = 6

Storage = (500,000 × 1024 × 604,800 × 3) / 6
        = 500,000 × 1024 × 604,800 × 0.5
        = ~153 TB total → ~25.5 TB per broker

Add 20% headroom → ~30 TB per broker (SSD or JBOD)
Throughput Headroom 
Network bandwidth per broker:
  - Each broker receives replicas from others
  - Replication traffic = msg_rate × (RF-1) / num_brokers
  - Plus consumer fetch traffic

Rule of thumb:
  Use at most 50% of NIC bandwidth for Kafka traffic
  10GbE NIC → 1.2 GB/s max → target 600 MB/s per broker

Q & A

Senior How would you design SLOs for a Kafka platform serving 50 microservices with different criticality levels?

I'd tier the services by criticality and assign SLOs per tier rather than per service:

  • Tier 1 (payments, fraud detection): 99.99% availability SLO, P99 produce latency <50ms, consumer lag <100. Error budget: 4.4 min/month. Any change requires error budget approval.
  • Tier 2 (order events, notifications): 99.95% availability SLO, P99 <200ms, consumer lag <10,000. Error budget: 21.6 min/month.
  • Tier 3 (analytics, batch): 99.9%, no latency SLO, lag <1M records. Error budget: 43 min/month.

SLI measurement: use separate Kafka clusters or at minimum separate listener + quota policies per tier. Measure SLIs at the Kafka API layer (broker-side metrics), not application-side, to avoid client bugs distorting the signal.

Error budget policy: at >75% burn, we freeze Tier 1 changes. Tier 3 changes always allowed unless they would impact Tier 1 (e.g., noisy-neighbor disk I/O). Quarterly reviews with service owners to adjust targets based on actual user impact.

Senior Walk me through a Kafka disaster recovery failover. What are the exact steps and where are the risks?

Pre-failover (maintain continuously):

  • MirrorMaker 2 replicating all topics from DC-A → DC-B with sync.group.offsets.enabled=true
  • Monitor MM2 replication lag via kafka_consumer_group_lag on MM2's internal consumer group
  • Run monthly DR drills to verify the process actually works

Failover execution:

  1. Confirm DC-A is truly down (not a network blip) — wait 2-3 minutes or confirm with cloud provider
  2. Stop MM2 replication (prevent stale writes to DR if DC-A recovers unexpectedly)
  3. Translate offsets: kafka-mirror-maker.sh --translate-offsets — this maps DC-A offsets to equivalent DC-B offsets
  4. Update DNS / load balancer to point bootstrap.servers at DC-B
  5. Verify offset translation: run kafka-consumer-groups.sh --describe on DC-B and confirm lag is reasonable (not replaying from 0)
  6. Monitor consumer lag and producer errors for 15 minutes

Risks:

  • Offset sync gap: MM2 syncs offsets every 60s by default. Last 60s of messages may replay. Consumers must be idempotent.
  • Topic name mismatch: MM2 prefixes topics with cluster alias by default (e.g., dc-a.payments). Consumer configs must be updated or use replication.policy.class=IdentityReplicationPolicy.
  • ACL migration: If ACLs not synced, producers/consumers may get authorization failures on DR cluster
  • Schema Registry: Must also be failed over or pointed at the DR SR instance. Schema IDs must match between clusters.
Senior How do you implement chaos engineering for Kafka without risking production data?

Guardrails first: Before any chaos experiment, I define the steady state mathematically (lag < X, URP = 0, etc.), a specific hypothesis, the rollback command ready in a terminal, and a hard stop condition (if metric Y is breached, stop immediately).

Blast radius control: Start with a single non-critical topic. Use traffic shadowing: replay 5% of production traffic to a chaos cluster before running on production. Never run chaos during high-traffic periods (market open for financial services).

For Kafka specifically:

  • Broker kill: use systemctl stop, not kill -9, so Kafka flushes logs cleanly. Only use kill -9 to specifically test unclean shutdown recovery.
  • Network chaos: use Toxiproxy or tc netem with a time limit: tc qdisc del dev eth0 root reverts instantly
  • Disk fill: use a temp file (fallocate) that can be instantly deleted

Observability during chaos: Keep a metrics dashboard open. Have Burrow/kminion consumer lag visible. Set up a temporary tight alert (lag > 500 → Slack) that you turn off after the experiment.

Game day format: Quarterly game days with cross-functional teams (SRE + app teams). Document results. Failed experiments (where the system didn't behave as hypothesized) are the most valuable ones — they reveal real gaps.

Mid A broker's UnderReplicatedPartitions suddenly spikes to 50 at 2 AM. Walk me through your response.

Immediate triage (<5 min):

  1. Open Grafana: which broker is the common follower across those 50 URPs? kafka-topics.sh --describe --under-replicated-partitions | grep -oP "(?<=Isr: )[0-9,]+"
  2. Is that broker alive? Check broker JMX metrics. Is it in ZK/KRaft membership?
  3. Check broker logs for obvious errors: OOM, disk full, network errors

If broker is alive but lagging:

  • Check disk I/O — is LogFlushRateAndTimeMs spiked? Could be a compaction or heavy write storm
  • Check GC logs — was there a long GC pause that caused the replica to fall behind?
  • Check replica.lag.time.max.ms — if lag exceeded this threshold, ISR eviction happened. Broker will rejoin ISR once it catches up (automatically).
  • Monitor URP trend — is it decreasing (broker catching up) or stable/increasing (broker stuck)?

If broker is dead:

  • Restart it: systemctl start kafka
  • Watch URP during recovery — it will stay high until replicas catch up
  • Apply replication throttle to avoid starving producers during catchup

SLA decision: Check min.insync.replicas. If ISR count is still above min.ISR, producers are not impacted. Document the incident, set follow-up for root cause analysis in business hours.
