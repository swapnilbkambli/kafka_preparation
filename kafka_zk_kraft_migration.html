🚀Why Migrate from ZooKeeper to KRaft

Operational Simplicity

ZK mode requires running two separate systems (ZooKeeper ensemble + Kafka cluster), each with its own monitoring, upgrades, security, and failure modes. KRaft eliminates ZooKeeper entirely — one system to operate.

Scalability

ZooKeeper is a bottleneck for metadata operations. Practical limit: ~200,000 partitions per cluster. KRaft stores metadata in a Kafka log — scales to millions of partitions. Critical for large-scale deployments.

Faster Controller Failover

ZK controller failover: new controller reads all metadata from ZK (~seconds to minutes for large clusters). KRaft: metadata already replicated to controller quorum — failover in milliseconds.

Deprecation Timeline

ZooKeeper support removed in Kafka 4.0 (released 2025). Any cluster still on ZK mode must migrate to upgrade past Kafka 3.x. This is not optional long-term.

Security Simplification

ZK mode requires securing both ZooKeeper (SASL, mTLS) and Kafka independently. KRaft has one unified security model. Fewer auth configurations, fewer attack surfaces.

Startup Time

ZK-mode brokers read all partition metadata from ZK on startup — slow for large clusters. KRaft brokers fetch metadata from the quorum log — much faster startup and recovery.

Active Controller (Raft Leader)

The single writer for cluster metadata. Processes admin requests (create topic, elect leader, etc.) and appends records to the __cluster_metadata log. Elected by Raft — majority quorum vote.

Controller Quorum

Typically 3 or 5 controllers. Quorum of (N/2)+1 must be available for writes. 3 controllers: tolerates 1 failure. 5 controllers: tolerates 2 failures. Controllers vote for leader via Raft protocol.

Metadata Log

__cluster_metadata is an internal Kafka topic. Brokers subscribe to it — they maintain a local snapshot + recent log entries to serve metadata requests without hitting the active controller.

Broker Registration

In KRaft, brokers register with the active controller via BrokerRegistration RPC (not ZK ephemeral nodes). Controller tracks live brokers. If a broker doesn't heartbeat, controller fences it.

⚠️
ZK failure modes to know: ZK session expiry causes the controller to lose its /controller znode → triggers controller re-election on another broker → during this window (~6s), no metadata changes can be made. GC pause on the controller broker > ZK session timeout causes spurious leader re-election even though the broker is healthy — a classic cause of instability.

🔀ZK → KRaft Migration Steps (Kafka 3.4+)
💡
Migration mode: Kafka 3.4+ supports dual-write migration — a cluster can run in a hybrid mode where the ZK controller and a KRaft controller quorum coexist temporarily. This allows zero-downtime migration. Kafka 4.0 requires KRaft — you must complete migration before upgrading to 4.0.
1
Upgrade Kafka to 3.4+ (while still on ZK)
All brokers must be on Kafka 3.4+ before starting KRaft migration. Perform rolling upgrade of the ZK-mode cluster first. Verify cluster health post-upgrade: zero URP, all brokers online.
2
Enable migration on brokers
Set zookeeper.metadata.migration.enable=true in broker configs. Do NOT restart yet — just prepare the config.
3
Provision KRaft controller nodes
Set up 3 (or 5) dedicated KRaft controller nodes with:
process.roles=controller
controller.quorum.voters=1@ctrl1:9093,2@ctrl2:9093,3@ctrl3:9093
node.id=1 (unique per controller)
Generate cluster ID from existing ZK cluster: kafka-storage.sh format --cluster-id $(kafka-storage.sh random-uuid) --config server.properties
Important: Use the SAME cluster ID as the existing ZK cluster. Get it from ZK: zookeeper-shell.sh localhost:2181 get /cluster/id
4
Start KRaft controllers in migration mode
Start the KRaft controller nodes. They connect to ZooKeeper and read existing metadata. The active KRaft controller becomes the migration leader — it reads all ZK metadata and writes it to the __cluster_metadata log. ZK controller demotes itself; KRaft controller takes over metadata management. Brokers still connected to ZK for now.
5
Rolling restart of brokers to KRaft mode
Update each broker's config to add KRaft controller quorum details. Remove ZK configs from broker properties. Restart brokers one by one (rolling restart protocol). Each restarted broker registers with KRaft controller instead of ZK. After all brokers migrated, cluster runs KRaft but ZK is still connected (hybrid mode).
6
Finalize migration (disconnect ZK)
Run: kafka-features.sh --bootstrap-server broker:9092 upgrade --feature metadata.version=N
This atomically completes migration — ZK is no longer used. After this step, rollback is impossible without a full restore. Monitor cluster health carefully.
7
Decommission ZooKeeper
Verify cluster runs cleanly in KRaft-only mode for several days. Monitor: controller failover, topic creation, partition reassignment. When confident: shut down ZK ensemble. Decommission ZK VMs/containers.
KEY CONFIGS FOR KRAFT CONTROLLERS: ───────────────────────────────────────────────────── # controller server.properties process.roles=controller # or "broker,controller" for combined node.id=1 # unique integer, different from broker IDs controller.quorum.voters=1@ctrl1:9093,2@ctrl2:9093,3@ctrl3:9093 listeners=CONTROLLER://:9093 controller.listener.names=CONTROLLER log.dirs=/var/kafka/metadata # where metadata log is stored # For migration (controller nodes also connect to ZK temporarily) zookeeper.connect=zk1:2181,zk2:2181,zk3:2181/kafka zookeeper.metadata.migration.enable=true # KRAFT BROKER CONFIGS (post-migration): process.roles=broker node.id=10 # broker IDs should not overlap controller IDs controller.quorum.voters=1@ctrl1:9093,2@ctrl2:9093,3@ctrl3:9093 # Remove all zookeeper.* settings

↩️Rollback Plan
🚨
Rollback is only possible before Step 6 (finalize migration). Once you run the feature upgrade command, ZK is permanently disconnected from the cluster state. After that point, the only rollback is restoring from a ZK snapshot (which means replaying all topic/consumer changes since the snapshot — significant data risk in production). Plan your migration during low-traffic windows with a clear rollback decision point.
1
Stop all KRaft controller nodes
Shutdown the new KRaft controllers before they write any non-revertible metadata.
2
Rolling restart brokers back to ZK mode
Revert broker configs: remove KRaft controller settings, restore ZK settings. Restart brokers one at a time.
3
Verify ZK connectivity and controller election
Check that a broker has been elected controller via ZK. Verify all partitions have leaders.
4
Decommission KRaft controller nodes
Wipe metadata logs from controller nodes to prevent stale state if re-attempted later.
⚠️Gotchas & Common Pitfalls
  • 🚨
    Node ID overlap: Controller node IDs must not overlap with broker node IDs. Reserve a range: brokers 1–100, controllers 101–103.
  • 🚨
    Wrong cluster ID: KRaft controllers must be formatted with the exact same cluster ID as the existing ZK cluster. Mismatch = cluster refuses to join.
  • ⚠️
    ZK not decommissioned too early: Keep ZK running through the full migration and for several days after. Don't rush decommissioning before confirming cluster stability.
  • ⚠️
    ACL migration: ZK-mode ACLs stored in ZK paths. KRaft stores them in the metadata log. They ARE migrated automatically, but verify all ACLs post-migration before decommissioning ZK.
  • ⚠️
    Old Kafka clients: Very old clients (pre-0.10) used ZK directly. Post-migration, those clients break. Modern clients (0.10+) use broker bootstrap servers only — unaffected.
  • ⚠️
    Monitoring tools: Tools that read ZK directly (e.g., Kafka Manager/CMAK, Kafdrop older versions) will stop working after ZK decommission. Migrate to tools with native KRaft support (Redpanda Console, AKHQ, Confluent Control Center 7.4+).
  • 💡
    Combined mode in small clusters: For clusters with ≤5 brokers, combined mode (brokers acting as controllers) is fine. For large clusters, dedicated controller nodes prevent controller GC pauses from affecting data path.
  • 💡
    Metadata snapshot size: After migration, __cluster_metadata may be large initially. KRaft periodically takes snapshots and truncates old log entries. Monitor disk usage on controller nodes.
🎯Concepts & Scenario Q&A
Q What is the Raft protocol and how does KRaft use it for metadata management? Mid
Raft is a distributed consensus protocol designed to be understandable (vs Paxos). It guarantees that a cluster of N nodes can make consistent decisions as long as a majority (N/2+1) are available.

Key Raft concepts:
  • Leader: One node receives all writes. Appends to log. Sends AppendEntries RPCs to followers.
  • Followers: Replicate the leader's log. Vote in elections.
  • Term: A monotonically increasing number. Each election starts a new term. Prevents stale leaders from confusing the cluster.
  • Log commitment: An entry is committed once a majority acknowledges it. Committed entries survive leader failures.
  • Leader election: When the leader fails, followers timeout and start an election. Candidate requests votes — wins if majority vote for it in the same term.
In KRaft: The __cluster_metadata log IS the Raft log. The active controller IS the Raft leader. All metadata changes (create topic, elect partition leader) are Raft log entries. Brokers subscribe to this log — they're Raft observers, not voters.
Q Your team runs Kafka 2.8. Leadership says you must be on Kafka 4.0 in 6 months. What's your migration plan? Senior
Kafka 4.0 requires KRaft — you cannot skip to 4.0 from 2.8 directly. The migration path is:
  1. Month 1–2: Upgrade to Kafka 3.4+ (still ZK mode) — Kafka supports rolling upgrades one major version at a time. Go 2.8 → 3.0 → 3.4 in rolling upgrades. Each hop: upgrade one broker at a time, verify health, proceed.
  2. Month 2–3: Set up staging KRaft cluster — Build KRaft cluster in non-prod. Validate all connectors, tools, monitoring work with KRaft. Fix tooling issues (CMAK → AKHQ, etc.).
  3. Month 3–4: ZK → KRaft migration in staging — Perform full migration procedure on staging. Validate rollback procedure. Document lessons learned.
  4. Month 4–5: Production ZK → KRaft migration — During low-traffic window. Have ZK snapshot backup ready. Clear rollback decision gate before Step 6.
  5. Month 5–6: Kafka 4.0 upgrade — Once fully on KRaft, upgrade brokers to 4.0 via rolling restart. This is now a standard upgrade — no migration complexity.
Risk mitigation: Never upgrade Kafka version and migrate to KRaft in the same step. Separate concerns — fix upgrade issues first, then migrate. Build confidence at each stage.
Q After completing ZK→KRaft migration, you notice that some consumer groups are missing their offsets. What happened? Senior
Consumer group offsets are stored in the Kafka topic __consumer_offsets — NOT in ZooKeeper (since Kafka 0.10). Therefore, the KRaft migration does NOT affect consumer group offsets. They remain in __consumer_offsets on the brokers.

If offsets are missing after migration, the cause is likely unrelated to KRaft:
  • Offset retention: __consumer_offsets has its own retention period (offsets.retention.minutes, default 7 days). Consumer groups that were inactive during migration may have had their offsets expire.
  • Group ID mismatch: New consumer instances started with different group IDs.
  • Partition reassignment during migration: If __consumer_offsets had URP or was reassigned, some offsets could be affected.
Fix: If offsets are genuinely gone (expired), reset consumers to the appropriate position: --to-datetime or --to-earliest depending on data availability and business requirements.
Q How does KRaft handle the case where the active controller fails mid-write? Senior
This is the core guarantee of Raft — it handles this safely:
  • If the controller fails before the write was committed (majority didn't ack) → the entry is NOT committed. After election, the new leader may or may not have this entry. If it does, it's truncated as it was uncommitted. Clients retry their request to the new controller.
  • If the controller fails after the write was committed (majority acked) → the entry IS durable. The new leader will have this entry in its log (since majority have it). It will be present in the new leader's log by definition of Raft commitment. The client may see a timeout if it didn't receive the response, but the write is safe — idempotent retries are safe.
Key property: A Raft leader can only be elected if it has the most up-to-date log (highest term + longest log). So the new leader always has all committed entries from the previous leader. No committed metadata is ever lost.
