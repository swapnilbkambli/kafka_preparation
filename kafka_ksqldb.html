⚡ksqlDB Architecture

ksqlDB is a streaming SQL engine built on top of Kafka Streams. It exposes SQL semantics for stream processing — no Java/Scala code required. Under the hood every ksqlDB query compiles to a Kafka Streams topology.

KSQLDB ARCHITECTURE: ────────────────────────────────────────────────────────────────── Client (CLI / REST) │ SQL statement ▼ ┌─────────────────────────────────────────────────────────────┐ │ ksqlDB Server (headless or interactive) │ │ │ │ ┌──────────────┐ ┌─────────────────────────────────────┐ │ │ │ REST API │ │ Query Execution Engine │ │ │ │ (:8088) │──►│ SQL Parser → Logical Plan │ │ │ └──────────────┘ │ → Physical Plan → KStreams Topology│ │ │ └──────────────┬────────────────────── ┘ │ │ │ │ │ ┌─────────────────────────────────▼──────────────────────┐ │ │ │ Kafka Streams Layer │ │ │ │ - KStream / KTable / GlobalKTable │ │ │ │ - State Stores (RocksDB) per partition │ │ │ │ - Changelog topics for fault tolerance │ │ │ └──────────────────────────────┬─────────────────────────┘ │ └─────────────────────────────────┼──────────────────────────── ┘ │ consume/produce ▼ Apache Kafka Cluster CLUSTER MODE: Multiple ksqlDB servers form a cluster. Queries are distributed across servers (partitions assigned via consumer group). Any server can accept queries — internal routing forwards to correct partition owner for pull queries.

Persistent Query

Runs continuously until explicitly terminated. Creates a Kafka Streams application under the hood. Results are materialized to a Kafka topic (CREATE STREAM AS SELECT or CREATE TABLE AS SELECT). Survives server restarts.

Transient (Ad-hoc) Query

Runs only for the duration of the client connection (SELECT ... EMIT CHANGES). Used for interactive exploration. Terminates when client disconnects. Does not persist results to a topic.

Internal Topics

ksqlDB creates internal Kafka topics for: query state changelog, repartitioning (when joining on non-partition keys), and aggregation results. These are prefixed with the application ID and should not be manually deleted.

Schema Integration

ksqlDB integrates with Schema Registry. When creating a stream/table, specify VALUE_FORMAT='AVRO' and ksqlDB auto-discovers the schema. Schema evolution in SR is reflected in ksqlDB queries automatically.

🔍Push vs Pull Queries

Push Query (EMIT CHANGES)

Subscribes to a continuous stream of results as new data arrives. Like a streaming SELECT. Keeps connection open. Used for: real-time dashboards, alerting, event-driven microservices.
SELECT * FROM orders EMIT CHANGES;

Pull Query (point-in-time)

Returns current state of a materialized table at the moment of the query. Like a traditional DB SELECT — returns immediately. Only works on tables with state stores (aggregations, joins).
SELECT balance FROM account_balance WHERE account_id = 'ACC001';

CREATE STREAM AS SELECT

Persistent transformation query. Creates a new stream (Kafka topic) populated by the result of transforming/filtering another stream. Runs forever as a Kafka Streams app.
CREATE STREAM high_value_orders AS SELECT * FROM orders WHERE amount > 10000 EMIT CHANGES;

CREATE TABLE AS SELECT

Persistent aggregation query. Creates a new table (materialized view) by aggregating a stream. E.g., running count of orders per customer. The table is continuously updated as new events arrive.
CREATE TABLE order_counts AS SELECT customer_id, COUNT(*) AS cnt FROM orders GROUP BY customer_id EMIT CHANGES;

💡
Pull query routing in clustered mode: In a multi-server ksqlDB cluster, state is partitioned across servers. A pull query for key "ACC001" is routed to the server that owns that partition. This is transparent to the client — any server accepts the query and internally routes it. This requires servers to know each other's addresses via ksql.advertised.listener.
🕐Windowing
WINDOW TYPES: ────────────────────────────────────────────────────────────────── TUMBLING WINDOW (fixed, non-overlapping): |──────10min──────|──────10min──────|──────10min──────| Window 1: 09:00-09:10 Window 2: 09:10-09:20 ... → Each event belongs to exactly ONE window → Use: hourly metrics, daily summaries SELECT account_id, SUM(amount) FROM payments WINDOW TUMBLING (SIZE 1 HOUR) GROUP BY account_id EMIT CHANGES; HOPPING WINDOW (fixed size, sliding step — OVERLAPPING): |─────1hr──────| |─────1hr──────| |─────1hr──────| → step = 15min, size = 1hr → each event in 4 windows → Use: rolling averages, moving sums WINDOW HOPPING (SIZE 1 HOUR, ADVANCE BY 15 MINUTES) SESSION WINDOW (event-driven, variable size): Events: ──●─●──────────────────●─●─●───── ... Gap: inactivity gap (30min) → new session → Window size varies. Gaps between events define boundaries. → Use: user sessions, activity bursts WINDOW SESSION (30 MINUTES) LATE EVENTS: ksqlDB handles late-arriving events via GRACE PERIOD: WINDOW TUMBLING (SIZE 1 HOUR) GRACE PERIOD 30 MINUTES; → Events up to 30min late are included in their window. → After grace period, window is finalized.
⚠️
Window results are NOT final until grace period expires. A tumbling window for 09:00-10:00 with a 30-minute grace period won't finalize until 10:30. Pull queries on windowed tables may return incomplete counts for active windows. Design dashboards to show "current window is in-progress" appropriately.

⚠️
Co-partitioning requirement: Stream-Stream and Stream-Table joins require both sides to have the same number of partitions and the same partition key. If not, ksqlDB automatically creates a repartition topic — adds latency and internal topic overhead. Design your topics with joins in mind, or use GlobalKTable to bypass this requirement for small dimension tables.
💾State Stores

RocksDB State Store

ksqlDB uses RocksDB (embedded key-value store) for local state storage — aggregation results, join lookups, windowed state. Stored on local disk of the ksqlDB server. Fast for sequential access. Partitioned: each server owns state for a subset of keys.

Changelog Topic

Every state store has a backing Kafka changelog topic. Stores every state update. On server restart, state is restored by replaying the changelog. This enables fault tolerance — state is not lost when a server crashes.

Standby Replicas

ksqlDB servers can maintain standby state replicas for faster failover. Configure ksql.streams.num.standby.replicas. Standbys continuously consume the changelog — on failover, they take over immediately without full state rebuild.

State Store Size

Monitor disk usage on ksqlDB servers — state stores can grow very large for high-cardinality aggregations or long retention windows. Set retention.ms on windowed state or use the GRACE PERIOD clause to bound state retention.

🔨UDFs & UDAFs

UDF (User Defined Function)

Custom scalar function: takes one or more column values, returns a single value. Example: masked credit card number, geohash from lat/lon, currency conversion. Implement in Java with @UdfDescription and @Udf annotations. Drop JAR in ksqlDB extensions directory.

UDAF (User Defined Aggregate Function)

Custom aggregation: takes many rows, returns one value (like SUM, COUNT but custom). Example: custom rolling percentile, weighted average. Implements Udi interface with aggregate(), merge(), map(), undo() methods.

UDTF (User Defined Table Function)

Explode one row into multiple rows (like SQL LATERAL). Example: split a comma-separated field into individual rows. Used in STREAM processing pipelines.

Deployment

Package UDFs as JARs. Place in ksql.extension.dir on each ksqlDB server. Restart required for new UDFs (or hot-reload in some versions). Functions appear in SHOW FUNCTIONS;.

🎯Concepts & Scenario Q&A
Q What is the difference between a Stream and a Table in ksqlDB, and when would you use each? Basic
Stream represents an immutable sequence of events — like a ledger or audit log. Every record is a new fact. Old records are never updated or deleted. Use a Stream when: capturing transactions (payments, clicks, logins), tracking state changes as events.

Table represents the current state of something. Only the latest record per key is meaningful — previous records are superseded. Use a Table when: you need the current account balance (not all historical transactions), current inventory level, latest user profile.

The duality: A Table IS a Stream at a lower level — it's a compacted view. You can convert: a stream of payment events → a table of running balances (via aggregation). A table → stream of change events (table-to-stream conversion).

Practical example: In banking: payment_events is a Stream (every payment is a new event), account_balances is a Table (latest balance per account). Join them: enrich payment events with current account info.
Q A ksqlDB persistent query is falling behind with growing consumer lag. How do you scale it? Mid
ksqlDB scales out by adding more servers to the cluster. Processing partitions of the input Kafka topic are distributed across servers:
  • Step 1: Add more ksqlDB servers to the cluster (same ksql.service.id). They join the consumer group automatically. Tasks are rebalanced — each server gets a subset of partitions.
  • Step 2: Check if input topic has enough partitions. If the topic has only 4 partitions and you have 8 servers, 4 servers idle. Increase topic partition count (requires creating a new topic and migrating, or rebalancing after increasing).
  • Step 3: If the query involves an aggregation or join with a state store, adding servers also splits state across more nodes — reduces per-node memory/disk pressure.
  • Step 4: Check if the bottleneck is the query logic itself — complex UDFs or deserialization. Profile the ksqlDB server: CPU, memory, disk I/O on RocksDB state stores.
Key insight: ksqlDB parallelism is bounded by the partition count of the input topics. Plan partition count at topic creation time.
