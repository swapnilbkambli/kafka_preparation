📈Scenario: Consumer Lag Spike — Payments Processor at 2M Lag
📟
Alert: "consumer_group_lag{group=payments-processor} > 500000 for 5m"
Impact: Payment confirmations delayed by 15 minutes. Business SLA breached.
1
Immediate: Triage scope
kafka-consumer-groups.sh --describe --group payments-processor
Is lag on ALL partitions or specific ones? If specific partitions → likely a consumer task issue. If all → production spike or consumer group failure.
2
Check if consumers are alive
Are consumers still running? Check app logs for errors. Check if the consumer group shows active members in the describe output. If no members → all consumers are down. If members present → consumers alive but slow.
3
Check for rebalance storm
Consumer logs: "Rebalance in progress". JMX rebalance-rate-per-hour. Frequent rebalances = processing pauses. Check if any consumer is repeatedly joining/leaving (GC pause, slow processing exceeding max.poll.interval.ms).
4
Check production side
Has the producer throughput spiked? Check producer metrics or topic LOG-END-OFFSET growth rate. If producers suddenly sent 10x normal volume, consumers need time to catch up — this may self-resolve without action.
5
Scale consumers (fastest fix if consumers are healthy)
With 12 partitions and 4 consumers: add 8 more consumers (up to 12 total). They join the group, partitions rebalance, throughput doubles/triples. Consider using cooperative rebalancing to minimize pause during scale-out.
6
Check downstream bottleneck
If adding consumers doesn't help: the bottleneck is downstream (DB writes, external API). Profile: how long does processing each record take? Batch downstream writes, parallelize API calls, or temporarily skip non-critical enrichment.
7
Post-incident: Prevent recurrence
Set consumer lag alerts earlier (warn at 100K, critical at 500K). Add auto-scaling (KEDA, HPA). Review max.poll.records and processing SLA. Add circuit breaker for downstream failures.
💾Scenario: Broker Disk Full — Producers Getting LEADER_NOT_AVAILABLE
📟
Alert: "kafka_broker_disk_usage_percent > 90" on broker-2
Symptom: Producers on partitions led by broker-2 get LEADER_NOT_AVAILABLE or timeout errors.
1
Confirm and quantify
df -h /var/kafka-data on broker-2. How much space is left? Minutes or hours? This determines urgency.
2
Immediate relief: delete old log segments
Kafka retains segments per log.retention.ms. Force retention cleanup:
kafka-configs.sh --alter --topic non-critical-topic --add-config retention.bytes=10737418240
Wait for log cleaner to run (usually within minutes). Also: kafka-log-dirs.sh to see which topics consume most disk.
3
Identify the culprit topic
du -sh /var/kafka-data/*/ sorted by size. Usually one or two topics dominate. Verify their retention config is appropriate. A topic with retention.ms=-1 (infinite) is a common mistake.
4
Throttle producers (stop the bleeding)
kafka-configs.sh --alter --entity-type clients --entity-name bulk-producer --add-config producer_byte_rate=1048576
This prevents more data from arriving while you free space.
5
Move partitions off the full broker
Use partition reassignment to move some partitions from broker-2 to brokers with free space. Run with throttle to avoid making disk worse during reassignment.
6
Long-term: capacity planning
Alert at 70%, not 90%. Enable tiered storage (Kafka 3.6+) to offload old segments to S3/GCS. Set log.retention.bytes per broker to cap total usage. Set up disk usage trending dashboards.
🚨
Critical: If a broker's disk hits 100%, the broker process may crash — it needs disk to write ISR checkpoint files and offsets. A crashed broker triggers partition leader re-elections and ISR shrinkage. Alert and act at 80% to prevent this cascade.
⚡Scenario: Leader Election Storm — Cluster Instability After Upgrade
📟
Symptoms: Hundreds of leader elections per minute. Producers and consumers getting frequent "Not leader for partition" errors. Cluster JMX shows ActiveControllerCount=0 intermittently.
1
Check controller stability
JMX: ActiveControllerCount should always be exactly 1. If it's 0 or fluctuating, the controller is unstable. Check controller broker logs for exceptions.
2
Check ZooKeeper session (ZK mode)
Controller stability requires a healthy ZK session. Check ZK logs for "session expired" events. Check controller broker GC — long GC pauses (> zookeeper.session.timeout.ms = 18s default) cause ZK session expiry → controller re-election. After upgrade: JVM version change may affect GC behavior.
3
Check for version incompatibility
After rolling upgrade: are all brokers on the same inter-broker protocol version? Check inter.broker.protocol.version config. Mismatched protocol versions can cause partition leadership instability.
4
Stabilize: reduce leader elections
Temporarily set auto.leader.rebalance.enable=false to stop auto-election. Focus on getting all brokers stable first, then re-enable and run preferred replica election once.
5
Check network between brokers
Network packet loss between brokers can cause heartbeat failures → ISR shrinkage → leader elections. iperf3 between broker nodes. Check cloud provider network events.
6
Rollback if needed
If upgrade caused the instability: roll back the last-upgraded broker to the previous version. Identify whether the issue is the new version or the upgrade process itself.
🔁Scenario: ISR Shrinkage — Producers Blocked on acks=all
📟
Symptoms: Producer latency p99 jumps from 20ms to 5s. Alert: "UnderReplicatedPartitions > 0". Consumer lag starts growing.
1
Identify affected partitions and lagging broker
kafka-topics.sh --describe --under-replicated-partitions
Find which broker's replicas are out of sync.
2
Check the lagging broker
Is it running? ps aux | grep kafka. Network reachable? Disk full? GC pauses? Broker logs: any errors?
3
Check replica fetch lag metric
JMX on the lagging broker: kafka.server:type=FetcherLagMetrics,name=ConsumerLag,clientId=ReplicaFetcherThread-*. High lag = follower can't keep up with leader's write rate.
4
If broker is alive but lagging: check causes
- Disk I/O saturated on that broker: iostat -x 1
- Network between leader and follower: iperf3
- Follower fetch throttled: check replica.fetch.max.bytes
- Long GC pauses on follower: GC log analysis
5
If broker is down: trigger leader election
Kafka auto-elects a new leader from ISR when a broker fails. Monitor UnderReplicatedPartitions — it should clear once the new leader is elected. If ISR = {leader only} and min.insync.replicas=2, producers get NotEnoughReplicasException. You may need to temporarily lower min.insync.replicas to unblock critical producers — with explicit awareness of the data risk.
6
After broker recovery: wait for ISR rebuild
When the broker restarts, it fetches missing segments from leaders. Monitor URP count — it should trend toward 0. Do NOT restart another broker until URP = 0.
🔍Scenario: Data Loss Investigation — Missing Records
📟
Report: "We sent 10,000 payment records but only 9,847 were processed by downstream. 153 records appear to be missing."
1
Determine where data is missing: producer or consumer side
Check if records are IN Kafka first. Use kafka-console-consumer.sh --from-beginning on the topic and compare total record count vs expected. If records are in Kafka → consumer dropped them. If not → producer didn't send or broker dropped them.
2
Check producer configuration
Was acks=0 or acks=1 used? These can result in data loss on broker failure. Check producer error metrics — were there send failures that weren't retried?
3
Check for unclean leader election
Was unclean.leader.election.enable=true? If a broker failed and a stale replica became leader, data written to the old leader but not replicated is lost. Check broker logs around the time of loss for "Unclean leader election".
4
Check consumer offset commit behavior
If consumer auto-committed offsets BEFORE processing: crash during processing skips those records. Check enable.auto.commit and auto.commit.interval.ms. Records may be in Kafka but consumer moved past them.
5
Check for topic retention
If records are older than log.retention.ms, they've been deleted. Check when the records were sent vs current retention policy.
6
Recovery options
If records are still in Kafka (consumer missed them): reset consumer group offset to replay. If records are gone from Kafka (broker loss, retention): replay from the upstream source system. If using exactly-once semantics: check transaction log for aborted transactions.
🔒Scenario: Producer Backpressure — Application Thread Blocked
📟
Symptom: Application threads are blocked for 30+ seconds. Heap dumps show threads stuck in KafkaProducer.send(). No TimeoutException — just blocking.
1
Diagnose: RecordAccumulator buffer full
JMX: kafka.producer:type=producer-metrics,name=bufferpool-wait-ratio > 0 means producer is blocked waiting for buffer space. Root cause: Sender thread can't drain batches to broker fast enough.
2
Why is Sender not draining?
- Broker is slow: check RequestQueueSize, LocalTimeMs
- ISR issue: acks=all waiting for lagging followers
- Too many in-flight requests: max.in.flight.requests.per.connection exceeded
- Network saturation between producer and broker
3
Immediate fix: increase buffer or reduce max.block.ms
Increase buffer.memory to give Sender more breathing room. Or reduce max.block.ms so send() throws an exception faster (fail-fast) rather than blocking indefinitely — letting the application handle the backpressure explicitly.
4
Fix the root cause
If broker is slow: fix broker (disk, IO threads, ISR). If producer is sending too fast: implement rate limiting in the application (token bucket, semaphore). Kafka is not a buffer for unbounded production rate.
5
Design consideration: async vs sync send
Use async send(record, callback) instead of send(record).get(). The callback fires when the record is acknowledged. Application doesn't block the calling thread. Handle errors in the callback. This makes backpressure visible as a callback error rather than a thread block.
🌐Scenario: Network Partition Between Kafka DCs
📟
Scenario: Your Kafka cluster spans 2 availability zones. The network link between AZ-A and AZ-B is severed. What happens?

Depends on your ISR configuration:

If RF=3, ISR replicas spread across AZs

With 2 brokers in AZ-A and 1 in AZ-B: AZ-A partition is the leader. AZ-B broker loses connectivity → removed from ISR after replica.lag.time.max.ms. With min.insync.replicas=2: AZ-A still has 2 ISRs → cluster continues producing normally. AZ-B broker is URP but not blocking.

If ISR = {AZ-A leader, AZ-B follower} only (ISR=2, min.insync=2)

Network split → AZ-B follower can't ack → removed from ISR → ISR becomes {leader only} → min.insync.replicas=2 not met → producers get NotEnoughReplicasException → writes blocked. This is the designed behavior: protect data integrity over availability.

Split-brain risk with unclean election

If unclean.leader.election.enable=true: AZ-B broker may elect itself as leader for partitions it had (out of sync). When network heals: two leaders for same partition → data divergence. This is why unclean election must be false in financial systems.

Consumer behavior during partition

Consumers in AZ-B lose connectivity to AZ-A leaders → session timeout → rebalance → consumers in AZ-A take over those partitions. This causes temporary lag spike but recovers when rebalance completes.

🏛️Scenario: Controller Failover — Broker Hosting Controller Crashes
1
What happens automatically (ZK mode)
ZK session for the controller expires (typically 6–18s). Remaining brokers race to write the /controller znode. First to succeed becomes new controller. New controller reads ALL partition state from ZK (can take seconds to minutes for large clusters).
2
What happens automatically (KRaft mode)
Raft election among controller quorum. Takes milliseconds. New controller already has complete metadata in its Raft log. No ZK round-trip. Brokers continue with cached metadata during brief election window.
3
During controller election: admin operations are blocked
Topic creation, partition reassignment, ACL changes all require the active controller. These are queued and resume after election. Data path (produce/consume) is NOT blocked — brokers serve data independently of the controller.
4
After new controller elected
New controller runs leader recovery for any partitions whose leader was the crashed broker. Elects new partition leaders from ISR. These leader elections cause a brief "not leader" period for those partitions — producers retry automatically.
5
Verify recovery
kafka-broker-api-versions.sh — returns controller broker ID. JMX ActiveControllerCount = 1. UnderReplicatedPartitions trending to 0. Run preferred replica election to restore optimal partition placement.
🔄Scenario: Rebalance Loop — Consumers Perpetually Rebalancing
📟
Symptom: Consumer logs show "Rebalance in progress" every 30–60 seconds. Consumer lag growing. Consumer group never reaches stable state.
1
Identify the unstable consumer
Consumer logs on each instance: which one is repeatedly joining/leaving? Look for: "Attempting to send heartbeat to group coordinator" followed by "Member heartbeat timeout".
2
Common cause: poll loop too slow
Consumer calls poll(), processes 500 records, takes 6 minutes → exceeds max.poll.interval.ms (default 5min) → kicked out → rebalance → reassignment → repeat.
Fix: Reduce max.poll.records so each batch is processed within the interval. Or increase max.poll.interval.ms if processing is legitimately slow.
3
Common cause: GC pause exceeds session timeout
Long stop-the-world GC on consumer → can't send heartbeat → kicked out. Fix: switch to G1GC/ZGC. Increase session.timeout.ms (but don't exceed group.max.session.timeout.ms on broker, default 30min).
4
Common cause: network issues
Heartbeat packets dropped between consumer and group coordinator. Check network stability. Consumer restart doesn't help — underlying network issue persists.
5
Use Cooperative Rebalancing as mitigation
Switch to partition.assignment.strategy=CooperativeStickyAssignor. This minimizes disruption: only partitions that change ownership are revoked. Consumers keep their other partitions during rebalance — lag impact greatly reduced even with frequent rebalances.
❓Scenario: Consumer Lag Shows Zero but Records Not Being Processed
📟
Symptom: kafka-consumer-groups.sh shows lag = 0 for the group but the downstream database shows missing records.
1
Most likely cause: auto-commit committed before processing
With enable.auto.commit=true and auto.commit.interval.ms=5000: the consumer commits the offset every 5 seconds regardless of whether processing succeeded. If the consumer fetched records and committed the offset but then crashed during processing → those records appear "consumed" (lag=0) but were never processed.
2
Check consumer for silent errors
Look for swallowed exceptions in the consumer's processing code. A try { process() } catch (Exception e) { /* ignored */ } will advance the offset without actually processing the record.
3
Check offset commit mode
Switch to manual commit: enable.auto.commit=false. Commit only after confirmed write to DB. This ensures lag=0 actually means "processed," not "fetched."
4
Recovery
Reset consumer group offset to replay the missing records: kafka-consumer-groups.sh --reset-offsets --to-datetime <time-before-incident> --execute. Ensure sink is idempotent (upsert by primary key) to handle duplicates from replay.
🔗 Scenario: Consumer Lag from Broker Failure → URP Cascade
📟
Alerts firing simultaneously:
1. UnderReplicatedPartitions > 0 — broker-3 shows 120 URPs
2. consumer_group_lag{group=orders-processor} > 200000
3. Producers reporting NotEnoughReplicasException intermittently
Impact: Order confirmation emails delayed 20 min. SLA breach imminent.
🧠
Why these three alerts fire together — the causal chain:
Broker-3 crashes → partitions it leads have no leader → producers get LEADER_NOT_AVAILABLE → writes fail/retry → throughput drops → consumer lag stops growing (no new data) but then ALSO: partitions where broker-3 was a follower are evicted from ISR → ISR shrinks to RF−1 → if ISR count drops below min.insync.replicas → producers get NotEnoughReplicasException → writes queue up in RecordAccumulator → consumer lag balloons as backlogged records flush in after broker recovers.

Step 1 — Separate the lag signal from the broker signal

# How many partitions are under-replicated, and which broker is missing?
kafka-topics.sh --bootstrap-server :9092 \
  --describe --under-replicated-partitions
# Output shows: Replicas: 1,2,3   Isr: 1,2   ← broker-3 evicted from ISR
# Count of URPs: 120 partitions → all on broker-3

# Is broker-3 actually down?
kafka-broker-api-versions.sh --bootstrap-server broker-3:9092
# If connection refused → broker is down

# Confirm via JMX (from any running broker):
# kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions

Step 2 — Understand which consumer lag is real vs artefact

kafka-consumer-groups.sh --bootstrap-server :9092 \
  --describe --group orders-processor

# For partitions where broker-3 WAS the leader:
#   CONSUMER-ID = none (no leader = no fetch possible)
#   LAG = (LOG-END-OFFSET - CURRENT-OFFSET) appears frozen
#   → Lag is NOT growing (no new writes landing), but also not draining

# For other partitions (broker-3 was follower, other broker is leader):
#   Those partitions CAN still receive writes, so lag may grow
#   if consumers can't reconnect to the new partition assignments

Parameter Deep-Dive: What controls ISR eviction?

ParameterWhat it doesDefaultTuning note
replica.lag.time.max.ms A follower that has not sent a fetch request to the leader within this window is evicted from ISR. This is the primary ISR-eviction trigger — it fires even if the follower is alive but slow (GC, disk saturation). 30 000 ms Reduce to 10–15s to detect failures faster; risk: transient GC pauses cause false evictions. Increase if GC pauses are long.
min.insync.replicas When acks=all, the leader will only acknowledge a write after this many replicas (including itself) are in the ISR. If ISR shrinks below this value, the producer gets NotEnoughReplicasException and must retry. 1 Production: always set to RF−1 (e.g., min.ISR=2 for RF=3). Set it per topic, not just cluster default.
unclean.leader.election.enable If true, a replica NOT in the ISR can be elected leader when the ISR is empty. Restores availability at the cost of data loss (the lagging replica becomes authoritative). If false, the partition stays OFFLINE until an ISR member recovers. false Always false for financial data. true only for analytics/logging topics where losing a few records is acceptable.
num.replica.fetchers Number of threads per broker that pull data from partition leaders on other brokers. More threads = faster replication catch-up after a broker returns, but higher CPU and network cost. 1 Set to 4–8 for high-throughput clusters. Increase if ISR recovery is slow after broker restart.

Step 3 — Triage: is broker-3 recoverable quickly?

# Check broker-3 host — is process dead or hung?
systemctl status kafka   # on broker-3 host
journalctl -u kafka -n 100 --no-pager  # last 100 log lines

# Scenarios:
# A) OOM / crash → restart: systemctl start kafka
#    Then monitor URP recovery: watch -n5 kafka-topics.sh --under-replicated-partitions
#    URPs should trend to 0 as broker-3 re-fetches missing data

# B) Disk full → free space first (see disk-full runbook), then restart

# C) Hardware failure → reassign broker-3 partitions to surviving brokers:
kafka-reassign-partitions.sh --bootstrap-server :9092 \
  --reassignment-json-file reassignment.json --execute \
  --throttle 50000000   # 50 MB/s: prevents starving producers during catchup

Step 4 — Address consumer lag during recovery

# While broker-3 recovers, consumers assigned to its partitions are in limbo.
# The group coordinator will trigger a rebalance and reassign those partitions
# to other consumers — but consumers need to reconnect.

# Force consumer group refresh (if consumers are stuck):
# Most modern clients auto-reconnect. If not:
kafka-consumer-groups.sh --bootstrap-server :9092 \
  --group orders-processor --reset-offsets \
  --to-latest --topic orders --execute
# Only if you are OK skipping the backlog (e.g., real-time only system)

# If you need to process ALL records: wait for broker-3 to recover
# and let consumers drain normally. Do NOT reset offsets.

# Scale up consumers to drain backlog faster:
# Add consumers (up to partition count). They pick up partitions from broker-3
# once its leaders are re-elected on surviving brokers.

Step 5 — Apply replication throttle during ISR rebuild

# Without throttle, broker-3 catching up will saturate its network/disk
# and ALSO impact producer latency on the same broker.
# Apply throttle before restarting broker-3:

kafka-configs.sh --bootstrap-server :9092 \
  --entity-type brokers --entity-name 3 --alter \
  --add-config 'leader.replication.throttled.rate=104857600,
                follower.replication.throttled.rate=104857600'
# 100 MB/s limit → recovery in ~hours instead of minutes, but producers unaffected

# Remove throttle after URP = 0:
kafka-configs.sh --bootstrap-server :9092 \
  --entity-type brokers --entity-name 3 --alter \
  --delete-config 'leader.replication.throttled.rate,follower.replication.throttled.rate'
💡
Interview answer structure: "When I see consumer lag + URP firing together, my first question is: did a broker go down? Because that's the most common root cause of both alerts simultaneously. I check URP first — it tells me which broker — then I look at consumer lag by partition to see if the lag is on the partitions that broker led. If yes, fixing the broker fixes the lag. If the lag is on unrelated partitions, there's a second independent problem."
🔥 Scenario: High CPU on Kafka Brokers — Three Root Causes
📟
Alert: broker_cpu_usage > 85% for 10m on 3 of 6 brokers
Symptoms: Produce latency P99 climbing from 50ms → 800ms. Consumers reporting slow fetch. No disk or network alerts.
🧠
Kafka is not CPU-intensive by design — it uses zero-copy (sendfile) and relies on OS page cache. High broker CPU almost always means one of: (1) compression mismatch forcing broker-side recompression, (2) too many small requests from unbatched producers, (3) a hot partition overloading one broker, or (4) JVM GC pressure. Each has a different fingerprint.

Root Cause A — Compression Mismatch (Broker Recompression)

How to detect:
  # CPU profile (async-profiler or JFR on the broker JVM):
  java -jar async-profiler.jar -d 30 -f profile.html $(jps | grep Kafka | awk '{print $1}')
  # Look for: sun.security.provider.SHA, org.xerial.snappy, net.jpountz.lz4
  # If compression codec methods are in top 5 hotspots → broker is recompressing

  # Alternatively via thread dump — look for IO threads spending time in compress/decompress

Root cause explanation:
  Producer sends: compression.type=snappy
  Broker config:  compression.type=lz4   ← MISMATCH

  Kafka broker must:
    1. Decompress the Snappy batch (CPU)
    2. Re-validate message structure
    3. Re-compress using lz4 (CPU)
  This happens FOR EVERY PRODUCE REQUEST on the broker's IO thread.

  Also occurs when:
    - Producer uses compression.type=none → broker has compression.type=lz4
      Broker must compress every uncompressed batch.
    - Large messages (20KB+) amplify the CPU cost per batch.

Fix:
  Option A (preferred): Set producer compression.type to match broker config.
    producer: compression.type=lz4
    broker:   compression.type=lz4 (or compression.type=producer to accept any)

  Option B: Set broker compression.type=producer (accept whatever producer sends, no recompression)
    server.properties: compression.type=producer
    ↑ Zero broker CPU for compression. But consumers see heterogeneous compression.

Parameter: compression.type

ScopeValueMeaning
Producernone (default)Send uncompressed batches. Highest network cost, zero CPU cost.
Producerlz4Fast compression. Best throughput/CPU tradeoff. Recommended.
ProducerzstdBest compression ratio. Slightly higher CPU. Good for large messages.
ProducersnappyLegacy. Use lz4 instead.
Broker (topic)producerAccept whatever the producer sends. No recompression. Consumers see producer's codec.
Broker (topic)lz4Force all data stored in lz4. Broker recompresses if producer used a different codec — CPU cost.

Root Cause B — Ineffective Batching (Too Many Small Requests)

How to detect:
  # On broker: monitor requests per second vs throughput (MB/s)
  # If: requests/s is HIGH but MB/s per request is LOW → unbatched producers

  # JMX metric: kafka.network:type=RequestMetrics,name=RequestsPerSec,request=Produce
  # Healthy: 1,000–5,000 req/s handling 200+ MB/s  → large batches
  # Unhealthy: 50,000 req/s handling 200 MB/s      → tiny batches = high overhead

  # Also check: RequestQueueSize climbing (broker IO threads overwhelmed)

Root cause:
  Producer config: linger.ms=0 (default), batch.size=16384 (16KB default)

  linger.ms=0 means: send immediately, don't wait for more records.
  Result: if records arrive at 10,000 msg/sec, producer sends 10,000 separate
  requests/sec (each with 1 record) instead of batching into 100 requests
  of 100 records each.

  Broker cost: Each request requires:
    - Network thread to read request header + body
    - IO thread to write to log
    - Network thread to send response
    → 3× thread operations per request, vs batched = 3× operations for N records

  High request rate → num.network.threads saturated → RequestHandlerAvgIdlePercent drops
  → broker appears CPU-bound but the real cause is thread context-switching overhead.

Fix:
  Producer side (always fix here first — zero broker changes needed):
    linger.ms = 20        ← wait up to 20ms for more records to accumulate
                             20ms is imperceptible to humans, 10x-50x fewer requests
    batch.size = 65536    ← 64KB batch (vs 16KB default) — larger batches = fewer requests
    buffer.memory = 67108864  ← 64MB — enough buffer so linger.ms can accumulate

  Verify fix: check kafka.producer:type=producer-metrics,name=batch-size-avg
  Should increase significantly after tuning.

Parameters: Batching Triangle

ParameterWhat it doesDefaultTuning
batch.size Maximum bytes accumulated per partition before the Sender thread flushes a batch to the broker. Does NOT mean the producer waits until full — it sends sooner if linger.ms expires. Acts as an upper bound on batch size. 16 384 (16 KB) Increase to 65536–524288 (64KB–512KB) for high-throughput. Larger batches = fewer requests + better compression ratio.
linger.ms How long the Sender thread waits for more records to arrive before sending an incomplete batch. linger.ms=0 sends immediately. Adding even 5–20ms dramatically increases batch fill rate without noticeable latency impact. 0 Set to 5–100ms depending on latency tolerance. 20ms is the sweet spot for most financial pipelines.
buffer.memory Total memory the producer uses to buffer records waiting to be sent. When this buffer is full, send() blocks for up to max.block.ms. Must be large enough to absorb producer bursts during broker slowdowns. 33 554 432 (32 MB) Increase to 64–256 MB for high-throughput producers. Monitor bufferpool-wait-ratio metric — if >0, buffer is too small.
max.in.flight.requests.per.connection Number of unacknowledged requests the producer can have in-flight to a single broker at once. Higher = more pipelining = more throughput. Must be ≤5 when enable.idempotence=true (Kafka's dedup works only up to 5 in-flight). 5 Keep at 5 with idempotence. For fire-and-forget (acks=0) can increase, but rarely needed.

Root Cause C — Hot Broker (Partition Imbalance)

How to detect:
  # Which brokers are hot? (CPU per broker, not cluster average)
  # If 2 of 6 brokers are at 90% CPU and 4 are at 10% → imbalance

  # Check leader distribution:
  kafka-topics.sh --bootstrap-server :9092 --describe | \
    grep "^Topic:" | \
    awk '{print $4}' | sort | uniq -c | sort -rn
  # $4 = Leader broker ID. High count = too many leaders on one broker.

  # Or use kafka-leader-election.sh to see imbalance:
  kafka-leader-election.sh --bootstrap-server :9092 \
    --election-type PREFERRED --all-topic-partitions

Root cause - scenario A: Leader imbalance after rolling restart
  After a broker restarts, its preferred replicas (the ones it was originally
  assigned as leader) stay with other brokers. auto.leader.rebalance.enable=true
  should fix this automatically, but rebalance is delayed by
  leader.imbalance.check.interval.seconds (default 300s = 5 min).
  During this window, restarted broker has 0 leaders but still handles
  replication traffic → other brokers have 2× leaders → hot.

  Fix: Trigger immediate rebalance:
    kafka-leader-election.sh --bootstrap-server :9092 \
      --election-type PREFERRED --all-topic-partitions

Root cause - scenario B: Hot partition (bad key design)
  One partition receives 80% of traffic because the partition key has
  low cardinality (e.g., all messages keyed by country="India").
  The broker leading that partition is saturated regardless of leader balance.

  Detect: Check bytes-in-rate per partition (per topic):
    kafka-log-dirs.sh --bootstrap-server :9092 --topic-list payments
    # Look for one partition significantly larger than others

  Fix (requires application change):
    - Add a secondary key (e.g., country + customerId hash)
    - Use round-robin partitioning (null key) if ordering is not needed
    - Increase partition count → distributes the hot partition's traffic
      to more leaders across more brokers

Parameters: Broker Threading — where CPU is actually spent

🔄 Scenario: Frequent Consumer Group Rebalance — Deep Parameter Dive
📟
Alert: kafka_consumer_rebalance_rate > 4 per hour
Symptom: Consumer lag oscillates rhythmically — grows during rebalance, drains briefly, grows again. Applications log "Rebalance in progress" every 3–5 minutes. Downstream SLA being breached.

The Three Timing Parameters — Relationships and Constraints

Parameter: heartbeat.interval.ms
  What it does: How often the consumer sends a heartbeat to the Group Coordinator
                broker. The heartbeat is a lightweight keepalive — it carries NO data.
                It only tells the broker "I am still alive and a member of this group."
                The heartbeat runs on a SEPARATE background thread (since Kafka client
                2.x), independent of the poll() thread. So even a slow processing
                loop does NOT delay heartbeats — only a complete consumer JVM freeze does.
  Default: 3 000 ms (3 seconds)
  Constraint: Must be set to LESS than 1/3 of session.timeout.ms
              (so at least 3 heartbeats fit within one session timeout)

Parameter: session.timeout.ms
  What it does: The maximum time the GROUP COORDINATOR (broker-side) will wait
                without receiving a heartbeat before declaring a consumer DEAD.
                When a consumer is declared dead, the coordinator triggers a GROUP
                REBALANCE — all partitions are revoked from all consumers and
                redistributed. This is a BROKER-SIDE timeout, not client-side.
                The consumer does not control when it "times out" — only the broker does.
  Default: 45 000 ms (Kafka 3.x, was 10 000 ms in earlier versions)
  Range: [group.min.session.timeout.ms, group.max.session.timeout.ms]
         Broker defaults: min=6 000 ms, max=1 800 000 ms (30 min)
  Constraint: Must be at least 3× heartbeat.interval.ms

Parameter: max.poll.interval.ms
  What it does: The maximum time between two consecutive poll() calls. This is
                a CLIENT-SIDE liveness check. If the consumer application does not
                call poll() within this interval, the Kafka client library ITSELF
                sends a LeaveGroup request to the broker, voluntarily removing
                itself from the group and triggering a rebalance.
                Unlike session.timeout.ms (which detects JVM/network death),
                max.poll.interval.ms detects "alive but stuck processing" — i.e.,
                the consumer thread is blocked processing records and can't call poll().
  Default: 300 000 ms (5 minutes)
  Key distinction: heartbeat thread keeps session alive even when poll() is blocked.
                   But if poll() isn't called within max.poll.interval.ms, the CLIENT
                   triggers self-eviction regardless of heartbeat success.
    
The timeline of a rebalance triggered by max.poll.interval.ms breach:

T=0:    consumer.poll() returns 500 records
T=0:    application starts processing 500 records (heavy DB writes)
T=30s:  heartbeat thread sends heartbeat → broker says "still alive"  ✓
T=60s:  heartbeat thread sends heartbeat → broker says "still alive"  ✓
T=5min: max.poll.interval.ms expires → Kafka CLIENT sends LeaveGroup to broker
T=5min: Broker triggers rebalance → ALL consumers in group revoke partitions
T=5min: Consumer finishes processing → calls poll() → gets RebalanceInProgressException
T=5min: Consumer rejoins group → rebalance again → partitions reassigned
T=7min: Stable again → consumer polls → gets same 500 records AGAIN (offset not committed)
→ Duplicate processing of the 500 records

Diagnostic Checklist — Finding the Cause

1
Is the rebalance triggered by heartbeat timeout or poll timeout?
Consumer logs: search for these distinct messages:
"group coordinator ... Heart beat session timeout" → session.timeout.ms breach (JVM hung, GC pause, network)
"exceeded max.poll.interval.ms" → poll() not called in time (slow processing)
2
If session.timeout.ms breach — GC pause or network? 
# GC pause: check broker-reported heartbeat gap vs GC log timing
# On the consumer host:
grep "pause" /var/log/kafka-consumer-gc.log | tail -20
# Look for stop-the-world pauses > session.timeout.ms / 2

# If GC pauses are the cause:
# Fix 1: Switch to ZGC (-XX:+UseZGC) — sub-millisecond pauses
# Fix 2: Increase session.timeout.ms (e.g., 60 000 ms)
#         But increase heartbeat.interval.ms proportionally:
#         session.timeout.ms=60000 → heartbeat.interval.ms=15000

# If network is the cause (packet loss consumer → broker):
# Run: ping -i 0.1 -c 1000 
# Check: are heartbeat intervals consistently arriving in broker logs?
3
If max.poll.interval.ms breach — processing too slow? 
# Measure actual processing time per batch:
long start = System.currentTimeMillis();
ConsumerRecords records = consumer.poll(Duration.ofMillis(100));
// process records...
long elapsed = System.currentTimeMillis() - start;
log.info("Batch processing took {}ms", elapsed);
# If elapsed consistently > max.poll.interval.ms × 0.8 → too close to limit

# Fix options:
# Option A: Reduce max.poll.records so each batch is smaller:
max.poll.records = 100  (was 500 default)
# If each record takes 50ms → 100 records = 5s total → safe within 5min limit

# Option B: Increase max.poll.interval.ms:
max.poll.interval.ms = 600000  (10 min)
# Risk: Kafka waits 10min before detecting a truly dead consumer.
# Acceptable if processing is legitimately slow (e.g., ML inference per record).

# Option C: Move processing off the poll thread (async processing):
// Poll quickly, hand to thread pool, poll again immediately
executor.submit(() -> process(records));
consumer.commitAsync(); // commit after submission, not after completion
// Risk: If worker thread crashes, offset is already committed → at-most-once
4
Is the consumer APPLICATION crashing and restarting? 
# Check consumer pod restarts (K8s):
kubectl get pods -n payments | grep consumer
# RESTARTS column: should be 0 or very low

# If consumer is crashing: check for deserialization errors, NullPointerException
# in the processing logic on specific records.
# A single malformed record can crash the consumer → restart → rejoin → rebalance
# → same record fetched again (if offset not committed) → crash → loop

# Fix: DLQ for bad records
try {
  process(record);
} catch (DeserializationException e) {
  produceToDLQ(record);  // Send to dead letter queue
  // DO NOT let the exception propagate — it will crash the consumer
}
5
Switch to Cooperative Rebalancing as immediate mitigation 
# Eager rebalance (default with RangeAssignor):
#   ALL consumers revoke ALL partitions → all stop processing
#   → rebalance completes → ALL partitions reassigned
#   → lag spike during the gap (can be minutes for large groups)

# Cooperative Sticky Assignor (incremental rebalance):
#   Only partitions that need to MOVE are revoked
#   → consumers keep their unchanged partitions throughout
#   → lag spike is minimal (only from the partitions being moved)

# Set in consumer config:
partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor

# Also set on the broker (group.coordinator must allow it):
# No broker config needed — the assignor runs on the client side.
6
Static membership — eliminate rebalances on rolling restarts 
# Problem: Every consumer pod restart causes a rebalance (leave + rejoin)
# With 10 consumers and rolling deploy → 10 rebalances during deploy

# Solution: Static group membership
# Each consumer gets a stable instance ID:
group.instance.id=consumer-pod-0   # unique per consumer instance

# Behaviour change:
#   Normally: consumer leaves → immediate rebalance → rejoin → rebalance again
#   Static:   consumer leaves → broker waits session.timeout.ms before removing
#             If consumer rejoins WITHIN session.timeout.ms → NO rebalance
#             It resumes with its previous partition assignments

# Perfect for K8s rolling deploys where restart takes < session.timeout.ms:
session.timeout.ms=60000  # 1 minute timeout
# Pod restarts in ~30s → rejoins before timeout → zero rebalances during deploy

🏆 Hero Scenario: "The Silent Payment Loss — EOS Configured, Data Still Gone"
💡
Interview framing: "Tell me about a complex problem you solved that others couldn't." — Use STAR format: Situation → Task → Action → Result. The key is showing how you thought, not just what the answer was. Walk through your wrong hypotheses too — it shows systematic diagnosis.
🔥
Situation: Production payments pipeline at a financial institution. Exactly-once semantics (EOS) was fully configured and had been running for 6 months. A regulatory audit discovered that ~0.02% of payment records (roughly 1 in 5,000) were missing from the downstream settlement system. Audit trail showed the payments were authorised, but never appeared in settlement. No exceptions in any log. No DLQ entries. Consumer lag was consistently zero.

The EOS Setup (for context)

Producer (payment-service):
  enable.idempotence=true
  acks=all
  transactional.id=payment-producer-${instanceId}
  transaction.timeout.ms=60000  ← 60 seconds to complete a transaction

Broker:
  min.insync.replicas=2
  RF=3

Consumer (settlement-service):
  isolation.level=read_committed  ← ONLY reads committed transaction records
  enable.auto.commit=false
  Manual commit after DB write

The Investigation — Three Wrong Hypotheses First

❌
Hypothesis 1 (first team, week 1): Consumer is skipping records
The settlement consumer had idempotency logic — "if we've seen this payment ID, skip it." Suspected the dedup window was too wide.
Disproved: Checked every missing payment ID — none appeared in the dedup store. The consumer had never seen these records at all. The consumer truly never fetched them.
❌
Hypothesis 2 (second team, week 2): Broker data loss — unclean election
A broker had failed 3 weeks ago. Suspected unclean leader election had caused data loss.
Disproved: unclean.leader.election.enable=false was confirmed in broker config. Also, the missing records' timestamps were spread across 6 months — not clustered around the broker failure event. And the producer logs showed successful delivery callbacks for each missing record.
❌
Hypothesis 3 (third team, week 3): Consumer offset commit bug
Suspected offset committed before DB write — records fetched but not processed.
Disproved: The settlement DB had no record of the payment IDs. But more conclusively: running kafka-console-consumer --isolation-level read_committed --from-beginning on the topic and counting records also showed these payment IDs were MISSING. The records were not in Kafka as committed data. This was broker-side, not consumer-side.

The Breakthrough — Read Uncommitted vs Read Committed

Key diagnostic step that changed everything:

# Re-read the topic with isolation.level=read_UNCOMMITTED:
kafka-console-consumer.sh --bootstrap-server :9092 \
  --topic payments.transactions.v1 \
  --from-beginning \
  --isolation-level read_uncommitted \
  --property print.offset=true

# Result: The missing payment records ARE HERE — but with a special marker.
# Kafka console showed them, but with a transaction control record
# immediately following: type=ABORT

# This means: the PRODUCER sent the records to Kafka, but the transaction
# containing them was ABORTED. From the consumer's perspective
# (isolation.level=read_committed), aborted records are invisible — correctly so.
# But the PRODUCER believed it had successfully committed.

# The records existed in Kafka as aborted transactions:
# Offset 1001: payment-id-ABC  (part of transaction tx-42)
# Offset 1002: ABORT_MARKER    (tx-42 aborted by broker)
# Consumer with read_committed skips offset 1001 → payment lost

Root Cause — transaction.timeout.ms and a Silent Exception

Parameter: transaction.timeout.ms

The maximum time a Kafka transaction can remain open (between beginTransaction() and commitTransaction() or abortTransaction()). If this timeout expires, the broker-side transaction coordinator automatically aborts the transaction — without the producer being immediately notified. The producer finds out only on its next produce() or commitTransaction() call, which returns a ProducerFencedException.

SettingDefaultImpact
transaction.timeout.ms (producer)60 000 (60s)If the transaction takes longer than 60s, the broker aborts it. Producer must detect and handle the resulting exception.
transaction.max.timeout.ms (broker)900 000 (15min)Maximum value a producer is allowed to set for transaction.timeout.ms. Broker rejects producers with higher values.
 
The actual code path that caused the loss (simplified):

// PaymentProducer.java
producer.beginTransaction();
ProducerRecord record = new ProducerRecord("payments.transactions.v1", paymentId, data);
producer.send(record).get();  // Record lands in Kafka (in open transaction)

// ← 60 seconds pass here →
// An external KYC validation API call was added in a recent release.
// This API call happened INSIDE the transaction window.
// The KYC API had P99 latency of 45s but P99.9 of 90s.
// 1-in-5000 calls exceeded 60s → broker aborts the transaction server-side.

kycService.validate(payment);  // occasionally takes 90+ seconds

// By now, transaction.timeout.ms has expired on the broker.
// The transaction was auto-aborted by the broker coordinator.

try {
    producer.commitTransaction();  // ← This call fails!
} catch (ProducerFencedException | InvalidProducerEpochException e) {
    // This was the CRITICAL BUG:
    log.warn("Transaction commit failed, reinitialising producer", e);
    producer.initTransactions();   // ← Resets producer state
    // NO retry of the payment. NO alert. NO exception propagation.
    // The payment silently dropped. The producer moved on to the next record.
}

// The warning log was there, but:
// 1. Log level was WARN (not ERROR) — alerts were on ERROR only
// 2. Log message said "reinitialising producer" — looked like a routine reset
// 3. Rate: 1 in 5,000 → only ~20 WARNs per day in a sea of millions of log lines

The Fix — Three Changes

1
Move KYC call OUTSIDE the transaction window 
// BEFORE (broken):
producer.beginTransaction();
producer.send(record).get();
kycService.validate(payment);  // ← inside transaction, can exceed timeout
producer.commitTransaction();

// AFTER (fixed):
ValidationResult kyc = kycService.validate(payment);  // ← outside transaction
if (kyc.isApproved()) {
    producer.beginTransaction();
    producer.send(record).get();
    producer.commitTransaction();   // ← transaction is now milliseconds long
}
2
Never swallow ProducerFencedException silently 
// ProducerFencedException means the producer's epoch was fenced by the broker.
// This is ALWAYS a data-integrity event. It must:
//   a) Alert (ERROR, not WARN)
//   b) Propagate up — do not produce more records until root cause is fixed
//   c) The payment must be re-enqueued or the upstream DB transaction rolled back

catch (ProducerFencedException e) {
    log.error("CRITICAL: Producer fenced. Transaction aborted. " +
              "Payment {} NOT delivered to Kafka.", paymentId, e);
    metrics.increment("kafka.transaction.fence.count");
    throw new PaymentDeliveryException("Kafka transaction aborted", e);
    // ↑ Propagates to the payment service's retry mechanism
}
3
Add a transaction duration metric and alert 
// Instrument transaction duration:
long txStart = System.currentTimeMillis();
producer.beginTransaction();
// ... produce ...
producer.commitTransaction();
long txDuration = System.currentTimeMillis() - txStart;
metrics.histogram("kafka.transaction.duration.ms", txDuration);

// Alert if P95 transaction duration > 80% of transaction.timeout.ms:
// 80% of 60s = 48s → fire a warning
// This would have caught the problem long before the audit found it.

// Also: set a shorter transaction.timeout.ms to fail fast:
transaction.timeout.ms=10000  // 10s — transactions should be milliseconds
// If KYC ever exceeds 10s: immediate exception, clear root cause

Why This Was Hard to Diagnose

Why it fooled peopleWhat it actually meant
EOS was "fully configured" — idempotent + transactional + read_committed EOS prevents duplicate delivery and ensures atomic writes. It does NOT prevent application-level bugs where a transaction is correctly aborted but the application fails to retry the operation.
Producer showed success (send() callback had no error) send() succeeds — the record enters the transaction in an open state. The transaction timeout fires LATER. The send callback fires before the transaction commit, so it always shows success for the record itself.
Consumer lag = 0 The consumer correctly skipped aborted transaction records (isolation.level=read_committed). It consumed the offsets (moving past the abort markers) but logically never "saw" the data. Lag = 0 but records missed.
Occurrence rate: 1 in 5,000 Low enough to hide in log noise. High enough to be a regulatory issue over 6 months (thousands of missing payments).
Log said "reinitialising producer" (WARN) Sounded routine. Was actually a silent data loss event every time it occurred.
