📤Producer Tuning

Producer performance is about the throughput-latency tradeoff. Batching increases throughput but adds latency. Compression reduces network I/O but adds CPU. Tune based on your SLA.

Rule of thumb for financial systems: Use acks=all, enable.idempotence=true, linger.ms=5, batch.size=65536, compression.type=lz4. This gives good throughput while maintaining durability. Only relax acks for non-critical analytics topics.

2. Increase max.poll.records + batch process downstream writes
3. Process records in parallel within a consumer (careful with ordering)
4. Increase partition count (requires new topic or future planning)
5. Improve downstream throughput (faster DB, async writes, connection pooling)
⚙️Broker Tuning
ConfigDefaultGuidance
num.io.threads8Match to number of data disks. 8 for 1 disk, 16–24 for multi-disk RAID or JBOD. Don't over-provision — more threads = more context switching.
num.network.threads3Increase for high-connection brokers. 4–6 for >10K active connections. Monitor NetworkProcessorAvgIdlePercent.

log.segment.bytes1073741824 (1GB)1GB is good. Smaller segments = more files (overhead). Larger = slower deletion.

⚠️
Large messages are an anti-pattern. Messages over 1MB increase latency (larger fetch responses, more GC), reduce throughput (fewer messages per batch), and complicate memory management. If you need large payloads: store the payload in object storage (S3) and send only a pointer (URL/key) in Kafka. This is the "claim check" pattern.
☕JVM Tuning
RECOMMENDED JVM FLAGS FOR KAFKA BROKER (kafka-server-start.sh): ────────────────────────────────────────────────────────────────── export KAFKA_HEAP_OPTS="-Xmx6g -Xms6g" # Fixed heap: 6GB # On 32GB server: 6GB JVM + 26GB page cache # DO NOT give broker 16GB+ heap export KAFKA_JVM_PERFORMANCE_OPTS=" -server -XX:+UseG1GC # G1GC: low-pause, generational -XX:MaxGCPauseMillis=20 # Target max GC pause: 20ms -XX:InitiatingHeapOccupancyPercent=35 # Start G1 concurrent GC at 35% full -XX:+ExplicitGCInvokesConcurrent # System.gc() uses concurrent GC -XX:+DisableExplicitGC # Or disable explicit GC entirely -XX:G1HeapRegionSize=16m # Larger regions for large objects -Djava.awt.headless=true -Dcom.sun.jndi.rmiURLParsing=legacy" # Security fix FOR ZGC (JDK 15+, for sub-millisecond pauses): -XX:+UseZGC -XX:SoftMaxHeapSize=5g # Keep 1GB headroom for ZGC overhead Note: ZGC trades slightly higher CPU for much lower pause times. Use when GC pauses cause ZK session expiry issues.

Why Small Heap for Brokers

Kafka's performance depends on the OS page cache (kernel memory) for hot data reads. Every GB you give the JVM is 1 GB less page cache. A broker with 6GB JVM heap and 26GB page cache outperforms one with 16GB heap and 16GB page cache for most workloads.

GC Pause Consequences

Long GC pauses on a broker: ZK session may expire → spurious controller re-election. Network threads and IO threads stall → request queue backs up → producer/consumer timeouts. G1GC's short concurrent pauses prevent these cascades.

Heap Sizing Rule

Start with 4–6GB on a 32GB broker. Monitor GC frequency and pause times (jstat -gcutil). If Old gen fills up or GC frequency is high → increase heap by 1GB increments. Rarely need more than 8GB for standard workloads.

Producer / Consumer JVM

Application-side producers and consumers: tune based on message throughput and object allocation. Monitor bufferpool-wait-ratio. If Kafka client classes are being GC'd: increase app heap. Separate Kafka client object pools from application objects.

🐧OS & Disk Tuning
LINUX OS TUNING FOR KAFKA BROKERS: ────────────────────────────────────────────────────────────────── # File descriptor limits (Kafka opens many files — one per segment) /etc/security/limits.conf: kafka soft nofile 128000 kafka hard nofile 128000 # Virtual memory — prevent aggressive swapping (Kafka dislikes swap) /etc/sysctl.conf: vm.swappiness=1 # Minimize swap usage vm.dirty_ratio=80 # Allow up to 80% dirty pages (write buffer) vm.dirty_background_ratio=5 # Start background flush at 5% dirty pages # Network buffers (for high-throughput Kafka) net.core.rmem_max=134217728 # 128MB max receive buffer net.core.wmem_max=134217728 # 128MB max send buffer net.core.netdev_max_backlog=5000 # Larger NIC queue net.ipv4.tcp_rmem=4096 65536 134217728 net.ipv4.tcp_wmem=4096 65536 134217728 # Disk I/O scheduler # For NVMe SSDs: use none (kernel bypass) echo none > /sys/block/nvme0n1/queue/scheduler # For HDDs: use deadline or mq-deadline echo mq-deadline > /sys/block/sda/queue/scheduler # XFS filesystem (preferred over ext4 for Kafka) # - Better performance with many small files (log segments + index files) # - mkfs.xfs -f -l su=128k,sw=1 /dev/nvme0n1 # Mount with: noatime,nodiratime,nobarrier (improves I/O performance) # Transparent Huge Pages — DISABLE for Kafka (causes GC latency spikes) echo never > /sys/kernel/mm/transparent_hugepage/enabled
⚠️
Swap kills Kafka performance. When the OS swaps JVM memory to disk, Kafka broker stalls — similar to a GC pause but can last minutes. Set vm.swappiness=1 (not 0 — that causes OOM killer to trigger instead). Monitor swap usage: if swap is non-zero, the broker is under memory pressure — increase RAM or reduce heap.
🏎️Benchmarking Tools
KAFKA BUILT-IN BENCHMARK TOOLS: ────────────────────────────────────────────────────────────────── # Producer performance test kafka-producer-perf-test.sh \ --topic benchmark-topic \ --num-records 10000000 \ # 10M records --record-size 1024 \ # 1KB per record --throughput -1 \ # -1 = unlimited (find max throughput) --producer-props \ bootstrap.servers=broker:9092 \ acks=all \ linger.ms=5 \ batch.size=65536 \ compression.type=lz4 Sample output: 10000000 records sent, 450321.3 records/sec (440.0 MB/sec), 3.2 ms avg latency, 25.0 ms max latency, 2 ms 50th, 8 ms 95th, 15 ms 99th, 22 ms 99.9th # Consumer performance test kafka-consumer-perf-test.sh \ --topic benchmark-topic \ --messages 10000000 \ --bootstrap-server broker:9092 \ --fetch-size 4194304 # 4MB fetch Sample output: start.time end.time data.consumed.in.MB nMsg.consumed Fetch.MB.sec nMsg.sec 2026-05-28 2026-05-28 9765.625 10000000 800.44 819500 # End-to-end latency test kafka-e2e-latency.sh broker:9092 benchmark-topic 10000 1 --acks=all
✅
Benchmarking methodology:
1. Establish baseline with default configs
2. Change ONE variable at a time
3. Run each test for at least 5 minutes (steady state)
4. Measure: throughput (MB/s), latency (p50, p95, p99, p999), CPU, disk I/O, network
5. Test at different throughput levels — Kafka behavior changes significantly at saturation
6. Always benchmark in an environment matching production (same hardware, same replication)
🎯Performance Q&A
Q Your team wants to achieve 1 million messages/second through Kafka. What's your approach? Senior
1M msg/sec at 1KB avg = 1 GB/sec. This requires careful design at every layer:
  • Cluster sizing: 10+ brokers, NVMe SSDs (3+ GB/s throughput), 25Gb+ NICs. Kafka can sustain ~100–200 MB/s per broker sustainably; 1 GB/s total needs 5–10 brokers minimum.
  • Topic design: 100+ partitions distributed evenly across brokers. Avoid hot partitions — use random or hash-based partitioning.
  • Producer: Multiple producer instances (or threads). linger.ms=20, batch.size=512KB, compression.type=lz4, acks=1 (if data loss tolerance exists). Multiple producer instances per topic.
  • Broker: num.io.threads=16, num.network.threads=6, num.replica.fetchers=4. Disable explicit fsync (log.flush.interval.messages=Long.MAX_VALUE). Use JBOD (log.dirs=/disk1,/disk2,...) for parallel I/O.
  • OS: Tuned network buffers, vm.dirty_ratio=80, io scheduler=none for NVMe, huge pages disabled.
  • Verify: Run kafka-producer-perf-test.sh --throughput -1 with realistic record sizes. Identify bottleneck (CPU, network, disk) before tuning further.
Q Consumer throughput is 200MB/s but the broker disk can serve 1GB/s. What's limiting consumption? Mid
Several possible bottlenecks between disk and consumer:
  • Consumer count vs partition count: If 4 consumers on 4 partitions each get 50MB/s = 200MB/s. Add more consumers (up to partition count) and partition leaders on different brokers to parallelise.
  • fetch.max.bytes: Default 50MB per fetch response. With 4 consumers: 4 × 50MB/s ceiling. Increase to 100MB+ and ensure Kafka buffers on client side are large enough.
  • Network between broker and consumers: 4 consumers × 50MB/s = 200MB/s of actual network egress from broker. Check NIC utilization.
  • Consumer processing throughput: If consumers can only process 200MB/s worth of data (deserialization, DB writes), they won't fetch faster regardless of broker capacity. Profile consumer CPU and downstream.
  • Zero-copy disabled: Check if SSL is enabled between broker and consumer — SSL prevents zero-copy (sendfile), adds CPU for encryption. This can halve effective consumer throughput.
