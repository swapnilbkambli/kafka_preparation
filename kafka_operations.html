🐢Producer Slowness — Diagnostic Checklist

When producers report high latency or throughput drops, work through this checklist systematically. Always start with metrics before changing configs.

💡
First action: Check JMX metrics kafka.producer:type=producer-metrics,name=request-latency-avg and record-error-rate. High error rate means retries are killing throughput. High latency without errors means the broker is slow to respond.
#CheckCommand / MetricWhat to Look For
1ISR health — acks=all bottleneckkafka-topics.sh --describe --under-replicated-partitionsAny URP = producer blocked waiting for ISR acks. Fix URP first.
2Broker request queue saturationJMX: RequestQueueSize, ProduceRequestQueueTimeMs>200 queue size → IO thread bottleneck. Increase num.io.threads.
3Broker disk I/Oiostat -x 1 on broker hosts%util near 100% → disk saturated. Add log dirs or move to NVMe.
4Broker GC pausesBroker GC logs: grep "GC pause" kafka.log>200ms GC → broker stalls ack responses. Tune JVM (G1GC, smaller heap).
5Network throughput producer→brokersar -n DEV 1 or iftopNIC saturation → batching helps (larger batch.size, linger.ms).
6Producer batch efficiencyJMX: batch-size-avg, batch-size-maxIf avg batch ≪ batch.size, batches are being sent before filling — increase linger.ms (5–20ms).
7Producer buffer full / blockingJMX: bufferpool-wait-ratio > 0Producer blocked waiting for buffer space → increase buffer.memory or reduce send rate.
8Retries and error rateJMX: record-retry-rate, record-error-rateHigh retries → transient broker issues or misconfigured delivery.timeout.ms.
9Compression overheadJMX: compression-rate-avg, CPU usage on producer hostHigh CPU + low compression ratio → try lz4 instead of gzip/zstd for CPU-constrained producers.
10Metadata fetch latencyJMX: metadata-ageIf metadata is frequently refreshed, producer may be hitting unknown partitions or leader changes.
11Schema Registry latencySchema Registry logs / HTTP latencyAvro/Protobuf producers call SR on first schema use — SR down or slow = producer stall.
12Broker leader imbalancekafka-topics.sh --describe | grep LeaderIf all leaders on one broker, that broker is overloaded. Run preferred replica election.
13Producer quota throttlingJMX: produce-throttle-time-avg>0ms → broker is throttling this client. Raise quota or reduce send rate.
14Request timeout vs actual latencyJMX: request-latency-avg vs request.timeout.msIf p99 latency approaches timeout → increase request.timeout.ms or fix root cause.
✅
Decision tree summary:
High latency, no errors → broker disk or IO threads slow → check iostat and RequestHandlerAvgIdlePercent
High error/retry rate → ISR issues or broker restarts → check UnderReplicatedPartitions
Producer blocks → buffer full → check bufferpool-wait-ratio → increase buffer.memory
Throughput low but latency OK → batching inefficient → increase linger.ms and batch.size
🐌Consumer Slowness — Diagnostic Checklist
💡
First action: Run kafka-consumer-groups.sh --describe --group <group>. Check LAG column. Is lag growing (consumer falling behind) or stable (caught up but slow processing)? These are fundamentally different problems.
#CheckCommand / MetricWhat to Look For
1Lag trendkafka-consumer-groups.sh --describe --group <g>Growing lag = consumer can't keep up. Stable lag = caught up but not at 0. Both need different fixes.
2Rebalance frequencyConsumer logs: "Rebalance in progress", JMX rebalance-rate-per-hourFrequent rebalances = consumption pauses. Check heartbeat/poll interval configs.
3max.poll.interval.ms violationConsumer logs: "max.poll.interval.ms exceeded", poll-idle-ratio-avgProcessing per batch takes longer than max.poll.interval.ms → consumer kicked out → lag spike. Reduce max.poll.records.
4Processing bottleneckApplication code profiling, consumer host CPU/memoryDB writes, HTTP calls, complex transforms inside poll loop → move to async, batch DB inserts.
5Consumer GC pausesConsumer GC logsLong GC pauses trigger session.timeout → rebalance. Use G1GC, tune heap size.
6Fetch throughputJMX: fetch-rate, bytes-consumed-rate, fetch-latency-avgLow fetch-rate with available data → consumer not polling fast enough. Separate processing from polling thread.
7Fetch config tuningfetch.min.bytes, fetch.max.wait.ms, max.partition.fetch.bytesIf fetch-latency-avg ≈ fetch.max.wait.ms, broker is waiting for data. Raise fetch.min.bytes or add partitions/producers.
8Partition-to-consumer ratioCount consumers in group vs topic partition countIf partitions < consumers → idle consumers. If partitions ≫ consumers → each consumer handles many partitions, increase consumer count or partitions.
9Broker disk / page cacheiostat on brokers, JMX LocalTimeMs (Fetch)High fetch local time → broker serving reads from disk (page cache miss). Old data or broker memory too small.
10Network between consumer and brokerping, traceroute, iperf3High network latency or packet loss → raises fetch-latency-avg. Check cross-AZ or cross-DC traffic.
11Consumer quota throttlingJMX: fetch-throttle-time-avg>0ms → broker is throttling this consumer group. Raise consumer quota.
12Schema Registry deserializationConsumer logs, SR response timeSchema ID not cached → SR called per message or per schema refresh. Check SR cache settings.
13Offset commit latencyJMX: commit-latency-avgSlow offset commits (slow group coordinator) add overhead. Use async commit where safe.
14Dead Letter Queue backpressureDLQ topic lag, error logsIf consumer writes failed records to DLQ synchronously and DLQ broker is slow → consumer stalls.
⚠️
Common anti-pattern: Setting max.poll.records=500 (default) while doing synchronous DB inserts in the poll loop. Each batch of 500 records takes 5+ seconds → exceeds max.poll.interval.ms=300s under load → rebalance storm. Fix: set max.poll.records=50 or move DB writes to async batch insert.
📋Topic Management
# Create topic kafka-topics.sh --bootstrap-server broker:9092 \ --create --topic payments \ --partitions 12 \ --replication-factor 3 \ --config retention.ms=604800000 \ # 7 days --config min.insync.replicas=2 # Describe topic (show ISR, leaders, offline partitions) kafka-topics.sh --bootstrap-server broker:9092 --describe --topic payments # List only under-replicated partitions kafka-topics.sh --bootstrap-server broker:9092 --describe --under-replicated-partitions # List only unavailable partitions (no leader) kafka-topics.sh --bootstrap-server broker:9092 --describe --unavailable-partitions # Increase partition count (CANNOT decrease) kafka-topics.sh --bootstrap-server broker:9092 --alter --topic payments --partitions 24 # Alter topic config kafka-configs.sh --bootstrap-server broker:9092 --entity-type topics \ --entity-name payments --alter \ --add-config retention.ms=2592000000 # 30 days # Delete topic (requires delete.topic.enable=true) kafka-topics.sh --bootstrap-server broker:9092 --delete --topic old-topic
⚠️
Partition count is permanent (increasing only). Increasing partitions breaks key-based ordering for existing keys — consumers that relied on "all records for key X go to same partition" will see re-routing. Plan partition count based on 2-3x expected peak consumers. Industry guideline: don't exceed 4,000 partitions per broker (ZK mode) or much higher with KRaft.
🔀Partition Reassignment

Used when: adding a broker (rebalance data), removing a broker, fixing skewed storage, or changing replication factor.

Step 1: Generate reassignment plan kafka-reassign-partitions.sh --bootstrap-server broker:9092 \ --topics-to-move-json-file topics.json \ --broker-list "1,2,3,4" \ # include new broker IDs --generate > reassignment.json Step 2: Review the plan (check source/destination brokers, balance) Step 3: Execute with throttle to limit replication bandwidth impact kafka-reassign-partitions.sh --bootstrap-server broker:9092 \ --reassignment-json-file reassignment.json \ --execute \ --throttle 50000000 # 50 MB/s replication throttle Step 4: Monitor progress kafka-reassign-partitions.sh --bootstrap-server broker:9092 \ --reassignment-json-file reassignment.json \ --verify Step 5: Remove throttle after completion kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type brokers --entity-default \ --alter --delete-config leader.replication.throttled.rate kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type brokers --entity-default \ --alter --delete-config follower.replication.throttled.rate
🚨
Always throttle reassignments in production. Without a throttle, inter-broker replication can saturate your network and disk, causing consumer lag spikes and broker instability. Start with 50 MB/s and adjust. Forgetting to remove the throttle after completion is a common mistake — it permanently limits replication speed.
🔄Rolling Restart Procedure

Used for: config changes, version upgrades, OS patches. Must be done one broker at a time to avoid simultaneous leader loss.

1
Pre-checks — cluster health gate
Verify zero under-replicated partitions: kafka-topics.sh --describe --under-replicated-partitions. Do NOT proceed with any URP — rolling restart on a degraded cluster risks data loss.
2
Identify the broker to restart
Choose a non-controller broker first. Restarting the controller triggers election overhead. Save controller for last.
3
Controlled shutdown (preferred)
Send SIGTERM — Kafka performs a controlled shutdown: transfers partition leadership to other brokers gracefully. Much faster than a hard kill. Verify with logs: "Controlled shutdown succeeded."
4
Wait for ISR recovery
After broker restarts, wait until all its partitions rejoin ISR: kafka-topics.sh --describe --under-replicated-partitions should return empty. This can take minutes for large clusters.
5
Proceed to next broker
Only restart the next broker once URP = 0. Repeat for all brokers. Restarting the controller broker last minimizes metadata disruption.
6
Post-restart: preferred replica election
After all brokers restarted, run preferred replica election to restore original partition-leader placement: kafka-leader-election.sh --bootstrap-server broker:9092 --election-type PREFERRED --all-topic-partitions
ConfigPurposeRecommended
controlled.shutdown.enableGraceful leadership transfer on stoptrue (default)
controlled.shutdown.max.retriesRetries for controlled shutdown3
controlled.shutdown.retry.backoff.msBackoff between retries5000ms
⚖️Leader Balancing

After broker restarts or failures, partition leadership may become skewed. One broker handles more produce/consume requests → becomes hot. Auto-balancing or manual election restores balance.

# Check leader distribution kafka-topics.sh --bootstrap-server broker:9092 --describe | grep -oP 'Leader:\K\d+' | sort | uniq -c # Manual preferred replica election (all partitions) kafka-leader-election.sh --bootstrap-server broker:9092 \ --election-type PREFERRED --all-topic-partitions # Auto-balancing (broker config) auto.leader.rebalance.enable=true # default true leader.imbalance.check.interval.seconds=300 # check every 5min leader.imbalance.per.broker.percentage=10 # trigger if >10% imbalanced
🚦Quotas & Throttling

Quotas protect broker stability by limiting per-client bandwidth. Essential in multi-tenant Kafka clusters (like a bank shared by multiple teams).

# Set producer quota for specific client kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type clients --entity-name my-producer-app \ --alter --add-config 'producer_byte_rate=10485760' # 10 MB/s # Set consumer quota kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type clients --entity-name my-consumer-app \ --alter --add-config 'consumer_byte_rate=20971520' # 20 MB/s # Set quota per user (with SASL) kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type users --entity-name alice \ --alter --add-config 'producer_byte_rate=52428800' # 50 MB/s # Set request quota (% of IO threads time) kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type clients --entity-default \ --alter --add-config 'request_percentage=25' # max 25% of IO thread time # View current quotas kafka-configs.sh --bootstrap-server broker:9092 \ --entity-type clients --entity-name my-producer-app --describe
✅
HSBC context: In a multi-team bank environment, quotas are critical. A rogue batch job or poorly configured producer can saturate shared brokers and affect payment processing pipelines. Set default quotas first (--entity-default), then override per team/application. Monitor produce-throttle-time-avg and fetch-throttle-time-avg per client.
📍Offset Management
# Check consumer group lag kafka-consumer-groups.sh --bootstrap-server broker:9092 \ --describe --group payments-processor Output: GROUP TOPIC PARTITION CURRENT-OFFSET LOG-END-OFFSET LAG payments-processor payments 0 1000 1050 50 payments-processor payments 1 800 800 0 # Reset offsets to earliest (replay all) kafka-consumer-groups.sh --bootstrap-server broker:9092 \ --group payments-processor --topic payments \ --reset-offsets --to-earliest --execute # Reset to specific timestamp (e.g., replay from midnight) kafka-consumer-groups.sh --bootstrap-server broker:9092 \ --group payments-processor --topic payments \ --reset-offsets --to-datetime 2026-05-28T00:00:00.000 --execute # Reset to specific offset kafka-consumer-groups.sh --bootstrap-server broker:9092 \ --group payments-processor --topic payments \ --reset-offsets --to-offset 500 --execute # Delete consumer group (must be inactive) kafka-consumer-groups.sh --bootstrap-server broker:9092 \ --delete --group old-consumer-group
⚠️
Offset reset requires the consumer group to be inactive. Stop all consumers in the group first. Resetting to earliest on a compacted topic still starts from the earliest retained offset, not necessarily offset 0.
🎯Operations Q&A
Q A producer is suddenly getting TimeoutExceptions after a routine config push to brokers. What's your investigation path? Senior
Structured approach:
  • Correlate timing: Did the config push trigger a rolling restart? If yes, is the cluster fully recovered (zero URP)?
  • Check if brokers are reachable: kafka-broker-api-versions.sh --bootstrap-server broker:9092. If this fails, it's network/listener configuration, not application.
  • Check listener config: A common mistake after config push is changing listeners or advertised.listeners incorrectly. Producers get metadata pointing to wrong host/port.
  • Check RequestQueueSize: If brokers just restarted and are catching up replication, they may be temporarily overwhelmed.
  • Check producer logs: Is it "Timeout expired while awaiting ack" (ISR issue) or "Failed to update metadata" (connectivity)? Very different root causes.
  • Rollback config change: If all else fails and you can't identify the cause quickly, roll back the config and re-apply changes one at a time to isolate.
Q Consumer lag is at 2 million and growing. You have 6 consumers and 12 partitions. What do you do? Mid
Step 1 — Identify the bottleneck: Is this all partitions lagging, or a few specific ones? kafka-consumer-groups.sh --describe per partition.

Step 2 — Add consumers: With 12 partitions and 6 consumers, adding 6 more consumers (to 12:1 ratio) doubles throughput immediately — no code change needed. This is the fastest fix if the bottleneck is consumer count.

Step 3 — Check processing bottleneck: If adding consumers doesn't help, the bottleneck is downstream (DB, API). Profile the processing code. Consider batching writes, async processing, or a faster downstream.

Step 4 — Reduce batch size: Lower max.poll.records to prevent max.poll.interval.ms violations if processing is slow per record.

Step 5 — Increase parallelism: If 12 partitions isn't enough even with 12 consumers, you'll need to increase partition count (for future topics) or increase per-consumer throughput.
Q How do you safely increase the replication factor of an existing topic with no downtime? Senior
You use partition reassignment with an updated replica assignment:
  1. Create a JSON file specifying the new replica list for each partition (adding the new broker ID): {"version":1, "partitions":[{"topic":"payments","partition":0,"replicas":[1,2,3]},...]}
  2. Execute with throttle: kafka-reassign-partitions.sh --execute --throttle 50000000 --reassignment-json-file rf3.json
  3. Monitor: kafka-reassign-partitions.sh --verify until all partitions report "completed"
  4. Remove throttle configs
This is fully online — producers and consumers continue during the reassignment. The new follower fetches data from the leader and joins ISR once caught up. No downtime required.
