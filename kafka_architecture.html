🧵Broker Thread Model

Kafka brokers are highly concurrent and use a specific multi-threaded architecture to serve thousands of connections efficiently. Understanding this is critical for SRE-level debugging.

CLIENT BROKER PROCESS ────── ────────────────────────────────────────────── Producer ──TCP──► [Acceptor Thread] (1 per listener) Consumer │ Admin CLI ┌────────┴────────────────────────────────┐ │ │ NetworkThread-1 NetworkThread-2 NetworkThread-N (NIO Selector) (NIO Selector) (NIO Selector) num.network.threads = 3 (default) │ │ │ └────────────────┴────────┬────────┘ ▼ [ Request Queue ] queued.max.requests = 500 queued.max.request.bytes = default unlimited │ ┌─────────────────────────┼─────────────────────────┐ │ │ │ IOThread-1 IOThread-2 IOThread-N num.io.threads = 8 (default, should match disk count) │ │ │ └─────────────────────────┼─────────────────────────┘ ▼ [ Response Queues ] (per NetworkThread) │ NetworkThreads drain → write back to client socket

Acceptor Thread

One per listener (PLAINTEXT, SSL, etc.). Accepts incoming TCP connections and hands them off to NetworkThreads using round-robin. Never does I/O itself — just connection registration.

Network Threads

NIO-based selector threads. Each manages many connections. Reads raw bytes from sockets, parses Kafka protocol frames, and enqueues RequestChannel.Request objects. Also drains response queues and writes bytes back to sockets. Config: num.network.threads (default 3).

Request Queue

A single bounded blocking queue shared by all network threads. IO threads consume from it. Config: queued.max.requests (default 500). When full, network threads stop reading from sockets — this is the back-pressure mechanism.

IO Threads

The workhorse threads. Dequeue requests, execute the actual logic (write to log, read from disk, update metadata), and enqueue responses. Config: num.io.threads (default 8). Should roughly match number of storage volumes for disk-bound workloads.

Purgatory (Delayed Requests)

Requests that can't be completed immediately (waiting for ISR acks, or waiting for enough fetch data) are parked in a Purgatory — a time-wheel based structure. When the condition is met, the delayed operation completes and the response is enqueued.

Response Queue

Per-NetworkThread queue. IO threads route responses to the correct NetworkThread's response queue. The NetworkThread drains it and writes bytes back to the client socket. This avoids lock contention on a single response queue.

💡
Key insight for interviews: Kafka's fetch purgatory is what enables long-polling (efficient push-like behavior). Consumers don't busy-poll — they issue a fetch request with fetch.max.wait.ms and the broker parks it until data arrives. This is why Kafka can handle millions of idle consumers without burning CPU.
⚠️
Request queue saturation: When queued.max.requests is hit, network threads stop reading from sockets. TCP receive buffers fill up. Eventually clients see TimeoutException — they get no response within request.timeout.ms. Metric to watch: kafka.network:type=RequestChannel,name=RequestQueueSize.
🌊How the Broker Handles Sudden Message Influx

A traffic spike (e.g., a batch job suddenly producing 10x normal volume) exercises multiple Kafka subsystems. Here's what happens layer by layer:

1
TCP receive buffer absorbs initial burst
OS buffers incoming TCP segments. Sized by socket.receive.buffer.bytes. Buys a few hundred milliseconds before clients see back-pressure.
2
NetworkThreads parse and enqueue
Parse incoming bytes into Request objects and enqueue to RequestQueue. Rate is limited by NIO throughput (~400K small requests/sec per thread).
3
IOThreads write to page cache (fast path)
Writing to Kafka is a sequential append to the OS page cache. This is extremely fast — can sustain GB/s. The OS decides when to flush to disk (log.flush.interval.ms).
4
If acks=all: ProducePurgatory fills up
Requests waiting for ISR acknowledgment pile up in purgatory. If ISR followers can't keep up, purgatory grows. If max.in.flight.requests is reached, producer blocks.
5
Followers experience fetch lag
Followers fetch from leader. Under load, followers may lag behind. If lag exceeds replica.lag.time.max.ms, they're removed from ISR — which makes acks=all complete faster (fewer waiters) but reduces durability.
6
RequestQueue hits limit (back-pressure)
If IOThreads can't keep up, RequestQueue fills to queued.max.requests. NetworkThreads pause reading sockets. TCP buffers fill. Producer sees connection stall → request.timeout.ms fires → TimeoutException or retry.
7
OS background flush + disk I/O becomes bottleneck
Eventually the OS flushes dirty pages to disk. If disk is slow (HDD, shared storage), flush can compete with serving reads from page cache. This is when you see read latency spike too.
✅
SRE mitigation strategies for traffic spikes:
1. Producer quotas: producer_byte_rate per client — throttle misbehaving producers before they saturate the broker.
2. More IO threads: Increase num.io.threads if CPU/disk capacity exists.
3. More partitions: Distributes load across more IOThread-level parallelism.
4. Tiered storage: Offload old segments to object storage, freeing local disk I/O for active writes.
5. Producer batching: Encourage producers to use larger batch.size and linger.ms — fewer, larger requests are cheaper for the broker.

.log file

The actual data. Append-only. Records are grouped into Record Batches. The active segment is the only one being written to. All others are immutable.

.index file (offset index)

Sparse index mapping logical offsets → physical byte positions in the .log file. Enables O(log n) offset lookups — binary search the index, then sequential scan within the segment.

.timeindex file

Maps timestamps → offsets. Used for offsetsForTimes() API — consumers seeking to a specific point in time. Enables replaying from "yesterday 9am."

Segment Rolling

A new segment is created when the active segment exceeds log.segment.bytes (1 GB default) or log.roll.hours (168h). Sealed segments are eligible for retention/compaction.

Retention

Broker periodically checks if segments can be deleted: log.retention.bytes (total size limit) or log.retention.ms (time-based, default 7 days). Only whole sealed segments are deleted.

leader-epoch-checkpoint

Records the starting offset for each leader epoch. Used by followers on recovery to truncate divergent entries — prevents data corruption after leader failover without needing full log comparison.

⚡Page Cache & Zero-Copy

Why Kafka Uses OS Page Cache

Kafka never reads/writes disk directly — it writes to the OS page cache (kernel memory) and lets the OS flush to disk asynchronously. Benefits: automatic read-ahead, write coalescing, and data survives JVM restarts (stays in kernel memory).

JVM Heap Sizing Implication

Give Kafka brokers small JVM heap (4–8 GB) and let the rest of RAM be page cache. A 32 GB server should have 6 GB heap and 26 GB free for OS. Oversizing heap causes GC pressure and evicts page cache — hurts read performance.

Zero-Copy (Consumer Reads)

When a consumer fetches data, the broker uses the sendfile() syscall: data transfers directly from page cache → network card via DMA, bypassing userspace entirely. Eliminates 2 data copies and 2 context switches per read.

When Zero-Copy is NOT Used

Zero-copy only works for uncompressed or same-codec data. If the broker must decompress-and-recompress (e.g., codec mismatch, or message format conversion), it goes through userspace — much slower. Keep producer and broker compression codecs aligned.

🎯Architecture Q&A
Q Why does Kafka use separate network threads and IO threads instead of one thread per connection? Mid
One-thread-per-connection (traditional Java IO model) doesn't scale beyond a few thousand connections — threads are expensive (1MB stack each) and context switching dominates CPU. Kafka's NIO model: few NetworkThreads serve many connections via non-blocking selector (epoll/kqueue). A single NetworkThread handles 10,000+ idle connections. IOThreads are separate because disk I/O can block — keeping them off the network path means a slow disk doesn't stall socket reads. This separation also allows tuning each pool independently to match hardware (e.g., 4 network threads for a 10G NIC, 16 IO threads for a 16-disk RAID array).
Q A broker is logging "Request queue is full" errors. What's happening and how do you fix it? Senior
What's happening: IOThreads can't drain the RequestQueue fast enough — either they're CPU-bound, disk-bound, or blocked waiting for ISR acks in purgatory.
  • Check RequestHandlerAvgIdlePercent: If <30%, IO threads are saturated. First action: num.io.threads.
  • Check RemoteTimeMs: High values mean produce requests are parked waiting for ISR acks — follower lag is the real bottleneck. Fix follower catching up (check MaxLag metric, network, disk on followers).
  • Check disk I/O (iostat): If %util is 100%, add more disks or distribute log dirs with log.dirs.
  • Short-term: Increase queued.max.requests to absorb burst — but this is a band-aid, not a fix.
  • Apply producer quotas: kafka-configs.sh --alter --add-config 'producer_byte_rate=10485760' to throttle problematic clients.
Q How does Kafka guarantee message ordering within a partition despite producer retries? Mid
Without idempotence, a producer sending records A → B could receive a timeout on A, retry, and end up with B → A on the broker if B was already accepted.

Idempotent producer solves this: the broker assigns each producer a ProducerID (PID) and tracks sequenceNumber per partition. If a batch arrives out of order or is duplicated, the broker detects it and either rejects (out-of-order) or deduplicates (duplicate). This is safe up to max.in.flight.requests.per.connection=5 in-flight per connection — the broker validates sequence numbers for all 5 in-flight batches simultaneously.
Q Walk me through what happens when a broker crashes while a ProduceRequest is in purgatory. Senior
1. The leader broker crashes. OS closes TCP connections. All in-flight requests are lost.
2. Producer sees ConnectionException or timeout — triggers retry.
3. Controller detects broker failure (ZK session expiry or KRaft heartbeat timeout). Triggers partition leader election for all partitions the failed broker led.
4. New leader is elected from ISR. It already has all data the old leader had (because it was in-sync).
5. Producer metadata refresh discovers the new leader.
6. Producer retries the ProduceRequest to the new leader.
7. With idempotence: broker deduplicates if the original request actually got written and replicated before the crash. Without idempotence: potential duplicate.

Key edge case: If the original request was written to the leader's page cache but NOT flushed to disk AND not replicated to any follower before the crash, the data is lost — even with acks=all — because the leader dies before replication. This is why min.insync.replicas=2 ensures at least one follower has confirmed before acking the producer.
